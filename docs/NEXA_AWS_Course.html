AWS, explained through NEXA

A course that teaches the core of AWS — VPC, IAM, EC2, ALB, RDS, S3 and the pieces around them — not in the abstract, but by mapping every concept onto a real application you already understand: the NEXA Business Development & Pricing tool you are moving off Vercel + Supabase onto AWS.

The through-line is hierarchy: how AWS nests an account inside an organisation, a network inside a region, a server inside a subnet, a permission inside a role — and how a project like NEXA is structured in layers that mirror that nesting.

For: an AWS newcomer shipping a real app Worked example throughout: NEXA 19 sections · diagrams · gotchas

How to read this

Most AWS material teaches services one at a time, in isolation, with a generic "to-do app" you don't care about. The problem is that AWS only makes sense as a system — the value is in how the pieces fit, not in any one of them. So this course does the opposite: it keeps one app in view the whole time and shows where each AWS service plugs in.

That app is NEXA. You already know it cold, so you can spend all your attention on the AWS half. As a one-line refresher, here is the shape of NEXA, because everything below hangs off it:

NEXA in one breath A single Next.js 16 application (UI and API in one deployable process) used by ~55 internal staff across the US and India. It tracks business-development opportunities through pricing, multi-level approval, and SOW/PO signing. Login is Azure AD SSO. All structured data lives in PostgreSQL; signed SOW/PO documents live in object storage; approval emails go out through Microsoft Graph. The data is commercially sensitive — deal pricing, cost & bill rates, margins. It runs on Vercel + Supabase today; you are moving it to AWS.

1The mental model: AWS is a data centre you rent by the piece

Before any single service, fix this picture. AWS is one enormous set of data centres around the world. You do not rent a building; you rent individual capabilities out of it — a virtual machine here, a managed database there, a chunk of disk for files — and you wire them together over a private network you also rent. Everything else is detail.

On Vercel + Supabase, those decisions were made for you. Vercel decided where your function ran, how it scaled, what TLS certificate it used, how logs were collected. Supabase decided the network around your database. Moving to AWS means you now make — and own — each of those choices. That is the whole reason this feels like more moving parts: AWS hands you the pieces Vercel had pre-assembled.

The shift in who owns each decision
DecisionOn Vercel/SupabaseOn AWS
Where the app runsPlatform picks an edge regionYou pick a Region, AZ, subnet, instance
The network around itInvisible / managedYou design a VPC, subnets, routes, firewalls
Who/what can touch whatPlatform tokensYou write IAM roles & policies
The public URL & certAutomaticYou wire Route 53 + ACM + ALB
Restart on crash, logs, backupsAutomaticYou configure systemd/PM2, CloudWatch, RDS backups

This is the trade: more control and a clean security story, in exchange for assembling pieces the old platform hid. NEXA's plan deliberately starts on the simplest assembly (one EC2) so the number of new pieces is small.

Two ideas that unlock everything else

If you internalise only two things from this course, make them these — every service later is a special case of one or the other.

Idea 1 — Containment (the nesting) AWS resources sit inside one another like Russian dolls. A server is inside a subnet, which is inside a VPC, which is inside a Region, which is inside your account. You cannot understand "where" something is until you can name every doll around it. This is the hierarchy of place. Section 2 is entirely about this.
Idea 2 — Explicit permission (deny by default) Nothing in AWS can talk to anything else unless something explicitly allows it. An EC2 server cannot read an S3 bucket, reach a database, or even leave the network unless a rule (a security group) or a permission (an IAM policy) grants it. This is the hierarchy of trust. Sections 6 and 12 are about this.

Place and trust. Almost every AWS error you will hit as a newcomer is one of two sentences: "it's not where I think it is" (a containment/routing mistake) or "it isn't allowed to do that" (a security-group or IAM mistake). Knowing which of the two you're debugging is half the battle.

2The AWS hierarchy — where everything lives

This is the spine of the course, so it comes first. AWS has a strict containment hierarchy. Every resource you ever create lands at some level of it. Learn the ladder once and every service afterward has an obvious home.

2.1 — The containment ladder, top to bottom

Read this from the outside in. Each box contains the ones below it; nothing can escape its container.

billing & policy rootAWS Organization — ProcDNA's whole AWS footprint
The top of the tree. Groups many accounts under one bill and one set of guardrails (Service Control Policies). The "AUXO" account NEXA targets lives here, beside other ProcDNA accounts.
isolation boundaryAccount — e.g. AUXO (or a nexa-dev sub-account)
The hard wall of AWS. Resources, permissions, and bills are scoped to an account. This is the cleanest blast-radius boundary — a mistake in one account can't touch another. Decision #2 in your plan ("existing VPC or new?") really lives here: which account does NEXA belong in.
geographyRegion — e.g. eu-central-1 (Frankfurt) or ap-south-1 (Mumbai)
A physical cluster of data centres in one part of the world. You pick one per deployment. Data residency and latency live here — this is your plan's Decision #1.
fault domainAvailability Zone (AZ) — e.g. eu-central-1a, …-1b
One or more discrete data centres inside a Region, isolated from the others (separate power, cooling, network). You spread across ≥2 AZs so one failing doesn't take you down. RDS requires two AZs even for a single-AZ database (see §10).
your private networkVPC — e.g. 10.0.0.0/16
Your own isolated virtual network inside the Region. It spans all AZs in that Region. This is the "data centre floor" NEXA's servers and database plug into. Everything below here is yours to wire. (§4)
a slice per AZSubnet — public 10.0.1.0/24 · private 10.0.11.0/24
A range of the VPC's addresses pinned to one AZ, marked public (can reach the internet) or private (cannot). NEXA's app server sits in a public subnet; RDS sits in private subnets. (§4–5)
the actual thingResource — the EC2 instance, the RDS database, an ENI
The leaf. An EC2 instance lives in exactly one subnet (one AZ). An RDS instance lives in a subnet group spanning AZs. S3 and IAM are the exceptions — see the note below.

The containment ladder. When someone asks "where is the NEXA app server?" the full answer is a path down this tree: AUXO account → eu-central-1 → AZ a → VPC 10.0.0.0/16 → public subnet 10.0.1.0/24 → this EC2 instance.

The two famous exceptions to the ladder
  • Amazon S3 and IAM are not inside a VPC. S3 buckets live at the Region level (with a globally-unique name) and IAM lives at the account level (global). They float above the network. That's why reaching S3 from a private subnet needs special routing (a NAT gateway or a VPC endpoint) — S3 isn't "in" the network with your server. Keep this in your pocket; it explains a confusing class of "why can't my app reach the bucket" problems.
  • A few services are global (IAM, Route 53, CloudFront, Organizations). Everything else is Regional. When the console says "N. Virginia" in the corner, global services ignore it.

2.3 — There is a second hierarchy: permission

The ladder above is the hierarchy of place. Sitting across it is a hierarchy of permission — who is allowed to do what. They are independent: a resource can be physically reachable (place) and still refuse an action (permission), or vice-versa. You debug them separately. We give permission its own full treatment in §12 (IAM), but here is the shape so the two hierarchies live side by side in your head from the start:

Two independent ladders. NEXA's EC2 app needs both to read a document: a network path to S3 (place) and an IAM role allowing s3:GetObject (trust). Miss either and the read fails — but with different error messages, which is how you tell them apart.

3Regions & Availability Zones — choosing geography

A Region is a named geographic area containing multiple isolated data centres. An Availability Zone is one (or a small cluster) of those data centres. You always pick a Region; within it you spread across AZs for resilience. This is the first real decision in NEXA's plan, so it's worth getting the reasoning right.

3.1 — Why two levels and not one

AWS splits geography into two levels for two different jobs:

LevelJob it solvesNEXA relevance
Region (far apart, e.g. Frankfurt vs Mumbai)Latency & law. Put data near users; keep data inside a legal boundary.Users in US + India; current data sits in Frankfurt. The plan's open question is whether residency/latency justifies moving Region.
AZ (close together, same Region)Fault tolerance. A power/cooling/network failure in one data centre shouldn't end your service.You spread the ALB and the RDS subnet group across ≥2 AZs so a single-DC outage doesn't take NEXA offline.

The mental shortcut: Regions are for users and lawyers; AZs are for uptime. They're chosen for completely different reasons, which is why AWS keeps them as separate levels of the ladder.

3.2 — The NEXA Region decision, laid out honestly

This is Decision #1 in the briefing. There is no single right answer; there's a trade you make explicitly with the SDE. The variables:

eu-central-1 (Frankfurt)
Matches where the data is today (fra1). Zero residency surprises if the firm already accepted EU hosting. Worst latency for both US and India.
ap-south-1 (Mumbai)
Best for the India user base. If most active users are in India, this is the latency win. Check whether US client data is allowed to sit in India.
us-east-1 (N. Virginia)
Best for US users; the cheapest, most feature-complete Region. Worst for India. Often the default for US-headquartered firms.
How to actually decide For an internal tool with ~55 low-concurrency users, latency differences of 100–200 ms are barely perceptible on a server-rendered app — so this is mostly a data-residency and firm-standard decision, not a performance one. Ask: (1) Is there a contractual or legal rule about where client financial data may live? That dominates. (2) Does the firm already standardise on a Region in AUXO? Match it. (3) Only if both are open, optimise for the larger active user base. Pick one Region and keep all of NEXA's pieces in it — cross-Region wiring is a tax you don't want for an app this size.
Gotcha: Region is sticky Most resources cannot move Regions. An RDS instance, an EC2 instance, an ACM certificate, an ALB — all are created in a Region and stay there. "Moving Region" later means rebuilding. This is exactly why it's Decision #1 and why your plan flags it as a long pole. Because NEXA's current data is throwaway, you get one cheap chance to choose; spend it deliberately.

3.3 — Spreading NEXA across AZs (what it actually means)

Concretely, "multi-AZ" for NEXA means:

One Region · two Availability Zones
AZ  eu-central-1a
Public subnet 10.0.1.0/24 → EC2 app server
Private subnet 10.0.11.0/24 → RDS primary
AZ  eu-central-1b
Public subnet 10.0.2.0/24 → ALB node
Private subnet 10.0.12.0/24 → RDS standby slot

The ALB always spans ≥2 AZs (that's how it stays up). The RDS subnet group must list subnets in ≥2 AZs even if you run a single-AZ database — AWS needs the second AZ available so it could fail the DB over later. The single EC2 app server starts in one AZ; if you later move to ECS/ALB with 2 tasks, you'd put one task per AZ.

4VPC & subnets — building NEXA's private network

A VPC (Virtual Private Cloud) is your own private, isolated network inside a Region. Think of it as renting an empty data-centre floor: AWS gives you the room and the address space; you decide how to partition it, what's wired to the internet, and what's locked away. Everything compute- or data-related in NEXA plugs into this network.

4.1 — Addresses: CIDR, the language of the network

A VPC is defined by a CIDR block — a range of private IP addresses, written like 10.0.0.0/16. The /16 says "the first 16 bits are fixed; the rest are mine to hand out," which gives you 65,536 addresses (10.0.0.0 → 10.0.255.255). You then carve that range into smaller subnets, each a slice like 10.0.1.0/24 (256 addresses).

The /16 ÷ /24 mental model The number after the slash is "how many bits are locked." Bigger number = smaller range. A /16 VPC split into /24 subnets gives you up to 256 subnets of 256 addresses each — far more than NEXA needs, which is exactly why it's the comfortable default. You'll use a handful: two public, two private.

4.2 — Public vs private subnets: the one distinction that matters

A subnet is just a range of addresses pinned to one AZ. What makes it "public" or "private" is not a checkbox on the subnet itself — it's whether its route table has a path to the internet. That subtlety trips up everyone once:

Subnet is "public" if…Subnet is "private" if…
Its route table sends internet-bound traffic (0.0.0.0/0) to an Internet Gateway (IGW). Resources here can have public IPs and be reached from outside.Its route table has no direct route to an IGW. Resources here have only private IPs and cannot be reached from the internet at all.
NEXA: the EC2 app server and the ALB live here — they must be reachable.NEXA: RDS lives here — it must never be reachable from outside.

So "public/private" is a property of routing, covered next in §5. For now, the design intent: put the front door in public, lock the valuables in private.

4.3 — NEXA's network, drawn out

This is the two-tier layout from the briefing, made concrete. Study it — almost every later section refers back to this single picture.

NEXA's two-tier VPC. Traffic enters through the IGW → ALB (public) → EC2 app (public) → RDS (private). The database has no path back to the IGW, so even if someone learned its address they could not reach it from the internet. This single property — "the DB is in a private subnet" — is the backbone of the finance-data security story.

4.4 — Existing VPC or new? (Decision #2)

The briefing asks whether NEXA goes into an existing AUXO VPC or a fresh one. The trade:

Reuse an existing AUXO VPCNew dedicated VPC for NEXA
ProsFollows firm CIDR/subnet standards; shared egress/inspection already set up; less for you to design.Clean isolation; you control the whole layout; no risk of colliding with other workloads.
ConsMust fit the firm's existing address plan and rules; shared blast radius.You own more setup; must coordinate non-overlapping CIDR if it ever needs to peer.
Pick when…The firm has a standard landing-zone VPC and wants everything in it.NEXA is going into its own sub-account, or the firm has no standard yet.

For a newcomer, defer to the SDE's standard. If AUXO has a landing zone, slot into it and inherit the network conventions; you'll spend your learning budget on the app-specific pieces (RDS, S3, IAM) rather than re-deriving a network design the firm already made.

5Routing — how a subnet reaches (or can't reach) the internet

Routing is what actually makes a subnet "public" or "private." Each subnet has a route table: a list of "for this destination, send traffic there." Two special destinations — the Internet Gateway and the NAT Gateway — control internet access in opposite directions.

5.1 — The two gateways, and the direction each one opens

GatewayWhat it doesDirection
Internet Gateway (IGW)The VPC's door to the public internet. A subnet whose route table points 0.0.0.0/0 at the IGW is, by definition, public. Allows traffic in and out.Inbound and outbound
NAT GatewayLets resources in a private subnet make outbound connections (e.g. download OS updates, call an external API) without being reachable from the internet. One-way valve.Outbound only
The valve analogy An IGW is a two-way door. A NAT gateway is a one-way valve: things inside can reach out, but nothing outside can reach in. Private resources that need to initiate calls outward (but must stay unreachable) sit behind NAT. Resources that need to be reached sit in public subnets behind the IGW.

5.2 — Why NEXA's plan says "NAT Gateway: not required"

The briefing notes NAT isn't needed, and the reasoning is a clean illustration of the whole routing model:

The app server is in a public subnetThe EC2 box already has a route to the IGW for its outbound needs (OS patches, calling Azure AD, calling MS Graph). It egresses directly through the IGW — no NAT needed.
RDS doesn't need outbound internetThe database only ever receives connections from the app on port 5432. It never calls out to the internet, so it has no reason for a NAT gateway either.
S3 access goes through the app, not the DBBecause the app server is public, it reaches S3 over its normal egress. (If you later moved the app into a private subnet for tighter security, you'd then need either a NAT gateway or — better — an S3 VPC endpoint so it could reach the bucket without touching the public internet.)
Cost footnote worth knowing NAT gateways bill per hour and per GB processed — they're a common surprise line on AWS bills. Avoiding one (as NEXA does, by keeping the app public) is a legitimate simplification for a small internal tool. If security review later pushes the app into a private subnet, prefer a VPC endpoint for S3 over a NAT gateway: it's cheaper and keeps S3 traffic off the public internet entirely.

5.3 — Route tables, concretely

What each subnet's route table says

Public subnet route table

# local VPC traffic stays internal
10.0.0.0/16   → local
# everything else → the internet
0.0.0.0/0     → igw-xxxx

Private subnet route table

# local VPC traffic stays internal
10.0.0.0/16   → local
# NO 0.0.0.0/0 route at all
# → cannot reach the internet

The presence or absence of that one 0.0.0.0/0 → igw line is the entire difference between public and private. RDS sits where that line doesn't exist.

6Security Groups & NACLs — the network firewalls

Routing decides where traffic can flow. Security Groups decide what's allowed to flow. A security group is a stateful firewall attached to a resource (an EC2 instance, an RDS database, an ALB) that says which traffic may come in and go out. This is the first half of "the hierarchy of trust" — the network half.

6.1 — The single most important trick: groups reference groups

A newcomer's instinct is to allow traffic by IP address: "let 10.0.1.5 talk to the database." AWS lets you do something far better — allow by security group. NEXA's design is the canonical example:

NEXA's two security groups
App Security Group  (on EC2 + ALB)
Inbound: 443/80 from the internet (or from the ALB SG)
Outbound: allowed (to reach RDS, S3, Azure, Graph)
▼   the app SG is named as the source in the DB rule
DB Security Group  (on RDS)
Inbound: port 5432 only from the App Security Group
Outbound: default
— no IPs hard-coded, nothing else can connect

The RDS rule says "accept 5432 from anything wearing the App SG badge." You never write an IP. When the app server's IP changes (a restart, a replacement, a second instance), the rule still works because membership — not address — is what's checked. This is why SG-references-SG is the AWS-idiomatic way and IP allow-lists are the amateur way.

6.2 — Stateful: you only write one direction

Security groups are stateful, which removes a whole category of mistakes. If you allow an inbound request, the response is automatically allowed back out — you don't write a matching outbound rule. So for RDS you write exactly one line ("inbound 5432 from App SG") and the query results flow back without a second rule. (NACLs, below, are the stateless exception.)

6.3 — Security Groups vs NACLs (and why you'll mostly ignore NACLs)

Security GroupNetwork ACL (NACL)
Attached toA resource (instance, RDS, ALB)A whole subnet
StateStateful — return traffic auto-allowedStateless — you must allow both directions
RulesAllow onlyAllow and deny, evaluated in order
Can reference a group?Yes (the killer feature)No — IP ranges only
You use it for…Almost everything. This is your primary tool.Coarse subnet-wide guardrails; an extra deny layer. Often left at default.
Practical guidance for NEXA Do all your work with security groups; leave NACLs at their default "allow all" unless the firm's security standard requires an explicit subnet-level deny. For an app this size, two well-written security groups (App SG, DB SG) plus the default NACL is the correct, idiomatic setup. Don't over-engineer the NACL layer — it's a common newcomer time-sink with little payoff here.
The classic first-day error "I can't connect to my RDS." Nine times out of ten it's the DB security group: either it doesn't reference the app's SG, or it's referencing the wrong one. The second most common cause is the database being marked publicly-accessible=false (correct!) while you try to connect from your laptop (which isn't in the VPC). For NEXA, the app connects from inside the VPC, so this is right by design — but it means you debug DB connectivity from the EC2 box, not your laptop.

7EC2 — the box that runs the app

EC2 (Elastic Compute Cloud) is a virtual machine you rent by the hour: a Linux box with CPU, RAM, and disk, living in one subnet of your VPC. For NEXA it's the single most important compute decision — it's the thing that actually runs next start and serves every page and API call. It replaces what Vercel did invisibly.

7.1 — Why NEXA needs a real server at all

This catches people coming from Vercel: Next.js 16 is not a static site. It needs a live Node process running next start to server-render pages and handle the API routes. Vercel ran that process for you on ephemeral serverless instances. On AWS, something has to run it continuously — and the simplest "something" is one EC2 instance. (The container alternative, ECS Fargate, is the same idea with the VM hidden; see §15/§17.)

7.2 — Anatomy of the NEXA instance

Instance type (CPU/RAM)t3.large (2 vCPU / 4 GB) to start; t4g.large for headroom4 GB is the comfortable floor for SSR. t-family is "burstable" — cheap, good for spiky internal load. t4g is ARM/Graviton, ~20% cheaper, and Node runs on ARM fine.

7.3 — Burstable instances (the t-family), explained

NEXA uses t3/t4g instances, which work on a credit system worth understanding so a billing or performance surprise doesn't blindside you:

How burstable works A t-instance earns CPU credits while idle and spends them when busy. For an internal tool that's quiet most of the day with occasional spikes (someone opens the pricing editor, an Excel export runs), this is ideal and cheap — you're not paying for full CPU you rarely use. The watch-out: a sustained heavy workload can exhaust credits and throttle. NEXA's one sustained-CPU risk is next build and the synchronous Excel parse — which is exactly why the plan says build in CI, not on the box (next note).
The OOM trap — don't build on the box next build is memory-hungry and will OOM-kill on a small instance. NEXA's plan handles this correctly: build the artifact in GitHub Actions (CI) and ship it to the instance, so the runtime box can stay a modest 4 GB. If you ever must build on the instance, add swap temporarily. Also note the synchronous xlsx parse and the 5000-row export run on the request thread and briefly block other requests on a single Node process — fine at NEXA's scale, but know it exists.

7.4 — Keeping the process alive: systemd / PM2

Vercel restarted crashed functions for free. On EC2 you arrange that yourself with a process manager: either systemd (built into Linux) or PM2 (Node-specific). Either way the contract is "if next start dies or the box reboots, bring it back automatically."

# systemd unit (sketch): /etc/systemd/system/nexa.service
[Service]
ExecStart=/usr/bin/node /opt/nexa/server.js   # or: next start
Restart=always                                # ← the crucial line
EnvironmentFile=/opt/nexa/.runtime-env        # pulled from SSM at boot
User=nexa
[Install]
WantedBy=multi-user.target
NEXA's plan, in one line EC2 + Nginx in front (TLS + reverse proxy) + systemd/PM2 keeping next start alive + a stable address (Elastic IP or ALB). That stack reproduces, piece by piece, what Vercel did automatically — and it's the easiest place to get the SSO cookie behaviour right, which is why it's the recommended starting point over jumping straight to containers.

7.5 — The TLS-and-cookies gotcha (the one that breaks login)

This deserves its own flag because it's the single most likely thing to break NEXA's login after migration:

Trust the proxy, or secure cookies vanish TLS terminates at Nginx/ALB, so the Node app receives plain http internally. If you don't tell it "you're really behind HTTPS," NextAuth won't set the Secure session cookie correctly and SSO silently fails. The fix is two settings: forward X-Forwarded-Proto: https from the proxy, and set NEXTAUTH_URL=https://nexa.procdna.com. This is audit item A2 in your plan — treat it as a release-blocker, because everything else can be perfect and login will still fail without it.

8ALB — the load balancer and stable front door

An Application Load Balancer sits in front of your app server(s), accepts incoming HTTPS, and forwards requests to healthy targets. For NEXA its job is less "balance load" (there's barely any) and more "be a stable, AZ-spanning front door with clean TLS" — which turns out to be the thing that protects login and email.

8.1 — The vocabulary (three nouns)

TermWhat it isNEXA
Listener"Accept traffic on this port/protocol." Usually HTTPS:443 (with the ACM cert) and a HTTP:80 → 443 redirect.443 with the nexa.procdna.com cert; 80 redirects to 443.

Health checkA URL the ALB polls; unhealthy targets stop receiving traffic.A /healthz route the app exposes (the plan calls for adding one).

8.2 — Why an ALB instead of just an Elastic IP (Decision #3)

The briefing asks: single EC2 + Elastic IP (cheapest) vs ALB (cleaner). Here's the honest comparison for an app whose hard constraint is a stable public URL:

Recommendation for NEXA Start with the ALB unless cost is genuinely scrutinised. It gives you ACM auto-renewing TLS (no manual cert chores), an attach point for WAF/rate-limiting (the plan wants this for the secret-bearing endpoints), and a painless path to ECS later — all for the price of one small fixed monthly charge. The Elastic-IP route is defensible for a pure cost-minimised single box, but you'll likely re-introduce an ALB the moment you want HA or WAF, so doing it now avoids a migration. Either way: the public DNS name must be stable, because Microsoft SSO redirect URIs and email Safe Links are registered against it.

9Route 53 & ACM — the name and the certificate

Two small services that together give NEXA its stable, trusted public identity. Route 53 is AWS's DNS — it maps a human name (nexa.procdna.com) to your ALB. ACM (Certificate Manager) issues and auto-renews the TLS certificate that makes the browser show a padlock. They're the literal answer to "what's the URL" — which, for NEXA, is the long-pole constraint.

9.1 — Route 53: hosted zones and records

ConceptMeaningNEXA
Hosted zoneA container for all DNS records of a domain (procdna.com).Likely owned by firm IT. You need a record inside it.
A / ALIAS recordMaps a name to an address. An ALIAS to an ALB is the AWS-native way (no IP to manage).nexa.procdna.com → ALIAS → the ALB.
CNAMEMaps a name to another name.Used during ACM validation (next).
Decision #6 is a people problem, not a tech one The plan flags domain ownership as the long pole — and it's right. Whoever controls procdna.com (firm IT) must create the subdomain record. This often takes longer than any AWS step because it crosses a team boundary and a change-approval process. Start this conversation on day one, because SSO and email can't be finalised until the name exists. You don't necessarily need Route 53 itself — if the firm runs DNS elsewhere, they just add a record pointing at the ALB. Route 53 is simply the option if the zone lives in AWS.

9.2 — ACM: the certificate, and the validation dance

ACM issues a free TLS certificate for nexa.procdna.com, attaches it to the ALB listener, and auto-renews it forever — no more expiry-day fire drills. The one step that confuses newcomers is proving you own the domain:

requestAsk ACM for a cert covering nexa.procdna.com. ACM gives you a special CNAME record to add.
validateWhoever owns the DNS zone adds that CNAME. This proves to ACM "yes, this domain is ours." (DNS validation is preferred because it's what enables auto-renewal.)
attachOnce validated, attach the cert to the ALB's 443 listener. The padlock appears; renewals now happen automatically.
Region gotcha An ACM certificate must be issued in the same Region as the ALB that uses it. (The exception is CloudFront, which only reads certs from us-east-1 — not relevant to NEXA's ALB setup.) So issue NEXA's cert in whatever Region you chose in §3. Mismatched Region is a quiet, common stumble.

9.3 — Why this trio is the SSO long-pole

The constraint that drives the whole migration NEXA's Microsoft SSO redirect URIs and email Safe Links are registered against a fixed domain. Nothing external may ever point at a raw EC2 IP, because the moment the address changes, login and email break. Route 53 + ACM + ALB solve this once: a stable name, a trusted cert, an AZ-spanning front door. After that, an EC2 stop/start, a redeploy, or a scale event never disturbs login. And critically — the new redirect URI must be added to the Azure App Registration before go-live, or SSO breaks the instant DNS flips. Pre-register it; don't discover this at cut-over.

10RDS — the managed PostgreSQL database

RDS (Relational Database Service) is AWS running PostgreSQL for you: it provisions the engine, patches it, backs it up, and can fail it over — you just get an endpoint and a password. It's the direct replacement for Supabase's managed Postgres, and it holds everything in NEXA except the documents: users, clients, opportunities, pricing versions, approvals, rate cards.

10.1 — "Managed" means AWS owns the boring parts

The point of RDS over "install Postgres on an EC2 box yourself" is that AWS handles the operational toil:

Backups
Automated daily snapshots + point-in-time recovery, retention you set
Patching
Engine minor-version patches in a maintenance window
Failover
Optional Multi-AZ standby that takes over automatically

Supabase did all of this invisibly. On RDS you now own the policy (how many days of backups, whether Multi-AZ) — but AWS still does the mechanics.

RDS requires a DB subnet group — a named set of subnets spanning at least two AZs. This holds even if you run a single-AZ database, because AWS needs a second AZ available so it could place a standby. This is why §4's diagram shows a "standby slot" private subnet in AZ-b even though the primary lives in AZ-a.

Single-AZ vs Multi-AZ (Decision #5)
Single-AZ

Multi-AZ

For NEXA (internal, low-stakes-uptime), single-AZ with solid backups is a reasonable start; Multi-AZ is a policy call for the SDE. Either way the subnet group needs two AZs.

10.4 — Two things that get simpler, one that needs care (migration notes)

Simpler: no more pooler gymnastics Supabase forced the pgBouncer transaction pooler (port 6543) because Vercel opened many short-lived serverless connections. A single long-lived Node process on EC2 doesn't do that — so point Prisma straight at RDS on 5432, and DATABASE_URL / DIRECT_URL can converge on one endpoint. You also get to run real migrations (prisma migrate deploy) instead of hand-applying SQL.
Care: the connection pool cap (audit A1) lib/prisma.ts hard-caps the pool at max=1 in production — correct for Vercel, but a severe bottleneck on one long-lived EC2 process, where the whole app would serialise through a single DB connection. Raise it (e.g. 10), sized to the RDS instance's connection limit. This is a one-line change that's easy to miss and produces baffling "the app feels slow under any concurrency" symptoms if you don't.

10.5 — Cut-over: there's no data to move

Because NEXA's current data is throwaway, the database migration is unusually easy — you build the schema fresh and seed, rather than copying rows:

# point Prisma at RDS, then build the schema and bootstrap
DATABASE_URL="postgresql://USER:PW@nexa.xxxx.eu-central-1.rds.amazonaws.com:5432/nexa?schema=procdna_database&sslmode=require"
npx prisma migrate deploy          # creates tables/enums fresh
npx tsx prisma/add-admin.ts        # bootstrap the first ADMIN
npx tsx prisma/sync-azure-users.ts # pull real BD users from Azure AD
The env-var-name trap (audit A10) prisma/sync-azure-users.ts reads AZURE_TENANT_ID / AZURE_CLIENT_ID / AZURE_CLIENT_SECRET (no _AD_), while the app uses AZURE_AD_*. Provide both naming styles in the EC2 environment or the sync silently exits and you get an empty user table with no error. This is the kind of thing that costs an afternoon if you don't know it's there.

11S3 — object storage for the documents

S3 (Simple Storage Service) stores files — not in a filesystem and not in the database, but as named "objects" in a "bucket." It's where NEXA's signed SOW and PO documents live, replacing Supabase Storage. Of all the migration pieces, S3 is the one that requires real code changes (rewriting the storage layer), and it's also the one with the single highest-impact security footgun.

11.1 — Buckets, keys, objects (and why S3 isn't in your VPC)

TermMeaningNEXA
BucketA top-level container with a globally-unique name, living at the Region level.One private bucket, e.g. nexa-docs.
KeyThe object's full "path"-like name. S3 has no real folders — the slashes are just part of the key.<oppId>/<version>_<ts>.pdf
ObjectThe file itself plus metadata.A SOW or PO PDF.
Remember the ladder exception S3 sits above the VPC (§2.1), not inside it. That's why your EC2 app reaches it over normal egress, and why a future private-subnet app would need a NAT gateway or — better — an S3 VPC endpoint. S3 access is gated by IAM, not security groups, precisely because it's not on your network.

11.2 — The NEXA S3 spec

SettingNEXA value
Public accessBlock Public Access = ON. Files are only ever reached via short-lived presigned URLs.
EncryptionDefault SSE (SSE-S3 or SSE-KMS) — encryption at rest by default.
VersioningON — recovers an overwritten/deleted document (the app soft-deletes; S3 versioning is the safety net).
Access from appIAM instance role granting s3:GetObject/PutObject on this one bucket. No AWS keys in .env.
URLs@aws-sdk/s3-request-presigner, expiresIn: 3600 — the 1-hour expiry the app already assumes.

11.3 — Presigned URLs: the pattern that keeps the bucket private

This is the elegant core of NEXA's document handling. The bucket is never public; instead, the app — which is allowed to touch S3 via its IAM role — mints a temporary, signed URL for one specific object, valid for one hour. The browser uses that URL directly. Files never stream through the app server in bulk, and the link dies after an hour.

Upload, then view — both via short-lived signed URLs
Uploadbrowser asks app → app issues presigned PUT → browser uploads straight to S3
the file goes browser → S3 directly; the app never holds the bytes
Viewbrowser opens doc → app issues 1-hour presigned GET → browser fetches from S3
link expires in 60 min; bucket stays private the whole time

The app authenticates to S3 with its instance role (no stored keys), and hands out scoped, expiring links. This is why "Block Public Access = ON" and "presigned URLs" are two halves of one design — together they mean a confidential SOW is reachable only by someone the app just handed a fresh link to.

The single highest-impact new risk in the whole migration One "Block Public Access = off" click and every confidential pricing/SOW document becomes world-readable. There is no scarier button in this project. Keep buckets private; serve only via presigned URLs; and constrain the upload presign to a content-type + size limit (which also closes the "client-claimed MIME" weakness, audit S11). If you set up one guardrail in this migration, make it an account-level "block public access" enforcement so this can't be toggled by accident.

11.4 — The code that actually changes

S3 is the one part that's real engineering, not just configuration:

  • lib/supabase.ts → a new lib/s3.ts: getSupabase() becomes an S3 client; getSignedUrl() becomes the presigner. Keep the 55-minute cache but add eviction — on a long-lived process the current unbounded cache grows forever (audit A5).
  • The four document routes (…/sow, …/sow/confirm, …/po, …/po/confirm): swap "signed upload URL" calls for S3 presigned PUT.
  • Env: drop SUPABASE_*; add S3_BUCKET + region. Credentials come from the instance role, not env — that's the whole point of §12.

12IAM — the hierarchy of permission

IAM (Identity and Access Management) is how AWS answers one question millions of times a second: "is this identity allowed to perform this action on this resource?" It's the second of the two great hierarchies (place was §2; this is trust). It's also the part that feels most alien coming from Vercel, where the platform handled identity for you. Get IAM right and the whole "no keys on disk" security story falls into place; get it wrong and you either can't deploy or you've left the door open.

12.1 — The four nouns of IAM

NounWhat it isNEXA example
PrincipalThe "who" — an identity that makes requests. A user, or (better) a role.The EC2 app server; the GitHub Actions deployer; you.
PolicyA JSON document listing allowed (or denied) actions on resources. The "what's permitted.""Allow s3:GetObject on nexa-docs/*."
RoleAn identity with policies attached that something assumes temporarily — no long-lived password.The NEXA instance role the EC2 box wears.
Action / ResourceThe verb (s3:PutObject) and the noun (a specific bucket/ARN) a policy grants.ssm:GetParameter on the /nexa/* path.

12.2 — Roles, not keys: the idea that makes this secure

The most important IAM concept for NEXA is the difference between a user (has a permanent password/access key) and a role (assumed temporarily, credentials auto-rotate). The migration's security depends on using roles everywhere a machine acts:

Why "no AWS keys in .env" is possible at all When the EC2 instance wears an instance role, the AWS SDK inside the Node app automatically receives temporary, auto-rotating credentials from the instance metadata — it never needs a stored access key. So lib/s3.ts calls S3 with zero secrets on disk. Compare the alternative — pasting an access key into .env — which would then bake into AMIs, snapshots, and backups forever. Roles eliminate an entire class of leak. This is the single biggest security upgrade the AWS move buys you.

12.3 — The trust hierarchy, drawn

A role has two policies, and conflating them is the #1 IAM confusion. One says who may wear it (the trust policy); the other says what the wearer may do (the permission policy). Here is NEXA's app role as the worked example:

NEXA's app role. The trust policy lets the EC2 service assume it; the permission policy grants exactly three things — read/write the one bucket, read the app's secrets, write its own logs — and nothing else. That "nothing else" is least privilege: the blast radius of a compromised app is bounded to precisely those three resources.

12.4 — Least privilege, made concrete

"Least privilege" means granting the minimum that works, scoped to specific resources — never "Action": "*" on "Resource": "*". NEXA's app policy is a good template:

{
  "Version": "2012-10-17",
  "Statement": [
    { "Effect":"Allow", "Action":["s3:GetObject","s3:PutObject"],
      "Resource": "arn:aws:s3:::nexa-docs/*" },          // only this bucket
    { "Effect":"Allow", "Action":["ssm:GetParameter","ssm:GetParameters"],
      "Resource": "arn:aws:ssm:*:*:parameter/nexa/*" },      // only /nexa/* secrets
    { "Effect":"Allow", "Action":["logs:CreateLogStream","logs:PutLogEvents"],
      "Resource": "arn:aws:logs:*:*:log-group:/nexa/*" }      // only its log group
  ]
}
Read this as three sentences "This identity may read & write objects in nexa-docs only; read parameters under /nexa/ only; and write to its own log group only." Everything else — other buckets, other secrets, deleting things, touching RDS's control plane — is denied because it isn't listed. That scoping is the security.

12.5 — The IAM you'll actually set up for NEXA (the briefing's §6)

PrincipalGetsWhy
EC2 instance role (the app)s3:GetObject/PutObject on the bucket; ssm:GetParameter* on /nexa/*; logs:* on its groupApp reads/writes documents and pulls secrets without static keys.
GitHub Actions role (deploy)OIDC-federated role scoped to the deploy action (push image to ECR / update the service / SSM-run a pull)Keyless CI/CD from the ProcDNA repo — no long-lived AWS creds in GitHub.
Your build accessPermission to create VPC/EC2/RDS/S3 in a dev sub-account, or the SDE pre-creates them and hands you endpoints + the instance roleSo you can build without holding god-mode on the firm's production account.

12.6 — OIDC federation: how GitHub deploys with zero stored keys

This is the slickest IAM pattern in the plan and worth understanding because it's the modern default. Instead of storing an AWS access key in GitHub (which could leak), GitHub Actions and AWS establish trust directly: AWS is told "trust tokens issued by GitHub for the ProcDNA/nexa repo," and at deploy time GitHub presents a short-lived identity token that AWS exchanges for temporary credentials.

Keyless deploy via OIDC
GitHub Actions run (push to main)requests a short-lived OIDC token from GitHub
▼ presents token to AWS STS
AWS checks the trust policy"is this token from ProcDNA/nexa, branch main?" → yes
▼ issues temporary credentials (minutes)
Actions deploysupdates the ECS service / triggers an EC2 pull — then the creds expire

No AWS secret ever lives in GitHub. The credentials exist for the length of one deploy and evaporate. This is the "OIDC federation preferred over stored AWS keys" line in your plan, unpacked.

IAM newcomer pitfalls (1) Confusing the two policies — "why can't EC2 assume the role?" is a trust-policy problem; "why does the app get AccessDenied on S3?" is a permission-policy problem. (2) Over-granting "to make it work" — resist s3:* on *; start minimal and add the specific action the error names. (3) Putting an access key in .env "just for now" — that "now" becomes forever and bakes into snapshots. Use the instance role from day one.

13Secrets & KMS — where the passwords live

NEXA has a fistful of secrets that are not AWS credentials: the database password, the Azure client secret, NEXTAUTH_SECRET, the HMAC mail/approval secrets. On Vercel these were env vars in a dashboard. On AWS they go into a managed, encrypted secret store — and the app pulls them at boot using its instance role. KMS is the encryption engine underneath.

13.1 — Parameter Store vs Secrets Manager (Decision #7)

SSM Parameter StoreSecrets Manager
What it isA key/value store; values can be KMS-encrypted ("SecureString").A purpose-built secret store with built-in rotation.
CostStandard params are freePer-secret monthly charge

Recommendation For NEXA's handful of secrets, SSM Parameter Store (SecureString) is the pragmatic default — free, simple, KMS-encrypted, and the instance role already grants ssm:GetParameter* on /nexa/*. Reach for Secrets Manager specifically if the firm wants automatic rotation of the RDS password. Either way the principle is identical: secrets are pulled at runtime by the role, never written into the AMI, the repo, or a plaintext .env that snapshots would preserve.

13.2 — KMS in one paragraph

KMS (Key Management Service) holds the encryption keys that protect everything "encrypted at rest" in this whole course — RDS storage, S3 objects, SSM SecureStrings. You rarely touch it directly; you just choose "encrypt with KMS" on each service and AWS handles the cryptography. The value is that access to the keys is itself IAM-controlled and audited, so "encrypted at rest" isn't a checkbox lie — decrypting genuinely requires permission KMS can log.

How a NEXA secret flows at boot

All current NEXA secrets are rotated during the move (the ideal moment), then placed under /nexa/*. The app fetches them at startup via the role — closing the loop with §12.

14CloudWatch — logs, metrics, and "is it alive?"

CloudWatch is AWS's observability service: it collects logs, metrics, and alarms. This matters more than newcomers expect, because Vercel's logging disappears the moment you leave it — without CloudWatch you'd be flying blind on the new infrastructure.

14.1 — The three things you'll use

FeatureWhat it gives NEXA
LogsThe app's stdout shipped off the box into a /nexa/* log group, searchable and retained — so a crash at 2am is still there in the morning. (The instance role grants the log permissions; §12.)

14.2 — The health check closes the loop

Recall §8: the ALB polls a /healthz route to decide if the app is alive. That route plus CloudWatch logs is your minimum viable operations setup — the ALB pulls a sick instance out of rotation automatically, and CloudWatch tells you why it was sick. Add a /healthz endpoint to NEXA as part of the migration (the plan calls for it); it's a few lines and it's what makes the whole self-healing story work.

What "production-ready ops" means for an app this size You do not need a heavyweight observability stack. For NEXA, "good enough and genuinely solid" is: app logs → CloudWatch, a /healthz route the ALB checks, RDS automated backups verified, and one or two alarms (5xx spike, no-healthy-targets). That's a proportionate, defensible operations posture for an internal tool — resist the urge to over-build it.

16Structuring a project like this — the hierarchies that matter

You asked to focus on hierarchy in a project like NEXA. There isn't one hierarchy — there are four, stacked on top of each other, and a well-run project keeps them aligned. This section names each, shows how NEXA expresses it, and — importantly — shows how they connect, because the magic is in the alignment.

The four hierarchies of a project like NEXA

16.1 — Hierarchy A → B: infrastructure layers map onto code layers

The cleanest projects make the network tiers and the code tiers line up one-to-one. NEXA does this naturally, which is why it's a good teaching example:

EDGE LAYERALB + ACM + Route 53 · TLS, routing, health, WAF
▼

Each infrastructure tier hosts exactly one code tier, and dependencies flow strictly downward — edge calls app, app calls services, services call data; never the reverse. That "state lives only in the data layer" property is what makes the app layer stateless, which in turn is what makes it horizontally scalable and easy to containerise later (§17). The hierarchy isn't bureaucracy — it's what buys you the scaling story.

16.2 — Hierarchy C: two permission systems, one for machines, one for humans

A subtlety newcomers miss: a real project has two permission hierarchies running in parallel, and they're deliberately separate.

Machine permissions = IAM (§12) Governs what code/servers may do to AWS resources. "The EC2 app may read the bucket and the secrets, nothing else." Enforced by AWS, outside the app.
Human permissions = app RBAC Governs what users may see/do inside NEXA. The SEL → Director → ED → Partner → Admin ladder, enforced by resolveOwnerFilter() in the code.

These are independent and that's correct: tightening one never weakens the other. A compromised user account is bounded by RBAC; a compromised server is bounded by IAM. Defence in depth comes from keeping them separate. Here is NEXA's human hierarchy, since it's the one your domain cares about:

The base case — an individual contributor sees the opportunities they own.

The crucial design point: this hierarchy is data-driven, not hard-coded. A user's role + managerId turns into a Prisma where filter at query time (resolveOwnerFilter). That's why §15's request trace has an "RBAC" step — every read is scoped by the requester's rung. (And the audit's warning: the read side is well-built, but the write side needs the same guard mirrored onto it — the IDOR release-blocker.)

17CI/CD & the path from EC2 to containers

The last operational layer: how code gets from the ProcDNA repo onto the running infrastructure, and how NEXA can graduate from a single EC2 box to containers when it's ready. This ties the GitHub work (your plan's §1) to the AWS work.

17.1 — The deploy pipeline

pushMerge to main in the ProcDNA org repo (branch-protected, review required) triggers GitHub Actions.
buildActions builds the Next.js artifact in CI — keeping the memory-hungry next build off the small instance (§7.4).
authenticateActions assumes its AWS role via OIDC — short-lived creds, no stored keys (§12.6).
deployEC2 path: ship the artifact and restart the systemd service. ECS path: push a new image to ECR and update the service for a rolling deploy.
verifyThe ALB health check (/healthz) confirms the new version is serving before old targets drain (ECS) — zero-downtime by construction.

17.2 — EC2 now, ECS Fargate later (Decision #4)

The plan recommends starting on EC2 and graduating to ECS Fargate once stable. Understanding why closes the loop on the whole course:

EC2 (start here)ECS Fargate (the upgrade)
What you manageThe OS — you patch it, you supervise the processNothing below the container — AWS runs it serverlessly
DeploysShip artifact + restart (brief blip possible)Rolling / blue-green, zero-downtime built in
ScalingBigger box, or wire up more boxes by handChange the task count — horizontal by design
Learning costLowest — easiest mental model for SSO/cookiesDocker, task defs, ECR up front
Best whenGetting live fast, fewest new conceptsApp is stable; you want no patching + clean deploys
Why the graduation is low-friction for NEXA Recall §16.1: the app is stateless — all state lives in RDS and S3. A stateless app is trivial to containerise, because a container can be killed and replaced at any moment with no lost data. So "EC2 first, Fargate later" isn't a rewrite; it's wrapping the same app in a Dockerfile and pointing the same ALB at tasks instead of an instance. The architecture you build in this course is already shaped for the upgrade — which is exactly the payoff of getting the layering (§16) right from the start.
A note to take to the SDE Decisions #3 (Elastic IP vs ALB) and #4 (EC2 vs ECS) interact. If there's any chance you'll want ECS later, choose the ALB now — ECS effectively requires one, so starting with the ALB means the only future change is swapping the target from "an instance" to "tasks." Match whatever the firm already supports; an org that standardises on containers may want you on ECS from day one.

18A learning path — what to actually do, in order

Reading is not the same as building. The fastest way to make all of this stick is to stand the pieces up yourself, smallest blast radius first, in a throwaway dev account. Here is an order that builds confidence layer by layer — each step uses what the last one created.

place — build the empty roomCreate a VPC (10.0.0.0/16) with two public and two private subnets across two AZs, an IGW, and route tables. Goal: you can point at the §4 diagram and say "I made that." No compute yet. (§4–5)
trust — write the firewallsCreate the App SG and DB SG, with the DB SG referencing the App SG (not an IP). You're practising the SG-references-SG pattern before anything depends on it. (§6)
data first — it has no dependenciesStand up RDS in the private subnet group, encrypted, not public. Confirm you can reach it only from inside the VPC. Create the S3 bucket with Block Public Access on. (§10–11)
identity — the role before the serverCreate the EC2 instance role with the least-privilege policy (bucket + /nexa/* secrets + logs). You'll attach it when you launch the box. (§12)
compute — launch the app boxLaunch one EC2 in the public subnet, wearing the role, in the App SG. Install Node + Nginx, run the app under systemd. Confirm it can reach RDS and S3 with no keys on disk. (§7)
edge — the public facePut an ALB in front, attach an ACM cert, point a (test) DNS name at it. Add /healthz. Confirm HTTPS works and the health check is green. (§8–9)
close the loop — secrets, logs, SSOMove config into SSM, confirm logs land in CloudWatch, register the test redirect URI in Azure AD and verify SSO end-to-end. Now you've reproduced the whole §15 diagram. (§13–14)
automate — then tear downWire GitHub Actions + OIDC to deploy. Then delete the whole dev stack and rebuild it from memory once — the rebuild is where it truly clicks. (§17)
The one habit that matters most When something doesn't work, first ask: is this a place problem or a trust problem? (§2.3). A timeout/refused-connection is almost always network (routing or security group). An "AccessDenied / not authorized" is almost always IAM. Naming which of the two ladders you're on cuts debugging time in half, because it tells you which console screen to open.

19Glossary — every term, one line each

Organization
The top container grouping all of a firm's AWS accounts under one bill and one set of guardrails (SCPs).
Account
The hard isolation boundary of AWS. Resources, permissions, and bills are scoped to it. NEXA targets the AUXO account.
Region
A geographic cluster of data centres (e.g. eu-central-1). Chosen for latency and data-residency law. Most resources can't move Regions.
Availability Zone (AZ)
An isolated data centre within a Region. Spread across ≥2 for fault tolerance.
VPC
Your private virtual network inside a Region, defined by a CIDR block like 10.0.0.0/16.
Subnet
A slice of the VPC's addresses pinned to one AZ; public (routes to the IGW) or private (doesn't).
CIDR
The notation for an address range; the number after the slash is how many bits are fixed (/16 = big, /24 = small).
Internet Gateway (IGW)
The VPC's two-way door to the internet. A subnet routing 0.0.0.0/0 to it is "public."
NAT Gateway
A one-way valve letting private resources make outbound calls without being reachable. NEXA doesn't need one.
Route table
The per-subnet list of "for this destination, send traffic there." Presence of an IGW route = public.
Security Group (SG)
A stateful firewall on a resource. Can reference other SGs as the source — the idiomatic way. NEXA: App SG, DB SG.
NACL
A stateless subnet-level firewall. Coarser than an SG; usually left at default.
EC2
A rented virtual machine in a subnet. Runs NEXA's Next.js process. t3/t4g are cheap burstable types.
AMI
The disk image an EC2 instance boots from (the OS template).
EBS
The virtual disk attached to an EC2 instance (gp3 is the standard SSD type).
Elastic IP
A fixed public IP you can keep across instance restarts; the cheap alternative to an ALB for a stable address.
ALB
Application Load Balancer — accepts HTTPS, terminates TLS, health-checks targets, spans AZs. NEXA's stable front door.
Listener / Target group / Health check
An ALB's three parts: the port it accepts on, the pool it forwards to, and the URL it polls (/healthz).
Route 53
AWS's DNS. A hosted zone holds a domain's records; an ALIAS record maps a name to the ALB.
ACM
Certificate Manager — issues and auto-renews the TLS cert for the ALB. Must be in the ALB's Region.
RDS
Managed relational database (PostgreSQL for NEXA). AWS handles backups, patching, optional failover.
DB subnet group
The set of subnets (≥2 AZs) RDS may live in — required even for a single-AZ database.
Multi-AZ
An RDS option adding a synchronous standby in another AZ for automatic failover (~doubles DB cost).
S3
Object (file) storage in named buckets, living above the VPC. Holds NEXA's SOW/PO documents.
Bucket / Key / Object
S3's container / the object's full name / the file itself.
Block Public Access
The S3 setting that keeps a bucket private. For NEXA, this must stay ON — the highest-impact safety switch.
Presigned URL
A temporary, signed link to one S3 object, valid for a set time (NEXA: 1 hour). Lets a private bucket serve files safely.
VPC endpoint
A private path from your VPC to a service like S3, avoiding the public internet (and NAT costs).
IAM
Identity and Access Management — decides if an identity may perform an action on a resource. Default is deny.
Principal
The "who" making a request — a user or (better) a role.
Policy
A JSON document granting (or denying) specific actions on specific resources.
Role
An identity something assumes temporarily, with auto-rotating credentials — no stored password. Use everywhere a machine acts.
Instance role
The role an EC2 box wears so its code reaches AWS services with no keys on disk. NEXA's core security win.
Trust policy vs permission policy
A role's two policies: who may wear it, vs what the wearer may do. Conflating them is the #1 IAM confusion.
Least privilege
Granting the minimum that works, scoped to specific resources — never *:* .
OIDC federation
Letting GitHub Actions assume an AWS role via short-lived tokens, so no AWS key is stored in CI.
SSM Parameter Store / Secrets Manager
Encrypted stores for non-AWS secrets (DB password, Azure secret). Pulled at runtime by the role.
KMS
Key Management Service — holds the keys behind every "encrypted at rest" (RDS, S3, SSM). Access is itself IAM-controlled.
CloudWatch
Logs, metrics, and alarms. Replaces Vercel's logging, which vanishes on migration.
ECS / Fargate / ECR
Container orchestration / the serverless way to run containers / the registry images live in. NEXA's optional upgrade from EC2.
WAF
Web Application Firewall — attaches to an ALB for rate-limiting and IP rules on the secret-bearing endpoints.

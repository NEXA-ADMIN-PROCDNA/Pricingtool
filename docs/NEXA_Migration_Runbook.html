AWS Migration Runbook

A detailed, phase-by-phase plan to move NEXA off Vercel + Supabase onto the firm's AUXO AWS account — what to do, in what order, who owns each step, the app-side code changes it requires, how to verify it, and how to roll back. Companion to the Migration Plan, the Architecture Briefing, the AWS course, and the Service Map.

AOverview & scope

NEXA is moving hosting providers, not changing its architecture. The same Next.js application, the same PostgreSQL data model, and the same Azure AD login are being re-platformed: Vercel → EC2, Supabase Postgres → RDS, Supabase Storage → S3, with the supporting AWS plumbing (VPC, IAM, ACM, Route 53, SSM, CloudWatch) that Vercel and Supabase previously hid.

What's in scope vs out of scope In scope: standing up the AWS infrastructure, the S3 storage-layer rewrite, connection/secret/cookie configuration changes, the schema + seed, SSO re-wiring, verification, cut-over. Out of scope for this runbook: feature changes, and the four application security release-blockers — which are tracked separately (§Release-blockers) and must be fixed before NEXA holds real client financial data, independent of the infrastructure move.
The single biggest simplifier The current data is throwaway test data. There is no data migration — no pg_dump/restore, no file copy. You recreate the schema fresh on RDS and start S3 empty. This removes the riskiest part of most migrations. The genuine engineering work collapses to: (1) the S3 storage-layer rewrite, and (2) getting the network/TLS/SSO wiring right.

PHASE 1

Network foundation & security groups

YouIT (VPC)
Goal: confirm the network layout and create the two security groups that enforce "the database is reachable only by the app." This is the backbone of the finance-data security story.
map the subnetsFrom Phase 0, write down: 2 public subnets (for the ALB + the EC2) and 2 private subnets (for RDS), each in a distinct AZ. RDS requires a DB subnet group spanning ≥2 AZs (Phase 3), so confirm two private subnets exist.
create App SGCreate nexa-app-sg. Inbound: 443/80 from the ALB SG (not the internet, since "no open ports" — the ALB is the only public ingress). No SSH rule — you'll use SSM Session Manager. Outbound: default allow (the app must reach RDS, S3, Azure, Graph).
create ALB SGCreate nexa-alb-sg. Inbound: 443/80 from the internet (0.0.0.0/0) — this is the public face. (If IT creates the ALB, they may create this; coordinate.)
create DB SGCreate nexa-db-sg. Inbound: 5432 only from nexa-app-sg — reference the SG, never an IP. This single rule is what makes the database private-by-trust as well as private-by-subnet.
tag everythingTag every resource from the first one: Project=NEXA, Environment=dev, Owner=Shreeraj Deshmukh, Vertical=AUXO. Tags drive cost reporting and let you find/tear-down the whole stack later.

▼

▼

Traffic is allowed strictly down the chain. Nothing on the internet can reach the EC2 directly (only via the ALB), and nothing can reach RDS except the app.

PHASE 2

Identity & secrets scaffolding

IT (role)You (SSM)
Goal: have the EC2 instance role ready (so the app reaches S3/secrets with no keys on disk) and load the application secrets into SSM Parameter Store, rotated.
instance role (IT)Confirm IT created the instance role with a least-privilege permission policy: s3:GetObject/PutObject on the NEXA bucket only, ssm:GetParameter* on /nexa/* only, and logs:* on the /nexa/* log group only. Its trust policy must allow the EC2 service to assume it.
rotateGenerate fresh values for every secret as part of the move (the ideal moment): NEXTAUTH_SECRET, EMAIL_ACTION_SECRET, the RDS password, etc. Treat the old Vercel/Supabase values as retired.
load SSMPut each secret into Parameter Store as a SecureString under /nexa/<ENV>/..., e.g. /nexa/dev/NEXTAUTH_SECRET. Use the full env-var inventory in §Env-var mapping.
the Azure dual-name trapStore the Azure values under both naming styles: the app uses AZURE_AD_TENANT_ID/CLIENT_ID/CLIENT_SECRET, but prisma/sync-azure-users.ts reads AZURE_TENANT_ID/CLIENT_ID/CLIENT_SECRET (no _AD_). Miss this and the user sync silently exits with an empty table. audit A10
# Example: load a secret (repeat per parameter)
aws ssm put-parameter --name /nexa/dev/NEXTAUTH_SECRET \
  --type SecureString --value "<newly-rotated-value>" --region eu-central-1
Why this order Identity before data and compute: the instance role must exist before you launch the EC2 (Phase 6) so you can attach it at launch, and the secrets must be in SSM before the app boots and tries to read them. Roles-not-keys is the single biggest security upgrade the move buys you (AWS course §12.2).

PHASE 12

Verification — end-to-end smoke tests

You
Goal: prove the whole system works before any cut-over. Test the seams the migration touched, not just "the page loads."
  • SSO: provisioned user logs in via Azure AD → dashboard; unprovisioned user rejected; session cookie is Secure.
  • RBAC: a SEL sees only their own opportunities; an ADMIN sees all (proves resolveOwnerFilter + the token role survived).
  • Database: create an opportunity, a pricing version, edit staffing — confirm writes hit RDS and the pool isn't serialising (Phase 5 #2).
  • S3 round-trip: upload a SOW doc (browser→S3 presigned PUT), then view it (presigned GET) — the core rewrite. Confirm the bucket is not publicly reachable.
  • Email: request an approval → the approver gets the Graph email with working Approve/Reject links → the one-click action updates state (validates MAIL_BASE_URL + the stable domain).
  • Resilience: restart the EC2 (stop/start) — Elastic IP holds, systemd brings the app back, SSO still works (proves the stable-URL design).
  • Logs/alarms: trigger a deliberate error and confirm it appears in CloudWatch.
The "did the migration break a seam?" tests The high-value checks are the ones that exercise what changed: S3 upload/view, SSO behind the ALB, and DB under light concurrency. If those three pass cleanly, the risky parts of the move are sound.

GSecurity release-blockers — fix before real financial data

These are application-level issues from the codebase audit, independent of the infrastructure move. The migration can proceed and be tested with them open, but they must be closed before NEXA holds real client financials in production. Treat them as gating items for the production cut-over.

#BlockerRiskFix direction
1Shared-password backup login (/login2) — one shared secret signs you in as any user, including ADMIN.CriticalRemove it (SSO-only), or per-user hashed passwords. At minimum IP-restrict + rotate.
2Emergency admin via URL token (/api/emergency) — static token in the query string mints an admin session.HighPOST + short-lived token + IP allow-list/WAF, or remove in prod.
3Write-side IDOR — most mutating routes authenticate but don't authorise; a user can act on another team's records by ID.HighOne shared ownership/role guard at the top of every mutating route (mirror the read-side filter).
4Audit logging not enforced — the ActivityLog table exists but isn't consistently written; finance data needs a who-did-what trail.MediumWrite ActivityLog on every create/update/approve/export.
Where these meet AWS AWS helps (WAF/rate-limiting in front of #1 and #2; CloudTrail for infra audit alongside #4), but it does not fix the app logic. The full findings, including the XSS/HTML-escaping and the chunked-cookie items, are in docs/codebase-audit.html.

JConcerns, risks & open decisions

The honest list of what could bite, what's unresolved, and what to decide with the SDE/IT. Ordered roughly by impact.

1 · Deployment shape is unresolved (microk8s vs simple) decide first The SDE's first note said microk8s; this runbook assumes EC2 + Nginx + systemd. microk8s on a single box adds real day-2 Kubernetes operations (a broken pod, ingress, upgrades) that a solo, app-focused owner without DevOps support will struggle to maintain — and it needs ECR (a permission gap). Action: confirm with the SDE. If he wants microk8s for firm consistency, ask explicitly who supports the k8s layer; if it's on you, the simpler path is the maintainable choice. This single decision changes Phases 6 and 8.
2 · Permission gaps will block specific phases act now Your IAM request omits load-balancer (ELB/ALB), iam:CreateRole (the instance role), Route 53/DNS, and ECR. "No open ports + ACM" requires an ALB, and the app needs the instance role. Action: ask IT to create the ALB + the instance role + the DNS record (or grant the matching permissions). Until then, Phases 6/7 are blocked. (See §Permissions.)
3 · The domain is the long pole start day one SSO and email can't be finalised until nexa.procdna.com exists, and DNS changes cross a team boundary with their own approval process. Action: open the domain + ACM-validation conversation immediately, in parallel with everything else.
4 · Region / data residency is the SDE's call confirm Default eu-central-1 matches today, but US+India users raise a residency/latency question. Region is sticky (resources can't move). Action: confirm before Phase 1 — it's cheap to choose now (no data), expensive to change later.
5 · "No open ports" + Elastic IP + ALB need reconciling clarify A public-subnet EC2 with an Elastic IP egresses fine (no NAT), but if the instance is genuinely private behind the ALB it needs a NAT gateway (or VPC endpoints) for outbound to Azure/Graph/S3. Action: confirm the intended placement (public-EIP vs private-behind-ALB) so the egress path is right — otherwise the app can't reach Azure AD and login dies.
6 · The S3 storage rewrite is the only real engineering risk Everything else is configuration; the Supabase→S3 layer (presigning, the four routes, cache eviction) needs its own testing. Action: isolate it on a branch, test upload/view thoroughly, and budget time for iteration. I can scaffold it against the current code when you reach Phase 5.
7 · The release-blockers are prod-gating, not migration-gating The four security items (login2, emergency, IDOR, audit logging) don't block testing on AWS, but must be fixed before real financial data. Action: schedule them into the production cut-over, not after.
8 · Solo ownership of new infrastructure support model You'll own EC2/RDS/S3/deploys with limited DevOps background, and IT is a separate vertical. Action: agree a support/escalation path with the SDE/IT for day-2 incidents (a broken deploy, a DB issue at 2am) before go-live — don't discover the gap during an outage.
9 · Smaller, known traps (already handled in the phases)
  • Prisma max=1 — raise it or the app crawls under concurrency (Phase 5 #2).
  • Chunked cookies — large Azure tokens cause a login loop without the .0 fix (Phase 5 #4).
  • Azure sync env names — dual-name or the user table silently stays empty (Phase 2/9).
  • next build OOM — build in CI, not on the box (Phase 6/8).
  • Demo seed is stale — don't run prisma/seed.ts (Phase 9).
  • xlsx@0.18.5 CVEs — patch as part of post-migration hardening (Phase 14).
  • Cost surprises — ALB + any data transfer bill hourly; the Budgets alarm is your safety net (Phase 0).
10 · Questions worth putting to the SDE/IT in one message
  1. microk8s or a plain container/Node deploy — and who operates the k8s layer if microk8s?
  2. Region (eu-central-1 vs ap-south-1/us-east-1) — any residency rule?
  3. Will IT create the ALB and the EC2 instance role, or grant me the permissions?
  4. Who owns the domain/DNS and the Azure App Registration (for the redirect URI)?
  5. Is the EC2 public-with-EIP or private-behind-ALB (decides NAT)?
  6. What's the day-2 support/escalation path once it's live?

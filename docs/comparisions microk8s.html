microk8s vs non-microk8s — the full comparison

The SDE floated microk8s (Kubernetes) for running NEXA on the EC2 box. This document lays out, in detail, what that choice actually costs versus the simpler EC2 + Nginx + systemd path the runbook assumes — every concrete change, the operational metrics, the complexity, and an honest recommendation for a single integrated app owned by an app-focused engineer.

Worked example: NEXA (one Next.js app, ~55 users) 3 options compared Rated scorecard + phase deltas

TL;DR

Recommendation for NEXA: non-microk8s (EC2 + Nginx + systemd) For a single, integrated, stateless app with ~55 internal users on one box, owned solo by an app-focused engineer, microk8s adds a full Kubernetes control plane and a steep day-2 operations burden while delivering almost none of its benefits — a one-node "cluster" has no real high-availability, and you have exactly one container to run. The simpler path gets you live faster, is dramatically easier to operate alone, and (because NEXA is stateless) loses nothing on the path to containers/ECS later if scale ever demands it.

microk8s is the right call only if the firm standardises on Kubernetes and there is a DevOps team that will actually operate the k8s layer for you. If that support doesn't exist, the simpler path is the maintainable one — and "who operates the k8s layer day-to-day" is the single question to settle with the SDE before committing.

1The three options on the table

"Non-microk8s" can mean two different things, so there are really three rungs on a ladder of how much machinery sits between your code and the OS. The headline decision is the first vs the third; the middle is an honest compromise worth knowing.

① EC2 + Nginx + systemd
"non-microk8s" · no containers · the runbook baseline
The app runs as a plain Node process managed by systemd, with Nginx reverse-proxying in front. No Docker, no orchestrator. The OS runs your app directly.

You manage: the OS, Node, Nginx, the systemd unit. New concepts: basically none beyond Linux service basics.
simplestfastest to livesolo-friendly
② Plain Docker
containerized · no orchestration · the middle ground
The app is packaged as a Docker image and run with docker run (kept alive by systemd or --restart). You get image portability and reproducibility, but no Kubernetes.

You manage: the OS, Docker, the image, one container. New concepts: Docker (a Dockerfile, image build) — but not orchestration.
a Dockerfileportableeasy path to ECS
③ microk8s
containerized + Kubernetes orchestration · the SDE's suggestion
The app is a Docker image run inside a single-node Kubernetes cluster (microk8s) via a Deployment + Service + Ingress, managed with kubectl.

You manage: the OS, microk8s itself, the image, k8s manifests, and the whole k8s control plane. New concepts: Docker and the entire Kubernetes model.
k8s control planesteep day-2heavy for 1 app
How to read the rest of this doc The big comparisons below put ① and ③ head-to-head (the real decision). ② appears wherever it changes the answer — it's the option to reach for if the firm wants containers but not Kubernetes.

2Architecture, side by side

The external architecture (browser → ALB → app → RDS/S3) is identical in all three — the ALB still terminates TLS with the ACM cert, RDS is still private, S3 is still presigned. What differs is only what's running on the EC2 box between Nginx/the ALB and your Node code.

① / ②   non-microk8s — what's on the box
ALB (TLS, ACM)unchanged
▼ HTTP
Nginx (reverse proxy)optional with an ALB
▼ localhost:3000
systemd → next start① the process directly, or ② a docker container

Two or three moving parts on the box. To debug: journalctl, systemctl status, (or docker logs). That's the whole surface.

③   microk8s — what's on the box
ALB (TLS, ACM)unchanged
▼ HTTP → NodePort
microk8s Ingressnginx-ingress addon
▼
Service → Deployment → Pod+ kubelet, API server, scheduler, controller-manager, etcd/dqlite
▼

A full Kubernetes control plane between the edge and your code. To debug: kubectl get/describe/logs, pod/ingress/service troubleshooting, plus the cluster's own health.

The core observation microk8s doesn't change what NEXA is or how users reach it — it inserts an entire orchestration layer to manage one container on one machine. On a single node, the orchestration has nothing to orchestrate. You pay the full complexity tax for a fraction of the benefit.

5Day-1 setup complexity, concretely

① non-microk8s — what you actually do

# on the EC2 box (via SSM)
sudo dnf install -y nodejs nginx
# drop the build in /opt/nexa, write 2 files:
#  /etc/nginx/conf.d/nexa.conf  (reverse proxy → :3000)
#  /etc/systemd/system/nexa.service  (Restart=always)
sudo systemctl enable --now nexa
sudo systemctl enable --now nginx

Two config files, two services. If it breaks, the failure is one of two places. A few hours, mostly familiar Linux.

③ microk8s — what you actually do

# install the cluster
sudo snap install microk8s --classic
sudo microk8s enable dns ingress registry storage
# build & push an image
docker build -t nexa . && push to registry…
# author & apply manifests:
#  deployment.yaml · service.yaml · ingress.yaml
#  configmap/secret for env
microk8s kubectl apply -f k8s/
# then wire the ALB → NodePort, debug ingress, probes…

A cluster to stand up, an image pipeline, 3–5 manifests to write and debug, ingress↔ALB wiring. Days, and most of it new.

The hidden day-1 cost of microk8s It's not the happy path that hurts — it's the first time a manifest is subtly wrong: a pod stuck CrashLoopBackOff, an ingress returning 503, a readiness probe failing, a service selector mismatch. Each is routine for a k8s practitioner and a multi-hour rabbit hole for a newcomer. With systemd, the equivalent failures are systemctl status away.

Setup happens once; operations happen forever. This is the most important section, because the gap between the two options is widest here — and it's the part the SDE's "just use microk8s" suggestion doesn't price in.

The honest framing for a solo, app-focused owner microk8s asks you to become a part-time Kubernetes operator on top of being the app developer. Initial setup is learnable (and I can scaffold it). But day-2 — a broken pod, an ingress misroute, a version upgrade, an expired cluster cert — is where zero DevOps background genuinely bites, and where "IT is a separate vertical that won't own the k8s layer" turns into you being stuck. That operational reality, more than any feature on the matrix, is why the simpler path wins here.

7TLS / ALB / ingress — the nuance that surprises people

A common assumption is that microk8s "handles TLS for you." For NEXA's setup it largely doesn't change the edge, and it adds a wrinkle:

  • ACM certificates cannot live inside microk8s. AWS public ACM certs only attach to an ALB/CloudFront — never to a microk8s ingress controller. So with microk8s you still run the ALB (with the ACM cert) in front, and it forwards to the cluster's NodePort. microk8s does not remove the ALB.
  • The alternative — terminating TLS in-cluster — means abandoning ACM for cert-manager + Let's Encrypt inside the cluster. That's more moving parts, not fewer, and diverges from the SDE's "ACM cert" instruction.
  • Net: on the edge, microk8s adds an ingress layer (ALB → NodePort → ingress → service → pod) where non-microk8s has none (ALB → app). Same cert, same ALB, one extra hop to configure and debug.
Implication The ALB + ACM work in your runbook (Phase 7) is required either way. microk8s doesn't simplify TLS; it adds an ingress to wire to the ALB. This quietly undercuts a common reason people reach for k8s ("it does TLS/routing") — here the ALB already does that.

At this scale it's a footnote, but a telling one Reserving ~1 GB and steady CPU to orchestrate one container is the overhead made concrete. It won't break a t3.large, but it's resources spent on machinery you don't need — and on a burstable instance, the constant control-plane CPU works against the very "quiet most of the day" profile that makes t3 cheap.

9Scaling & resilience — the usual case for k8s, examined honestly

Kubernetes earns its complexity at scale: many services, many nodes, autoscaling, self-healing across machines. Holding that up against NEXA:

k8s strengthDoes NEXA get it on one microk8s node?
Self-healing (restart crashed workloads)Yes — but systemd Restart=always already does this for a single process. parity
High availability (survive a machine failure)No — one node is a single point of failure regardless. Real HA needs multiple nodes = more EC2 = more cost/ops. not delivered
Horizontal autoscalingOnly up to the one node's capacity; true scale-out needs more nodes. At 55 users you won't approach the limit. low value now
Rolling / blue-green deploysYes — a genuine win, but a brief restart blip on an internal tool is rarely a real problem, and you can get zero-downtime non-k8s by running 2 instances behind the ALB if it ever matters. nice-to-have
Bin-packing many services onto nodesNEXA is one service. Nothing to bin-pack. N/A
The crux Kubernetes' marquee benefits are about many services across many machines. NEXA is one service on one machine. The shape of the problem doesn't match the shape of the tool. If NEXA ever became many services or needed true multi-node HA, the right move wouldn't be self-managed microk8s anyway — it'd be managed ECS Fargate or EKS (no control plane for you to babysit). Which is the future-proofing point next.

The microk8s pipeline adds an image build + registry push and needs cluster access from CI (a kubeconfig/credential to reach the microk8s API) — another secret/access path to set up and secure. The non-microk8s pipeline reuses the SSM access you already have.

11Future-proofing — does microk8s actually help later?

The strongest honest argument for microk8s is "we'll want orchestration eventually, so start now." Examined:

  • NEXA is stateless (all state in RDS + S3). Containerizing later is low-friction whenever you want it — you're not painting yourself into a corner by starting on systemd.
  • If you ever need orchestration, the destination is managed — ECS Fargate (simplest) or EKS — where AWS runs the control plane. Self-managed microk8s on EC2 is not that destination; it's a different thing you'd still have to migrate off.
  • The genuinely useful "phase-2" step is to containerize (option ②, plain Docker) — that gets you the portable image and the on-ramp to ECS/EKS without running a Kubernetes control plane yourself.
The future-proof move isn't microk8s — it's "stateless now, container when needed" Start simple (systemd), keep the app stateless (it already is), and if scale ever demands orchestration, jump straight to managed ECS/EKS by adding a Dockerfile. microk8s is a self-managed-Kubernetes detour that costs you operational pain now and isn't even the place you'd end up.

12If you drop microk8s — exactly what changes

The direct answer to your question. Choosing non-microk8s means you remove a whole column of work and add almost nothing. Here's the delta.

What you NO LONGER have to do
  • Write a Dockerfile and maintain an image build
  • Stand up and patch microk8s (the cluster)
  • Author/debug Deployment, Service, Ingress manifests
  • Learn kubectl and the Kubernetes model
  • Set up ECR (and the IAM permission for it)
  • Wire the ALB → NodePort → ingress hop
  • Sync secrets into k8s Secrets / manage ConfigMaps
  • Ship logs out of pods via a log collector
  • Give CI cluster credentials for kubectl deploys
  • Carry ~0.5–1 GB control-plane overhead on the box
  • Own day-2 Kubernetes operations solo
What you do INSTEAD (much smaller)
  • Install Node + Nginx on the box
  • Write one Nginx reverse-proxy config
  • Write one ~10-line systemd unit (Restart=always)
  • Pull secrets SSM → env at boot (a small script)
  • Deploy = ship build + systemctl restart

Everything else in the migration — VPC, security groups, RDS, S3, the instance role, ACM, the ALB, Route 53, SSM secrets, CloudWatch, the S3 storage-layer rewrite, the schema/seed, the Azure redirect URI — is identical in both worlds. The deployment choice touches only what runs on the box.

The headline Dropping microk8s removes the single biggest source of new complexity and ongoing operational risk in the whole migration, and replaces it with two config files. The app, the data, the network, the security, and the edge are unchanged. You lose declarative rollouts and a k8s-standard footprint; you gain a system you can actually build and operate alone.

If the firm wants containers (for image portability, reproducibility, or a future ECS path) but not Kubernetes, plain Docker is the sensible compromise — most of the container upside, almost none of the orchestration cost.

It's the strongest answer if the SDE's real concern is "we want containers," not "we want Kubernetes." If even that feels like extra, option ① is perfectly sufficient for NEXA today.

15When each genuinely wins

Pick non-microk8s (① or ②) when…
  • It's one app (or a few) — NEXA is one.
  • It runs on one box for now.
  • The owner is app-focused, operating solo, without a platform/DevOps team behind them.
  • You want to be live quickly and operate it sustainably.
  • The firm has no hard Kubernetes standard, or will accept a documented exception for a small internal tool.
→ This is NEXA.
Pick microk8s when…
  • The firm standardises on Kubernetes and there's a platform/DevOps team that operates the k8s layer for you.
  • You're running many services that benefit from shared orchestration.
  • You need declarative, GitOps-style deploys across environments as a hard requirement.
  • There's a genuine multi-node story (real HA, autoscaling) on the roadmap.
  • The team's existing tooling/runbooks all assume kubectl.
→ Only if the firm's reality matches this.

16What to settle with the SDE

The decision turns on a few facts only the SDE/firm can confirm. Put these to him:

  1. Is microk8s a hard firm standard, or a default suggestion? If it's "how we run things," that weighs heavily. If it's "seemed reasonable," the simpler path is on the table.
  2. Who operates the Kubernetes layer day-to-day? If there's a platform team, microk8s is viable. If it's you alone (app-focused, IT is a separate vertical), that's the strongest argument for non-microk8s.
  3. Would the firm accept containers-without-Kubernetes (plain Docker)? If the real goal is portability/standardization rather than orchestration, option ② satisfies it with far less burden.
  4. Is there a real multi-service / multi-node future for NEXA, or is it a stable single internal app? The former leans (managed) orchestration; the latter doesn't.
A fair, confident way to put it "For a single integrated app on one box that I'll own solo, microk8s adds a Kubernetes control plane and day-2 operations without the HA/scale benefits that justify it — a one-node cluster has nothing to orchestrate. I'd propose EC2 + Nginx + systemd (or plain Docker if you want a container), keeping the app stateless so we can move to managed ECS/EKS later if it ever needs real orchestration. Happy to go microk8s if it's a firm standard and there's a platform team to operate it — just want whatever stays maintainable."
Bottom line For NEXA as it is today, non-microk8s wins clearly: same external architecture, dramatically less to build and operate, no loss of the future path (the app is stateless), and no real sacrifice of capability at this scale. microk8s is the answer to a problem NEXA doesn't have yet — and if it ever does, the answer would be managed ECS/EKS, not a self-run cluster on one EC2. Settle "is k8s a firm standard with platform support?" with the SDE; absent a clear yes, ship the simple path.

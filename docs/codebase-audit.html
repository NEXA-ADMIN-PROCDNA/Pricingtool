Personal working notes. Covers (1) security holes, (2) places the current Supabase + Vercel build can break under future conditions, (3) new risks the proposed EC2 / Nginx / RDS / S3 stack introduces, and (4) redundant/dead files. Nothing here has been changed in code — every item lists a suggested fix only.

The one-line theme: every API route authenticates (getAuthToken) but most don't authorise — they act on any record by ID with no owner/role check. That's the largest single class of bug here (IDOR). The second theme is "works on Vercel serverless, will bite on a long-lived EC2 process" — connection pool size, in-memory caches, CPU-bound parsing, and TLS-termination assumptions all change.

1 · Security vulnerabilities // current build, also carry to AWS

These are exploitable today on Supabase+Vercel. They get worse on raw EC2 (no platform DDoS/rate-limit shield) — see §3.

S1 Shared-password login = sign in as anyone Critical

lib/auth.js:31–42   app/login2/page.tsx   /login2 (excluded from proxy)

The credentials provider compares the submitted password against a single shared CREDENTIALS_SECRET and then logs you in as whatever email you typed, as long as that user exists and is active. There is no per-user password.

Why it matters: one leaked env value → full account takeover of any user, including an ADMIN. /login2 is public (proxy matcher excludes it). This is the single highest-impact issue in the repo.
Current
// lib/auth.js — authorize()
const secret = process.env.CREDENTIALS_SECRET
if (!secret || !credentials?.password || credentials.password !== secret) return null
const user = await prisma.user.findUnique({ where: { email: credentials.email }, ... })
// → anyone who knows the one secret can become any email
Suggested fix
1. Add a passwordHash column; store a bcrypt/argon2 hash per user.
2. Compare with await bcrypt.compare(credentials.password, user.passwordHash).
3. If you only want SSO: delete the CredentialsProvider + /login2 entirely
   and keep Azure AD as the sole path (you already gate on provisioned users).
4. Interim: scope /login2 to a tiny allow-list of emails AND rotate CREDENTIALS_SECRET,
   put it behind IP allow-list / WAF (see A8).

S2 Emergency endpoint: admin session from a URL token High

app/api/emergency/route.ts:5–49

A GET with ?token=EMERGENCY_TOKEN mints an 8-hour ADMIN session cookie for the first admin user. No expiry on the token, no rate limit, no IP gate, and it's deliberately excluded from proxy auth.

Why it matters: the secret travels in the URL → it lands in server/proxy access logs, browser history, the Referer header, and (given your mail flows) potentially Safe-Links scanners. A static, long-lived admin bypass. On Vercel there's some shielding; on EC2 (§A8) it's directly brute-forceable.
Suggested fix
• Make it POST only, body (not query) for the token, so it can't be logged via URL.
• Add a short TTL: sign a time-limited token (reuse lib/approval-tokens style) instead of a static env string.
• Restrict by source IP (office/VPN) at Nginx/WAF.
• Rate-limit (e.g. 5/min) and alert on use.
• Best: remove it once SSO is stable in prod, or guard behind NODE_ENV !== 'production'.

S3 Stored XSS in the email-action pages High

app/api/approvals/email-action/route.ts:35–113, 220–269

The branded confirm/approved/rejected HTML pages interpolate opportunityName and approverName straight into markup with no escaping. opportunityName is free user input set at opportunity-creation time.

Why it matters: name an opportunity <img src=x onerror=...> and the script runs in the approver's browser when they open the email link — the one page in the app served to a privileged user, with no auth wall in front. Classic stored XSS.
Current
// values dropped into HTML unescaped
`...you are approving the request for <strong>${opportunityName}</strong>.`
resultPage('approved', 'Approved.', oppId, `...${opportunityName}...`)
Suggested fix
// escape every interpolated value before it touches HTML
const esc = (s) => String(s).replace(/[&<>"']/g,
  c => ({'&':'&amp;','<':'&lt;','>':'&gt;','"':'&quot;',"'":'&#39;'}[c]))
approveConfirmPage(token, esc(opportunityName), esc(approverName))
// apply to opportunityName, approverName, requesterName, and the reject `reason`

S4 Broken access control: approval creation trusts the client High

app/api/opportunities/[id]/approvals/route.ts:61–116

The route reads requestedById and approverId from the request body, not from the session. It never checks that the caller owns the opportunity, nor that they are the requester they claim to be.

Why it matters: any signed-in user can open an approval "as" someone else, route it to an arbitrary approver, and advance any opportunity's stage by guessing its OPP-YY-NNNN id. The trusted token.id is available but unused.
Suggested fix
const { approverId, requestedById, ... } = await req.json()
const requestedById = token.id as string            // never from the body
// load opp, then enforce ownership/role before creating:
if (opp.ownerId !== token.id && token.role !== 'ADMIN') return apiError('OPP_EDIT_FORBIDDEN')
// validate approverId is a real, eligible approver (role gate)

S5 IDOR everywhere: authenticated ≠ authorised High

sow/route.ts · po/route.ts · sow/confirm · po/confirm · other-costs · comments · pricing-versions/* · staffing/*

These routes check that you're logged in, then operate on whatever opportunityId / pvId / docId you pass. No resolveOwnerFilter equivalent is applied. IDs are sequential and enumerable.

Why it matters: a SEL can read/upload/delete SOW & PO documents, edit pricing, and add costs on another team's deal just by changing the id in the URL. The RBAC you built in lib/db/opportunities.ts only guards the read/list path, not the mutations.
Suggested fix — one shared guard, call it at the top of every mutating route
// lib/authz.ts (new)
export async function assertCanEditOpp(token, opportunityId) {
  const opp = await prisma.opportunity.findUnique({ where:{ opportunityId }, select:{ id:true, ownerId:true }})
  if (!opp) throw apiError('OPP_NOT_FOUND')
  const role = token.role as string
  if (role === 'ADMIN' || role === 'PARTNER') return opp
  if (opp.ownerId === token.id) return opp
  // optional: DIRECTOR/ED team checks here
  throw apiError('OPP_EDIT_FORBIDDEN')
}

S6 PATCH opportunity: status / projectCode / preContract not owner-gated High

app/api/opportunities/[id]/route.ts:34–70

Only the "details" fields (BU, Star Connect, dates) are behind the owner/admin gate. status (OPEN/WON/LOST/ABANDONED), preContractAgreed, and the permanent projectCodeProceed commit are reachable by any authenticated user on any opportunity.

Why it matters: someone can mark a rival team's deal WON, push it to SOW, or fire the irreversible project-code commit. projectCodeProceed is described as permanent in AGENTS.md — no undo.
Suggested fix
// move the gate ABOVE all branches, not just the details branch
const opp = await prisma.opportunity.findUnique({ where:{opportunityId} })
if (!opp) return apiError('OPP_NOT_FOUND')
const isOwner = opp.ownerId === token.id, isAdmin = token.role === 'ADMIN'
if (!isOwner && !isAdmin) return apiError('OPP_EDIT_FORBIDDEN')
// then handle status / preContractAgreed / projectCodeProceed

S7 Pipeline revenue leaks company-wide totals to every role Medium

app/api/pipeline-revenue/route.ts:10–28

Authenticated, but it sums estimatedRevenue / final billings across all active opps with no owner filter, while getDashboardStats (the same numbers) does apply resolveOwnerFilter.

Why it matters: a SEL hitting this endpoint sees total company pipeline value they're not entitled to. Also note: it appears to have no client caller (see R12) — may be deletable rather than fixable.
Suggested fix
Either delete the route (if unused) or thread the same auth + resolveOwnerFilter(auth)
into its where clause exactly like getDashboardStats does.

S8 Directors/EDs see every team's deals (no team scoping) Medium

lib/db/opportunities.ts:30–53

resolveOwnerFilter gives DIRECTOR "own + any SEL-owned" and ED "own + any DIRECTOR/SEL-owned" — there is no managerId subtree, so it's company-wide, not team-wide. AGENTS.md admits this ("No managerId traversal").

Why it matters: horizontal data exposure between teams. Fine if that's the business intent; a real hole if directors are meant to see only their own org.
Suggested fix
Walk the manager tree: collect report ids via managerId and filter
{ ownerId: { in: [self, ...descendantIds] } } instead of { owner: { role: 'SEL' } }.

S9 Stale role / disabled account survives up to 8h Medium

lib/auth.js:75–104 (jwt re-fetch only on signIn) · session maxAge 8h

Role and location are baked into the JWT at sign-in and only refreshed on the next sign-in. Deactivating a user (isActive=false) or downgrading a role doesn't take effect until the token expires (8h) — there's no server-side session check on each request.

Why it matters: off-boarding / role changes aren't enforceable in real time. A fired user keeps access for hours.
Suggested fix
In the API guard (or proxy), periodically re-check the DB:
const u = await prisma.user.findUnique({ where:{id: token.id}, select:{isActive:true, role:true}})
if (!u?.isActive) return apiError('UNAUTHORIZED')
// or shorten maxAge, or set jwt callback to re-fetch when token age > N min

S10 xlsx 0.18.5 has known high-severity CVEs Medium

package.json:37 · used in rate-cards/upload & export/opportunities

SheetJS xlsx@0.18.5 (npm) is affected by prototype-pollution (CVE-2023-30533) and a ReDoS (CVE-2024-22363). The fix ships only via SheetJS's own CDN build, not the npm registry version you have pinned.

Why it matters: you parse attacker-influenced spreadsheets (admin uploads a rate card). Even admin-only, a malicious file can pollute prototypes or hang the parse.
Suggested fix
• Move to the SheetJS CDN tarball (their advisory): npm i https://cdn.sheetjs.com/xlsx-0.20.x/xlsx-0.20.x.tgz
• Or swap to a maintained lib (exceljs) for both read and write.
• Validate parsed cells defensively regardless.

S11 Upload type/size trusts client-claimed values Medium

app/api/opportunities/[id]/sow/route.ts:59–66 · sow/confirm:22–41 (same for PO)

The MIME allow-list and size check run against JSON fields (mimeType, fileSize) the browser sends. The bytes go browser→Supabase directly via the signed upload URL, so the server never sees the real file. confirm then stores whatever storagePath/mimeType the client passes.

Why it matters: a client can claim image/png and upload HTML/SVG; if ever served inline it's XSS. Limit is also inconsistent: code says 49 MB, AGENTS.md/UI says 20 MB.
Suggested fix
• Constrain the signed upload URL to an exact content-type and max size at issue time.
• On confirm, re-derive mimeType from the stored object (HEAD it) rather than trusting the body;
  validate that storagePath starts with this opp's prefix.
• Pick one size limit and align code + UI + error copy (DOC_TOO_LARGE says 49 MB).

S12 HTML injection into outbound emails Medium

lib/mail.ts — businessJustification, reason, names, opportunityName, clientName all unescaped

Same root cause as S3 but in the email bodies. Free-text fields (businessJustification, rejection reason) are interpolated into HTML sent to approvers/partners.

Why it matters: a requester can inject markup/phishing links into a mail that lands in a partner's inbox looking like it came from NEXA. Lower severity than S3 (mail clients sandbox script) but real for link-injection.
Suggested fix
Reuse the same esc() helper from S3 on every interpolated value inside wrap()/metaRow().

S13 Email-action token: empty-secret fallback + non-constant-time compare Low

lib/approval-tokens.ts:5–7, 28–31

secret() returns '' if NEXTAUTH_SECRET is unset — tokens then sign/verify under an empty key (forgeable). The HMAC check uses sig !== expected (string compare), not a timing-safe compare.

Suggested fix
if (!process.env.NEXTAUTH_SECRET) throw new Error('NEXTAUTH_SECRET required')
// constant-time compare
import { timingSafeEqual } from 'crypto'
const a = Buffer.from(sig), b = Buffer.from(expected)
if (a.length !== b.length || !timingSafeEqual(a, b)) return null

S14 Error detail / debug rows leak in responses Low

rate-cards/upload/route.ts:64–78 (debug preview) · errors.ts apiError(detail) · export passes raw Graph errors

Parse failures return the first 30 spreadsheet rows; several routes return raw exception messages / raw Graph error bodies as detail.

Suggested fix
Gate detail/debug behind NODE_ENV !== 'production'; log full detail server-side, return a generic message to the client.

2 · Crash & failure points // works now, will break later

Things that are fine today but fail on a future input, scale point, or env condition — including the cookie / token issues you flagged.

C1 getAuthToken cookie detection breaks on chunked cookies → login loop High

lib/getAuthToken.ts:19–27

Cookie-name detection keys off the literal base name:

const useSecureCookies = req.cookies.has('__Secure-next-auth.session-token')

When the session JWT exceeds ~4 KB, NextAuth splits it into chunks — __Secure-next-auth.session-token.0, .1, … — and does not set the un-suffixed base cookie. So .has(base) returns false, the helper falls through to the non-secure name, getToken looks for a cookie that isn't there, returns null, and the proxy redirects to /login — on every request. An infinite login loop, only for users with big tokens.

Why it matters: Azure AD tokens grow with group/role claims. The day a user is in enough AD groups to cross 4 KB, they can't log in — and it'll look random. This is the "fixed string can't find the token" class you suspected.
Suggested fix
const useSecureCookies =
  req.cookies.has('__Secure-next-auth.session-token') ||
  req.cookies.has('__Secure-next-auth.session-token.0')
// getToken() reassembles the .0/.1 chunks itself once given the right base name + secureCookie

C2 Two Prisma clients — wrong one silently targets the default schema High

lib/prisma.ts (pg adapter, schema: 'procdna_database') vs lib/prisma.js (bare new PrismaClient())

Both files export prisma and both answer the import @/lib/prisma. Today TS/the bundler resolves .ts first, so it works. lib/prisma.js has no pg adapter and no schema: 'procdna_database' — if extension-resolution order ever flips (tooling change, Turbopack quirk, a stray .js import), every query hits the public schema and blows up with "table does not exist".

Why it matters: a latent, environment-dependent footgun that would present as a total DB outage with a confusing error. Also two different global singleton keys (_prisma vs prisma).
Suggested fix
Delete lib/prisma.js — nothing imports it explicitly; lib/prisma.ts is the real one (confirmed: all 30+ importers use the bare specifier).

C3 Multi-step mutations aren't transactional Medium

approve/reject/withdraw routes · opportunities/[id]/approvals · email-action POST

Each does several sequential prisma.*.update calls (approval status → opportunity stage → sometimes cascade withdraw) outside a transaction. If the process dies or a query fails between them, you get a half-applied state: approval marked APPROVED but stage unchanged, or vice-versa.

Why it matters: rare, but produces "stuck" opportunities that need manual DB surgery. More likely on EC2 if the box restarts mid-request.
Suggested fix
Wrap the related writes in await prisma.$transaction(async (tx) => { ... })
(the date-reset branch in opportunities/[id]/route.ts already does this — mirror that pattern).

C4 Opportunity-ID generation can collide & mis-sorts past 9999 Medium

app/api/opportunities/route.ts:13–27

"Find the max OPP-YY-NNNN, add 1" runs as a plain read followed by an unguarded insert. Two concurrent creates compute the same number. Also orderBy: opportunityId desc is lexicographic — OPP-26-10000 sorts before OPP-26-9999, so numbering breaks at 10k.

Suggested fix
• Use a DB sequence / counter row in a transaction with row lock, or a unique constraint + retry loop.
• Or store the numeric part in its own int column and orderBy that, not the string.

C5 Missing env vars fail loudly (500) or silently (login loop) Medium

lib/supabase.ts:10–12 · lib/mail.ts:12 · getAuthToken (NEXTAUTH_SECRET) · approval-tokens (S13)

Supabase and mail throw on construction if their env vars are missing; the SOW/PO/export routes 500. getToken with a missing/wrong NEXTAUTH_SECRET just returns null → everyone silently bounced to /login.

Why it matters: the #1 way the AWS cut-over will "mysteriously" break — one env var typo and you get either 500s or an un-debuggable login loop. Make startup fail fast with a clear message instead.
Suggested fix
Add a startup env check (a small lib/env.ts that asserts all required vars on boot and logs exactly which is missing). Run it from instrumentation / server entry.

C6 serialize() JSON round-trip: slow on big detail, throws on BigInt, fragile Decimal check Medium

lib/db/opportunities.ts:16–22

JSON.parse(JSON.stringify(data, replacer)) runs on the full opportunity-detail graph (pricing versions × staffing × up to 520 weekly-hour rows). It detects Decimals by value.constructor.name === 'Decimal' (breaks under some minification) and will throw if any field is ever a BigInt.

Suggested fix
Convert Decimals explicitly at the query boundary (you already do Number(x) elsewhere),
or use Prisma's driver-adapter number coercion, instead of a whole-tree stringify/parse.

C7 POST opportunity: no date-validity check; revenue parseInt corrupts Low

app/api/opportunities/route.ts:67–86

new Date(startDate) on garbage → Invalid Date → Prisma throws → generic 500. parseInt('1.5M',10) → 1, silently storing wrong revenue. (The PATCH route validates dates; POST doesn't.)

Suggested fix
Validate isNaN(d.getTime()) and reject; use Number() + finite check for money, return OPP_DATE_INVALID / 400 instead of 500.

C8 Graph/mail fetch has no timeout; mail errors are swallowed Low

lib/mail.ts:42–64 · export/opportunities Graph calls

fetch to Microsoft Graph has no AbortController. On Vercel a function timeout eventually kills it; on a long-lived EC2 process a hung Graph call just ties up the request. Mail failures are logged and swallowed (intentional — don't block approvals — but silent).

Suggested fix
Add AbortSignal.timeout(10_000) to Graph fetches; surface a non-fatal MAIL_FAILED hint so the UI can say "saved, email pending".

3 · New risks on the AWS stack // EC2 + Nginx + RDS + S3

These are not bugs today — they're created by moving off Vercel serverless onto a single long-lived Node process behind a TLS-terminating proxy, with RDS and S3 instead of Supabase. Read this section next to the migration runbook.

A1 Prisma pool max=1 in prod throttles the whole EC2 box High

lib/prisma.ts:9–11

The pool is hard-capped at one connection in production — correct for Vercel (one short-lived connection per lambda, pgBouncer handles fan-out). On a single always-on EC2 Node process, max=1 means every request in the whole app serialises through one DB connection.

Why it matters: the app will feel fine in testing and fall over the moment two people use it at once — every query queues behind the others. This is the classic "it was instant on Vercel, why is it crawling on EC2" trap.
Suggested fix
const max = process.env.NODE_ENV === 'production' ? 1 : 3
const max = Number(process.env.DB_POOL_MAX ?? 10)   // size to RDS max_connections / #instances
// On RDS you connect directly (no pgBouncer), so a real pool is both safe and necessary.
// DATABASE_URL/DIRECT_URL can converge on the same RDS endpoint; add sslmode=require (A6).

A2 TLS terminates at Nginx/ALB → app sees http:, cookie logic misfires High

app/api/emergency/route.ts:37–47 · proxy / NextAuth secure-cookie logic generally

Behind Nginx (or an ALB) the Node process receives plain HTTP internally even though the browser used HTTPS. Code that reads req.nextUrl.protocol === 'https:' (emergency route) will see http: and set the non-secure cookie name with secure:false. NextAuth's own secure-cookie behaviour depends on NEXTAUTH_URL being https://….

Why it matters: get this wrong and SSO sets a cookie the browser won't return over HTTPS, or sets the wrong name → login loop. This is the same family as C1 and is the most likely day-one breakage on EC2.
Suggested fix
• Nginx: proxy_set_header X-Forwarded-Proto $scheme; (ALB sets it automatically).
• Set NEXTAUTH_URL=https://nexa.yourdomain (https, the public URL).
• In code that needs the scheme, trust the header:
  const proto = req.headers.get('x-forwarded-proto') ?? req.nextUrl.protocol.replace(':','')
  const isSecure = proto === 'https'
• Keep the stable domain (Elastic IP / ALB) so the redirect URI & Safe-Links never change.

A3 S3 rewrite: keep buckets private, use an instance role, match URL expiry High

replaces lib/supabase.ts · sow/route.ts · po/route.ts · *_/confirm · export

Moving Supabase Storage → S3 is the one real code change. Risks introduced: a public bucket exposes every SOW/PO doc; long-lived AWS keys in .env can leak; presigned-URL expiry must match the current 1-hour signed-URL assumption (and the 55-min cache in getSignedUrl).

Why it matters: a single "Block Public Access = off" misclick on the bucket and confidential pricing documents are world-readable. This is the highest-impact new security surface.
Suggested fix
• Bucket: Block Public Access ON; access only via presigned GET/PUT.
• Auth: attach an IAM instance role to EC2 (s3:GetObject/PutObject on the one bucket) —
  the AWS SDK auto-picks rotating creds. No AWS keys in .env.
• Presign with @aws-sdk/s3-request-presigner, expiresIn: 3600; keep the 55-min cache.
• Constrain PUT presign to content-type + content-length (fixes S11 at the same time).
• Enable default SSE encryption + bucket versioning (A11).

A4 No rate limiting / WAF in front of the secret-bearing endpoints Medium

/api/emergency · /login2 · /api/approvals/email-action

Vercel gave you some platform-level abuse protection for free. On bare EC2/Nginx there's none by default, and you have three endpoints whose security rests entirely on a guessable/brute-forceable secret (S1, S2) or token.

Why it matters: combined with S1/S2 this turns "needs the secret" into "brute-force the secret at 10k req/s". Rate limiting is what keeps those mediums from becoming criticals on AWS.
Suggested fix
• Nginx limit_req_zone on /api/* and especially the three routes above.
• Front with AWS WAF (rate-based rule + optional geo/IP allow-list for /emergency).
• IP-allow-list /emergency and /login2 to office/VPN ranges.

A5 In-memory caches & singletons leak on a process that never recycles Medium

lib/supabase.ts:29 (urlCache Map) · any module-level Map/global

urlCache adds an entry per bucket::path and never evicts expired entries — it only checks freshness on read. On Vercel the instance recycles constantly so it's self-cleaning; a long-running EC2 process grows it without bound.

Why it matters: slow memory creep → OOM after weeks of uptime, the kind of thing that pages you at 3am. Applies to anything you cache in a module-level structure once the process is long-lived.
Suggested fix
• Evict on read when expired, and cap size (simple LRU), or use a TTL map.
• Or drop the cache entirely once on S3 — presigning is cheap and local.

A6 RDS requires SSL; pool/driver config must say so Medium

lib/prisma.ts (pg Pool) · DATABASE_URL / DIRECT_URL · prisma.config.ts

RDS Postgres expects TLS. The raw pg.Pool won't negotiate it unless the connection string / pool says so, and you may need the RDS CA bundle for full verification. Supabase handled this for you.

Suggested fix
• Append ?sslmode=require to DATABASE_URL/DIRECT_URL, or pass
  new Pool({ connectionString, ssl: { rejectUnauthorized: true, ca: rdsCaPem } }).
• Both URLs can point at the same RDS endpoint (no pooler split needed); see A1 for pool size.
• Put RDS in a private subnet; SG allows 5432 only from the EC2 SG.

A7 CPU-bound xlsx parsing blocks the single Node event loop Medium

rate-cards/upload/route.ts:37 (XLSX.read) · export/opportunities buildBuffer (≤5000 rows)

SheetJS parse/write is synchronous and CPU-heavy. On Vercel each invocation is its own isolate; on a single EC2 process, a big upload or a 5000-row export freezes every other request for the duration.

Suggested fix
• Cap upload size hard before parse; consider a worker thread / child process for parse+build.
• Stream the export or paginate; don't hold 5000 rows × all relations in memory on the request path.

A8 Secrets land on disk & in AMIs/snapshots Medium

.env on the EC2 box (Azure secret, NEXTAUTH_SECRET, EMERGENCY_TOKEN, CREDENTIALS_SECRET, DB creds)

A plaintext .env on the instance gets baked into any AMI you image and any EBS snapshot you take, and is readable by anyone who can SSH in.

Suggested fix
• Store secrets in SSM Parameter Store (SecureString) or Secrets Manager; load at boot.
• If you must keep a .env: chmod 600, owned by the app user, never in an AMI.
• Rotate every secret during the move (they've been in Vercel + git history-adjacent files).

A9 No process manager / backups story (Vercel did both) Medium

deployment · RDS · S3

On Vercel, crashes self-heal and the DB was backed up for you. On EC2 you own: keeping next start alive across crashes/reboots, RDS backup retention, and S3 durability/versioning.

Suggested fix
• Run under systemd (Restart=always, WantedBy=multi-user.target) or PM2; verify it survives reboot.
• RDS: automated backups + a known retention (e.g. 7d); snapshot before the cut-over.
• S3: enable versioning (recovers overwritten/deleted docs) + lifecycle as needed.

A10 Azure sync script reads different env-var names than the app Medium

prisma/sync-azure-users.ts:102 vs lib/mail.ts/auth.js

You said the existing scripts must keep working post-migration — this one has a trap. The app uses AZURE_AD_TENANT_ID / AZURE_AD_CLIENT_ID / AZURE_AD_CLIENT_SECRET; the sync script reads AZURE_TENANT_ID / AZURE_CLIENT_ID / AZURE_CLIENT_SECRET (no _AD). If you only set the app's names on EC2, the script prints "Missing…" and process.exit(1) — silently does nothing.

Why it matters: directly answers "will my scripts still run?" — only if you provide both naming conventions in the EC2 environment, or unify them.
Suggested fix
Either set both sets of names in the EC2 env, or change the script to read the
AZURE_AD_* names the rest of the app uses. Also note the script uses DATABASE_URL —
point it at the RDS endpoint and it's fine (direct connection, no pooler quirks).

Bonus bug spotted in the same file (role mapping): TITLE_TO_ROLE matches a lower-cased job title against entries like 'Co Founder', 'Senior Engagement Lead', 'D' — all of which contain uppercase letters and therefore never match. Several rules are dead, and 'director' is listed twice. Worth fixing before you rely on it to seed RDS.

A11 Hardcoded EC2 IP in next.config & data-residency Low

next.config.ts:9 (allowedDevOrigins ['13.48.196.130'])

The current EC2 public IP is committed. It's dev-only (no effect on next start), but it's exactly the value your "IP must not change" concern is about, and it leaks infra into git. Pair with: put RDS+S3+EC2 in eu-central-1 to match today's fra1 residency.

Suggested fix
Drive it from an env var, or remove it for prod; rely on the stable domain (Elastic IP/ALB) so it never needs editing.

A12 Logging/observability disappears with Vercel Low

console.* throughout

All your diagnostics are console.error/warn. Vercel captured those; on EC2 they go to stdout and vanish unless you capture them.

Suggested fix

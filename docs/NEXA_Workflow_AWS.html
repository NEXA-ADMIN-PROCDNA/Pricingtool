Everything runs inside ProcDNA's own AWS account (Auxo). NEXA shares the existing dev cluster and VPC but keeps its own service, own load balancer target group, own DB schema, and own S3 bucket — isolated from neighbouring projects.

02 Stack Components what each piece is & owns
ECS
Fargate ComputeLive
Serverless container — no VM to patch. AWS restarts it on failure. Runs the Next.js 16 / React 19 / Prisma 7 app. Rolling deploys keep the old task serving until the new one passes health checks. cluster: auxo-frontend-dev
ALB
Load BalancerLive
The public front door. Terminates HTTPS with an ACM certificate, health-checks /api/health, and forwards to the container on port 3000. Its subnets were trimmed to the 2 AZs the target group lives in.
RDS
PostgreSQLnexa_devMigrating
Managed database on the shared auxo-development PG13 instance. NEXA owns the nexa_dev schema with dedicated nexaDeveloper credentials — other projects can't read its tables. Structure applied; data cutover pending.
S3
Object Storagenexa-dev-bucket
Holds uploaded SOW & PO documents. Access is via short-lived signed URLs that expire after 1 hour, so links can't be forwarded and reused. Bucket created; app still reads/writes Supabase Storage until cutover.
SSM
Parameter StoreLive
Encrypted vault for every secret — DB URL, NextAuth signing key, Azure client secret, Graph credentials, HMAC keys. Injected into the task at boot; only the ECS task role can read them. No secret ever ships inside the code or image.
CI
GitHub Actions + ECRLive
Push to main → build image → push to the ECR registry → force new ECS deployment. Fully automated; a failed health check leaves the previous version running. No manual server access needed to ship.

05 Approval Logic two gated sign-offs

Pricing Approval

stage S3 → S4
Approved→ S5
Advances to SOW Pending. The final pricing version locks — no silent edits.
Rejected→ S3
Back to Price Linked with the reason attached; owner revises and re-requests.
Withdrawn→ S3
Requester pulls a pending request back (reason logged) if something changed.
Dual-approver mode: nominate Finance (A1) + Business (A2). A1 must decide first — A2 stays in a "waiting" state until then. The stage only advances when both approve; either rejection stops the deal.

SOW Verification

stage S6 → S7
Verified→ S8
Paperwork confirmed. Owner commits the project code through a deliberate "are you sure?" step.
Rejected→ S6
Back to SOW Submitted so documents can be corrected and re-submitted.
Email actions: approvers act straight from their inbox. The email carries deal context + metrics; Approve/Reject opens a signed confirmation page (blocks scanner false-clicks); links expire in 7 days. Sent from nexa_admin@procdna.com.

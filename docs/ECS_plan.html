NEXA on ECS — containers, Fargate & a CI/CD you own

The plan moved. Instead of a single EC2 box you build and manage, NEXA now runs as a containerised service in the existing Auxo dev ECS cluster, against a separate schema in the shared RDS cluster, for a ~1-month first-version test before production on separate infra.

This doc explains the whole new shape: what Docker is, how the ECR / Fargate / ECS trio divides the work, where your service sits in someone else's VPC, how security groups and SSM apply, the IAM you need (incl. iam:PassRole), and the GitHub Actions CI/CD pipeline you'll run yourself.

For: the sole dev shipping NEXA to ECS Assumes: you get iam:PassRole & run CD 17 sections · diagrams · gotchas

How to read this

The earlier docs (NEXA AWS Course, Migration Runbook, microk8s comparison) all assumed you'd run NEXA on one EC2 box you own. Himanshu's approval changed that: you're deploying into a cluster ProcDNA already runs. That's a smaller job in some ways (no VPC/NAT/EC2/RDS to build) and a different job in others (you must containerise, and deploys work by shipping images, not by SSH-ing a box). This doc is the ECS-specific replacement for those assumptions.

NEXA in one breath A single Next.js 16 app (UI + API in one process), ~55 internal users, tracking BD opportunities through pricing → approval → SOW/PO. Login is Azure AD SSO; data in PostgreSQL; documents in object storage; mail via Microsoft Graph. Sensitive commercial data (rates, margins). Today on Vercel + Supabase; moving to ECS + RDS + S3.
The decision this doc bakes in Per the plan: you run your own CD. The assumption throughout is that the IT Team grants you scoped IAM including iam:PassRole, and you wire a GitHub Actions pipeline that builds the image, pushes it to ECR, and tells ECS to deploy. If instead they choose to run deploys for you, only Part V changes — everything else holds.

The app code is unchanged. What changed is everything around it — how it's packaged, where it runs, who owns the plumbing, and how it gets deployed.

The single biggest mental change: you stop shipping commands to a server and start shipping images to a registry. The SSM + git-pull deploy model from the EC2 runbook does not apply here.

The good news A large slice of the migration's hardest infra work — designing a VPC, NAT, ALB, security groups, provisioning RDS — is now ProcDNA's job, not yours. Your real work shrinks to four things: (1) write a Dockerfile, (2) wire image-based CI/CD, (3) the Supabase→S3 storage rewrite (still the big engineering task), and (4) DB schema + env/secret config.

2You are now a tenant, not an owner

On the EC2 plan you were the landlord — you owned the building and every utility in it. On ECS you're a tenant in a building ProcDNA runs. That reframes who you talk to and what you can touch.

What ProcDNA owns (you consume) The VPC and its subnets, the NAT gateway, the ECS cluster and its Fargate capacity, the shared RDS cluster, the ALB, and the IAM boundaries. You don't create or break these — you're given a slot inside them.
What you own (your service) Your Docker image, your task definition (image + CPU/RAM + env + roles), your service's desired count, your DB schema's contents, your S3 bucket's objects, and — under the assumed plan — your CI/CD pipeline.
The tenant risk to respect The RDS schema sits in a shared cluster, so be a good neighbour: your DB user is scoped to your schema only (you can't read or break other projects' data), and keep the Prisma connection count modest — see §7 and §11.

3Docker: a frozen box, not a box you set up

Hold onto your EC2 instinct — "a remote Linux where I run commands" — because Docker is the opposite philosophy, and the contrast is the fastest way to get it.

On EC2 you take an empty machine and mutate it by hand: install Node, git clone, npm ci, build, start. The box accumulates state; if it breaks you SSH in and fix it. It's a pet. Docker flips this: you write the entire setup down as a recipe, bake it into a frozen, self-contained package, then run that package. You never hand-tweak a running one — you rebuild and replace. Containers are cattle: disposable, identical, interchangeable.

The three words you must keep straight

WordWhat it isEC2 analogy
DockerfileA text recipe of every setup stepThe install script you'd run by hand
ImageThe recipe executed and frozen — Node + deps + your code + the build, sealed into one artifactA box set up perfectly, then snapshotted
ContainerA running instance of an imageThe box actually booted and running

One image → many identical containers. Build once, run anywhere, always identical. That reproducibility is the whole point: the thing CI built is byte-for-byte the thing that runs in production.

4The NEXA Dockerfile

Everything you used to do by hand on the EC2 box — install, generate, build — now happens once, at image-build time. Here's a production-shaped multi-stage Dockerfile for NEXA using Next.js standalone output (smaller, faster images).

# ---- 1. deps: install node_modules from the lockfile ----
FROM node:20-alpine AS deps
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci

# ---- 2. build: generate Prisma client + next build ----
FROM node:20-alpine AS build
WORKDIR /app
COPY --from=deps /app/node_modules ./node_modules
COPY . .
RUN npx prisma generate
RUN npm run build          # needs output:'standalone' in next.config

# ---- 3. run: a tiny image with only what runtime needs ----
FROM node:20-alpine AS run
WORKDIR /app
ENV NODE_ENV=production PORT=3000
COPY --from=build /app/.next/standalone ./
COPY --from=build /app/.next/static ./.next/static
COPY --from=build /app/public ./public
EXPOSE 3000
CMD ["node", "server.js"]
Two NEXA-specific gotchas (1) Add output: 'standalone' to next.config.ts or the standalone copy steps won't have anything to copy. (2) prisma generate must run inside the build (it's already above) — the generated client is platform-specific and isn't committed, so it must be produced in the Alpine image, not copied from your Windows machine.
What is not in the image No secrets, no database, no uploaded files. The image holds only the app. DATABASE_URL, NEXTAUTH_SECRET, Azure creds, etc. are injected at runtime by the task definition (§10). Documents go to S3, data to RDS — never inside the container (§5).

5Immutable infrastructure & the stateless rule

Two consequences of "containers are cattle" decide how NEXA must behave. Get these wrong and the app breaks in confusing ways on ECS.

Immutable, not mutated You never git pull + restart a running container. To ship a change you build a new image and ECS swaps the old container for the new one. Bad deploy? Redeploy the previous image tag. This is why the deploy-key-on-the-box idea from the EC2 plan disappears entirely — the code is baked into the image at build time.
The filesystem is ephemeral When a container stops, anything written to its local disk is gone. So: uploads must go to S3 (never local disk), the DB is external (RDS), and secrets come from env vars — not a file you edit. This makes the Supabase→S3 rewrite non-negotiable: any code path that writes a file locally will silently lose data on ECS.
Where NEXA's state lives on ECS Structured data → RDS (shared cluster, your schema). Documents (SOW/PO) → S3 (presigned URLs). Secrets/config → SSM Parameter Store. Logs → CloudWatch. The container itself holds zero durable state — that's what lets ECS kill and replace it freely.

6ECR · Fargate · ECS — three jobs people merge into one

The single most common confusion is treating these as one thing. They're three distinct services with three distinct jobs. Keep them separate and ECS stops being mysterious.

ECR — store
The registry. A warehouse that holds your images. Like a "git remote" but for images. Does nothing else.
Fargate — run
Serverless compute. Runs one container when told, with the CPU/RAM you specced. No server for you to manage.
ECS — orchestrate
The brain. Decides what runs, how many, watches health, swaps versions, wires the ALB.

The division of labour

ServiceWhat it knows / doesWhat it does not do
ECRStores & serves imagesNothing else — it's storage
FargateRuns one container exactly as instructedDoesn't know "how many," health, deploys, or the ALB
Amazon ECSDesired count, self-healing, rolling deploys, ALB registration, scalingDoesn't store images; doesn't provide raw compute
Restaurant analogy ECR = the freezer of prepared meals (images). Fargate = an oven that heats one meal when handed it. ECS = the head chef: decides how many meals to keep ready, watches them, tosses a burnt one and reheats a replacement, swaps the menu on a new release, and tells the front desk (ALB) which tables are ready. The freezer doesn't pick the menu; the oven doesn't decide how many to cook. The chef runs the show.

What ECS actually does (the orchestration jobs)

  1. Maintains desired count — "keep N NEXA tasks running," always.
  2. Self-healing — a task crashes or fails health checks → ECS kills it and starts a fresh one.
  3. Rolling deployments — new image → start new tasks, wait until healthy, drain & stop old ones, roll back if the new ones never go healthy.
  4. ALB integration — registers/deregisters tasks in the target group so the ALB only sends traffic to healthy ones.
  5. Reads the Task Definition and instructs Fargate exactly how to run each task.
  6. Scaling — add/remove tasks under load (autoscaling policies).

7The ECS hierarchy: Cluster → Service → Task Definition → Task

Inside ECS there's its own small ladder. Learn these four words and the AWS console stops being a wall of jargon.

Cluster
A pool of compute where containers run (Fargate capacity, or EC2 instances). Auxo's shared dev cluster — you get a slot in it, you don't create it.
Service
The long-running controller for your app. "Keep N tasks alive, wire them to the ALB, do rolling deploys." This is the thing you point CD at (ecs update-service).
Task Definition
The blueprint. Which image, how much CPU/RAM, container port, env vars, secrets, and which IAM roles. A new image = a new revision of this.
Task
A running instance of the task definition = your container(s) running. Fargate provides the compute for each task.
How they relate in one sentence The Service keeps N Tasks running from a Task Definition revision (which names an image in ECR), on compute supplied by Fargate, inside a Cluster, fronted by the ALB.

9Security groups & how traffic actually flows

A security group is a stateful virtual firewall wrapped around a resource. On ECS/Fargate, each task gets its own network interface (ENI) with a security group — so the rules attach to your task, just as they attached to the EC2 box before.

The two directions, and the one rule that confuses everyone

Inbound = who may start a connection to the task Only the ALB should. The task's security group allows inbound on port 3000 from the ALB's security group only (SG-references-SG), not from the internet. Users never reach the task directly — they hit the ALB, which forwards.
Outbound = connections the task starts The task initiates outbound to Azure AD (login), Microsoft Graph (mail), S3, and RDS. These leave via the NAT gateway. "No open inbound ports" does not block this.
"Outbound" ≠ "send only" — the stateful trick Inbound/outbound describe who initiated a connection, not which way data flows. Once the task dials out (e.g. git-less here, but Azure/Graph/S3 calls), the security group and NAT are stateful: they automatically allow the reply back in — with no inbound rule. That's why a sealed-inbound task can still log users in via Azure, send mail via Graph, and read/write S3. It only ever receives data as the answer to a call it placed.

Inbound is one door (the ALB). Everything the app reaches is an outbound call whose reply returns on the same connection.

The three security groups, layer by layer — exact ports

There's one SG per layer. Read each as "inbound = who may start a connection to me; outbound = where I may start connections." Because SGs are stateful, replies to allowed connections come back automatically — you never write a rule for the return direction.

The ports in one glance 443 HTTPS — users→ALB, and task→internet (SSO, mail, S3, ECR pulls). · 80 HTTP — optional, only to redirect to 443. · 3000 — the app's port; ALB→task only. · 5432 — Postgres; task→RDS only. Each "only" is the security: the task is reachable only from the ALB, the DB only from the task. Nothing in a private subnet is reachable from the internet.
Finer detail — why outbound is "to 0.0.0.0/0 on 443" and that's still safe The task must reach Azure AD, Graph, S3, and ECR — all public HTTPS endpoints with IPs that change — so the simplest rule is "outbound 443 to anywhere." That sounds open, but it only lets the task initiate calls out; nothing can initiate in (no inbound rule allows it). Combined with the private subnet (no public IP) and NAT, the task can talk to the world but the world cannot reach the task. If you wanted to tighten it, S3/ECR can go via VPC endpoints (keeping that traffic inside AWS, off the public path), leaving only Azure/Graph as true internet egress.
Tenant note — no SG cross-reference for you to arrange The security groups belong to ProcDNA, and IT configures the ALB + its networking. Per the MoM, no security-group cross-reference is required from you — you do not relay an ALB-SG ↔ task-SG ID between IT and Akshay. The section above is the conceptual model (how inbound/outbound actually work); the wiring is IT's to set, not a coordination task on your plate.

10SSM on ECS: secrets injection & the debug shell

SSM (AWS Systems Manager) played the lead role in the EC2 deploy plan (send-command + git pull). On ECS that deploy role is gone — but two other SSM features matter.

Parameter Store — where secrets live Instead of an .env on a box, NEXA's secrets sit in SSM Parameter Store (or Secrets Manager) as encrypted parameters. The task definition references them under secrets:, and the task execution role is allowed to read them. At launch, ECS fetches them and injects them as env vars into the container. Your code reads process.env.DATABASE_URL exactly as before — it just arrives from Parameter Store, not a file.
ECS Exec — the debug shell Need a shell inside a running container (the ECS equivalent of "SSH into the box")? aws ecs execute-command opens one through the SSM channel — no inbound port, same outbound-broker model. But this is a debug escape hatch, not the workflow: you don't fix containers, you replace them. Use it to look, not to change.
# task definition (excerpt) — secrets injected from Parameter Store
"secrets": [
  { "name": "DATABASE_URL",    "valueFrom": "arn:aws:ssm:...:parameter/nexa/dev/DATABASE_URL" },
  { "name": "NEXTAUTH_SECRET", "valueFrom": "arn:aws:ssm:...:parameter/nexa/dev/NEXTAUTH_SECRET" }
]
Why this is a security upgrade over Vercel Secrets never sit in GitHub, never sit in the image, and never sit in a file on disk. They live encrypted in Parameter Store, are decrypted only at task launch, and access is gated by IAM. GitHub only ever holds the AWS deploy credentials — not the app's secrets.
Two different SSM permissions — don't confuse them (1) Who reads at launch: the combined role holds ssm:GetParameters + kms:Decrypt — that's how the task pulls secrets when it starts. (2) Who writes/manages: my IAM user holds ssm:PutParameter + ssm:GetParameter(s) + kms:Encrypt/Decrypt on nexa/dev/* — so I can populate and verify the parameters myself, instead of routing every secret through IT/Himanshu. Two principals, two purposes, both scoped to nexa/dev/*. (See §11 for the exact policy lines.)

11IAM: one combined role + the CD user, and where iam:PassRole fits

ECS technically has two role slots in a task definition — an execution role and a task role — for two jobs at two moments. For the dev trial we collapse them into one combined role used in both slots: simpler to create, one ARN to track, one PassRole target. (We split them back for production — see below.)

SlotUsed byWhat it needs
Execution role slotThe ECS/Fargate agent, at launchPull image from ECR · read secrets from Parameter Store · kms:Decrypt · write CloudWatch logs
Task role slotYour app code, at runtimeThe app's own AWS calls — chiefly S3 (put/get/presign documents under nexa/*)
Dev simplification — one role in both slots Both executionRoleArn and taskRoleArn point at the same role, which carries all the permissions above. AWS allows this. Trade-off: the running app then also holds the secret-reading power, so a compromised app could read secrets directly — acceptable for a dev trial with fake data, not for prod. Split into two roles for production (it's on the prod-hardening list).

The combined role (created by IT — you only consume it)

# attached permissions on the ONE role
ssm:GetParameters      # on arn:aws:ssm:<region>:<acct>:parameter/nexa/dev/*
kms:Decrypt            # on the KMS key encrypting those params
AmazonECSTaskExecutionRolePolicy   # AWS-managed: ECR pull + CloudWatch logs
s3:PutObject/GetObject/DeleteObject # on arn:aws:s3:::<bucket>/nexa/*
s3:ListBucket          # on arn:aws:s3:::<bucket>  (prefix nexa/*)
# trust policy: ecs-tasks.amazonaws.com  (lets ECS assume it)
What iam:PassRole means here When your CD pipeline registers a task definition, it tells ECS: "run this task as this role." Handing a role to an AWS service is privileged, so AWS requires the caller to hold iam:PassRole on that role. You don't create the role (that's iam:CreateRole, IT's) — IT creates the one role and grants your CD user PassRole on its single ARN.

The IAM your CD user needs (the ask)

// scoped to NEXA — least privilege
// --- push the image ---
ecr:GetAuthorizationToken      # must be Resource: *  (account-level)
ecr:BatchCheckLayerAvailability, ecr:PutImage,
ecr:InitiateLayerUpload, ecr:UploadLayerPart, ecr:CompleteLayerUpload   # on the ECR repo ARN
// --- deploy to ECS ---
ecs:RegisterTaskDefinition     # must be Resource: *  (not scopable)
ecs:UpdateService, ecs:DescribeServices   # on the service ARN
ecs:DescribeTaskDefinition
iam:PassRole                   # on the ONE combined-role ARN
// --- load & read the secrets myself (Parameter Store) ---
ssm:PutParameter, ssm:GetParameter, ssm:GetParameters   # on parameter/nexa/dev/*
kms:Encrypt, kms:Decrypt       # on the params' KMS key (for SecureString)
Why the SSM + KMS lines (new) So I can load and manage the nexa/dev/* parameters myself instead of emailing each secret to IT/Himanshu. ssm:PutParameter writes a value, ssm:GetParameter(s) reads it back to verify, and kms:Encrypt/Decrypt are needed because the secrets are stored as encrypted SecureString parameters. All scoped to the nexa/dev/* path — I can't touch other projects' parameters. This is separate from the role's runtime ssm:GetParameters (which is what the task uses at launch); these are my permissions to populate the store.
The one gotcha that blocks deploys If iam:PassRole is missing or not scoped to the role ARN, RegisterTaskDefinition/UpdateService fails with "not authorized to perform: iam:PassRole" — even though every ecs:* is present. Confirm PassRole names the exact role ARN. Likewise, if the role's trust policy doesn't allow ecs-tasks.amazonaws.com, ECS can't assume it and the task never starts. Note the two Resource: * actions above (GetAuthorizationToken, RegisterTaskDefinition) can't be ARN-scoped — everything else is.

12CI/CD — CI is GitHub-only; CD ships an image

Split the two halves cleanly. CI proves the code is sound and touches no AWS. CD — which you own — builds the image, pushes it to ECR, and tells ECS to deploy.

CI — runs on every push On a GitHub-hosted runner, no AWS involved: npm ci → prisma generate → typecheck → next build. If the build breaks, the pipeline stops red and CD never runs. CI answers "is this code sound?"
CD — runs on merge to main Build the Docker image → push to ECR → register a new task-def revision → ecs update-service. ECS then does a rolling deploy on Fargate. CD answers "make it live."
The ECS deploy pipeline
push to mainyour commit reaches GitHub

▼ green

▼

No SSM send-command, no git pull on a box, no deploy key. The code is inside the image; the pipeline pushes an image and calls the ECS API. Fargate is invisible to the pipeline — it talks to ECR and ECS only.

Auth from GitHub → AWS Per the plan, long-lived AWS keys stored as GitHub Actions secrets (you accepted the trade for a stable, dependency-free CD as the sole dev). Use a dedicated, narrowly-scoped CD user — the §11 permissions only — never your personal IAM user. Mitigate the static-key risk with tight scope, a protected GitHub Environment for the secrets, and quarterly rotation. (OIDC remains the cleaner long-term option if you ever want short-lived creds — it needs the admin to create one role.)

13The GitHub Actions workflow

One file, two jobs. ci on a GitHub runner; deploy gated behind it.

# .github/workflows/deploy.yml
name: Deploy NEXA (dev)
on: { push: { branches: [main] } }

jobs:
  ci:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '20' }
      - run: npm ci
      - run: npx prisma generate
      - run: npx tsc --noEmit
      - run: npm run build        # gate: broken build never deploys

  deploy:
    needs: ci
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: aws-actions/configure-aws-credentials@v4
        with:
          aws-access-key-id:     ${{ secrets.AWS_ACCESS_KEY_ID }}
          aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
          aws-region:            eu-central-1
      - uses: aws-actions/amazon-ecr-login@v2
      - name: Build & push image
        run: |
          docker build -t $ECR_REPO:${{ github.sha }} .
          docker push  $ECR_REPO:${{ github.sha }}
      - uses: aws-actions/amazon-ecs-deploy-task-definition@v2
        with:
          task-definition: taskdef.json       # template w/ the new image tag
          service: nexa-dev
          cluster: auxo-dev
          wait-for-service-stability: true     # block until the new task is healthy
What each half talks to The deploy job talks to exactly two AWS services: ECR (push the image) and ECS (deploy). It never touches Fargate directly, never opens a connection to a task, never needs an inbound port on anything. wait-for-service-stability makes the Actions run go red if the rolling deploy doesn't stabilise — so a bad image surfaces immediately.

14Redirect URIs, Safe Links & the env vars that must match

Push for a stable subdomain (e.g. https://nexa-dev.procdna.com) rather than the raw ALB DNS name, so the ACM cert, Entra URIs, and Safe Links allowlist all stay put if the ALB is recreated.

17Glossary

Image
A frozen, self-contained snapshot of your app + its runtime + dependencies, built from a Dockerfile. Immutable. Stored in ECR.
Container
A running instance of an image. Disposable; its local filesystem is ephemeral.
Dockerfile
The recipe that defines how an image is built.
ECR (Elastic Container Registry)
AWS's image registry — stores and serves your images. The "warehouse."
Fargate
Serverless compute that runs containers with no servers for you to manage. The "oven."
ECS (Elastic Container Service)
The orchestrator — decides what runs, how many, heals failures, does rolling deploys, wires the ALB. The "chef."
Cluster
A pool of compute (Fargate capacity / EC2 instances) where tasks run.
Service
The controller that keeps N tasks running, integrates with the ALB, and performs rolling deploys.
Task Definition
The blueprint: image, CPU/RAM, port, env vars, secrets, IAM roles. A new image = a new revision.
Task
A running instance of a task definition — your container(s) on Fargate.
Task execution role
Used by the ECS agent at launch — pulls the image, reads secrets, writes logs.
Task role
Used by your app code at runtime — its own AWS calls, chiefly S3.
iam:PassRole
The permission to hand an existing role to an AWS service (here, ECS) when launching. Required by the CD user to deploy.
Parameter Store
SSM's encrypted key-value store for config & secrets, referenced by the task definition.
ECS Exec
Opens a shell inside a running container via SSM — a debug escape hatch, not the deploy path.
Rolling deployment
ECS starts new tasks, waits for health, drains and stops old ones — zero downtime, auto-rollback on failure.

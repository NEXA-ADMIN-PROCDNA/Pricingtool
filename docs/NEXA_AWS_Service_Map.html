The AWS service map

The first doc taught the core of AWS through NEXA. This one zooms out to the catalog: the services you'll actually provision for NEXA, plus the ones sitting one step away that you'll keep bumping into — organised by the job each does, so the 200-service menu becomes a short, navigable list.

Every service is tagged so you always know how much to care: Core (NEXA needs it), Adjacent (you'll meet it soon), or Further (just know the name exists).

~50 services, triaged Decision trees for the hard calls A consolidated NEXA shopping list

How to read this — the three tiers

AWS has well over 200 services. The number is paralysing only if you treat them as equals — they aren't. For a project like NEXA, a dozen do the real work, a dozen more are worth recognising, and the long tail you only need to be able to name so you know where to look later. This doc sorts every service into exactly those three buckets.

Core
NEXA needs it
You will create, configure, or directly depend on this to ship NEXA. There are about a dozen. Master these.
Adjacent
meet it soon
Not required for NEXA's first deploy, but one decision away — the natural upgrade, alternative, or "phase 2" service. Recognise these and know when they'd come up.
Further afield
know the name
Real, important services for other shapes of problem (big data, global scale, ML, IoT). You don't need them for NEXA; you only need to recognise the name so that when a problem calls for one, you know the category to search.
The pairing with doc 1 Where a Core service already got a deep dive in “AWS, explained through NEXA,” this doc gives the short version and points you back. Think of doc 1 as the tutorial and this as the map.

0The service landscape — nine jobs, not 200 services

The trick to not drowning: AWS services cluster into a handful of jobs. Almost everything you'll touch answers one of nine questions. Learn the nine buckets and any new service name slots into a bucket you already understand.

1 · Compute
"Where does my code run?" EC2, containers (ECS/Fargate), functions (Lambda), and the PaaS wrappers.
2 · Networking
"How do packets get there?" VPC, subnets, load balancers, DNS, CDN, the firewalls.
3 · Storage
"Where do files/disks live?" S3 (objects), EBS (disks), EFS (shared filesystems).
4 · Databases
"Where does structured data live?" RDS/Aurora (SQL), DynamoDB (key-value), ElastiCache (cache), Redshift (warehouse).
5 · Security & identity
"Who's allowed, and is it encrypted?" IAM, KMS, Secrets, ACM, WAF, the threat-detection family.
6 · Management
"Is it healthy, and what did it cost?" CloudWatch, CloudTrail, Organizations, the cost tools, infra-as-code.
7 · Developer tools
"How does code ship?" ECR, the Code* pipeline family (or, for NEXA, GitHub Actions).
8 · Integration
"How do parts talk async?" SQS (queues), SNS/EventBridge (events), Step Functions, SES (email).
9 · Analytics / AI
"Make sense of data / add intelligence." Athena, Glue, Kinesis, QuickSight, Bedrock/SageMaker.
NEXA touches just four jobs to go live Compute (1), Networking (2), Storage (3), Databases (4) — plus the cross-cutting Security (5) and a little Management (6). Jobs 7–9 are either already solved (you chose GitHub Actions for CI/CD) or simply not part of this app. That's the whole point of triage: a finance-data internal tool is a small, well-understood shape, and most of AWS's surface area is for other shapes.

1Compute — where your code runs

This is the category with the most options and the most agonising over them, so start with the decision rather than the list. The question is always: how much of the machine do you want to manage? AWS gives you a spectrum from "here's a bare VM, you run everything" to "just give me a function, I'll handle the rest."

The compute spectrum — more control ← → less to manage
EC2  ·  you own the OSa raw VM — patch it, supervise the process, scale it by hand · maximum control
▼
ECS / Fargate  ·  you own the containership a Docker image; AWS runs it · no OS to patch · rolling deploys
▼
App Runner / Beanstalk  ·  you own the code + a confighand over a repo/image; the platform wires up load balancing, scaling, TLS
▼
Lambda  ·  you own only a functionno server at all; runs on a request/event, scales to zero · least to manage

NEXA sits near the top by choice: a long-lived Next.js server is simplest to reason about on EC2 (and the SSO/cookie behaviour is easiest to get right there), with ECS/Fargate as the planned next rung down. Lambda is the wrong shape for a persistent SSR app — noted below.

Amazon EC2 Elastic Compute Cloud
NEXA needs it
A rented virtual machine — CPU, RAM, disk — that you fully control. The default place to run anything that needs a persistent process.
Reach for it when: you want maximum control or the simplest mental model, or you're running a long-lived server (like an SSR web app) and don't yet want containers.
NEXA: runs the Next.js process here (t3.large/t4g.large), under systemd/PM2, in a public subnet. Deep dive: doc 1 §7.
Amazon ECS + Fargate Elastic Container Service
NEXA's planned upgrade
ECS orchestrates containers; Fargate is the "serverless" mode where AWS runs them with no VM for you to patch. You ship a Docker image and define how many copies (tasks) to run.
Reach for it when: you want zero OS patching, clean rolling/blue-green deploys, and easy horizontal scale — and your app is stateless.
NEXA: the recommended graduation from EC2 once stable. Because NEXA is stateless (state is in RDS/S3), containerising is low-friction. Deep dive: doc 1 §17.
Amazon ECR Elastic Container Registry
with ECS
A private registry that stores your Docker images — the place ECS/Fargate pulls from.
Reach for it when: you containerise. CI builds the image and pushes it here; the service pulls the new tag to deploy.
NEXA: needed only on the ECS path — GitHub Actions pushes the image here, then updates the service.
AWS Lambda functions as a service
meet it soon
Runs a single function in response to an event (an HTTP call, a file landing in S3, a schedule). No server; scales to zero; you pay per invocation.
Reach for it when: you have small, event-driven, bursty jobs — a nightly cleanup, a "resize this upload," a webhook handler — not a persistent web server.
NEXA: not for the app itself (a long-lived SSR server fights Lambda's model and cold starts). But a fine future home for a side task like the Azure-AD user sync on a schedule.
AWS App Runner
alternative
A high-level "give me your container or repo and a port, I'll run it on the internet with autoscaling + TLS" service. Fargate with the knobs hidden.
Reach for it when: you want container simplicity without learning ECS task definitions, and you don't need fine VPC control.
NEXA: a plausible shortcut, but NEXA needs tight VPC placement (private RDS, SG references), which is exactly what App Runner abstracts away — so EC2→ECS gives more control here.
AWS Elastic Beanstalk
alternative
An older PaaS that provisions EC2 + load balancer + scaling from your code and a config file. Hides detail you'll often want to see.
Reach for it when: a firm already standardises on it, or you want a managed EC2 setup without hand-wiring.
NEXA: would work, but it obscures the very pieces this migration is meant to teach you. Skip unless the firm uses it.
Amazon EKS
know the name
Managed Kubernetes. The heavyweight container orchestrator for large, multi-team platforms.
NEXA: overkill. ECS is the right weight for one small app; EKS is for fleets of services.
Lightsail · Batch · Outposts
know the name
Lightsail = simplified VPS for tiny projects; Batch = run large batch compute jobs; Outposts = AWS hardware in your own data centre.
NEXA: none apply — listed so the names aren't mysterious.
The compute decision in one question "Is it a long-lived server, a container, or a short event-driven job?" Long-lived server → EC2 (then ECS). Container-native team → ECS/Fargate from the start. Bursty event jobs → Lambda. NEXA is a long-lived server, which is why the whole first doc lives on EC2.

2Networking & content delivery — how packets get there

This category builds the private network your resources live in and the doors that let the right traffic in. Doc 1 covered the core of it deeply (VPC, subnets, routing, security groups, ALB, Route 53); here's the catalog view plus the adjacent pieces — CDN, API gateway, private connectivity — you'll meet as soon as you go past one app.

Amazon VPC + subnets, route tables, IGW, NAT
NEXA needs it
Your private virtual network in a Region. Inside it you carve public/private subnets, set route tables, and attach gateways (IGW for two-way internet, NAT for outbound-only).
Reach for it when: always — it's the floor everything else plugs into.
NEXA: 10.0.0.0/16, public subnets for the app/ALB, private subnets for RDS, no NAT needed. Deep dive: doc 1 §4–5.
Security Groups & NACLs
NEXA needs it
The network firewalls. Security groups (stateful, attach to a resource, can reference other groups) are your main tool; NACLs (stateless, per-subnet) are a coarse backstop usually left at default.
Reach for it when: every resource — define what may reach it.
NEXA: App SG (443/80 in) and DB SG (5432 only from the App SG). Deep dive: doc 1 §6.
Elastic Load Balancing ALB / NLB / GWLB
NEXA needs it
Distributes incoming traffic across targets and health-checks them. ALB (layer 7, HTTP/S) is the web default; NLB (layer 4, raw TCP/UDP, extreme throughput) and GWLB (for inline appliances) are the specialised siblings.
Reach for it when: you need a stable front door, TLS termination, health checks, or HA across AZs.
NEXA: an ALB — the stable HTTPS front door that protects SSO. (NLB/GWLB don't apply.) Deep dive: doc 1 §8.
Amazon Route 53
NEXA needs it
AWS's DNS. Maps nexa.procdna.com to the ALB; also does health-based and latency-based routing if you ever need it.
Reach for it when: you need a name pointing at AWS resources (or AWS to host the zone).
NEXA: the stable subdomain — the SSO long-pole. Deep dive: doc 1 §9.
Amazon CloudFront CDN
meet it soon
A global content-delivery network: caches your content at edge locations worldwide so users hit a nearby copy. Also a natural place to attach WAF and serve a cert.
Reach for it when: you have a geographically spread audience and cacheable assets, or want edge DDoS protection.
NEXA: not needed at 55 internal users — but it's the honest answer to "US + India latency" if it ever mattered: put CloudFront in front, and users hit a near edge. A clean phase-2 lever, not a day-1 need.
Amazon API Gateway
meet it soon
A managed front door specifically for APIs — throttling, auth, request validation — usually fronting Lambda functions.
Reach for it when: you're building a serverless (Lambda-backed) API and want managed throttling/keys without running a server.
NEXA: not applicable — NEXA's API routes live inside the Next.js app behind the ALB, so the ALB already plays the "API front door" role.
VPC Endpoints & PrivateLink
meet it soon
A private path from your VPC to an AWS service (like S3) or another VPC's service, without traversing the public internet — and without a NAT gateway.
Reach for it when: a private-subnet resource needs to reach S3/other services, or security policy forbids public-internet egress.
NEXA: not needed while the app is in a public subnet. The moment security review moves the app to a private subnet, an S3 Gateway Endpoint is the cheaper, safer answer than a NAT gateway. Worth knowing now.
Transit Gateway · VPC Peering
know the name
Ways to connect many VPCs together (peering = 1:1; transit gateway = hub-and-spoke for many).
NEXA: only relevant once the firm runs many VPCs that must talk.
Direct Connect · Site-to-Site VPN · Global Accelerator
know the name
Private leased line to AWS / encrypted tunnel from on-prem / anycast network acceleration.
NEXA: enterprise-network plumbing — not part of this app.

3Storage — where files and disks live

Three storage shapes, and the mistake is using the wrong one. Objects (whole files, accessed by name) → S3. Block (a virtual disk attached to one server) → EBS. Shared filesystem (many servers mount the same folder) → EFS. Get the shape right and the service is obvious.

Three storage shapes
OBJECT — S3whole files by key · infinite · accessed over HTTPS · NOT a filesystem
use for: documents, uploads, backups, static assets — NEXA's SOW/PO files
BLOCK — EBSa virtual hard disk attached to ONE EC2 instance · the OS + app live here
use for: the boot/data disk of a server — NEXA's EC2 root volume
SHARED FILE — EFSa network filesystem MANY instances can mount at once
use for: shared state across many servers — NEXA doesn't need this
Amazon S3 Simple Storage Service
NEXA needs it
Object storage — files in named buckets, durable, effectively infinite, reached over HTTPS (not mounted). The default home for anything that's a "file."
Reach for it when: documents, user uploads, backups, exports, static web assets, data-lake files.
NEXA: the SOW/PO documents — private bucket, Block Public Access on, served via 1-hour presigned URLs. Deep dive: doc 1 §11.
Amazon EBS Elastic Block Store
comes with EC2
The virtual disk attached to an EC2 instance. gp3 is the standard general-purpose SSD. Lives in one AZ; snapshot it to S3 for backup.
Reach for it when: any EC2 instance — it needs a root volume. You rarely think about it beyond picking size/type.
NEXA: a small gp3 root volume (~20–30 GB) for OS + app + build. Documents go to S3, so it stays small. Mentioned in doc 1 §7.
S3 storage classes & Glacier
meet it soon
S3 isn't one price — it's tiers. Standard (hot), Infrequent Access (cooler/cheaper), Glacier (archive, retrieval in minutes-to-hours). Lifecycle rules move objects between them automatically.
Reach for it when: you have data that ages — keep recent docs hot, auto-archive old ones to cut cost.
NEXA: not needed now, but a tidy future optimisation — lifecycle old SOW/PO docs to a cheaper class after N months. Pair with the versioning that's already on.
Amazon EFS Elastic File System
meet it soon
A managed network filesystem (NFS) that many instances/containers can mount simultaneously — shared, POSIX, grows automatically.
Reach for it when: multiple compute nodes genuinely need to read/write the same files (legacy apps, shared upload dirs).
NEXA: deliberately not used — files go to S3, which is the cloud-native choice and keeps the app stateless. Know EFS exists so you don't reinvent it.
FSx · Storage Gateway · AWS Backup
know the name
FSx = managed Windows/Lustre filesystems; Storage Gateway = bridge on-prem storage to AWS; AWS Backup = centralised backup policy across services.
NEXA: AWS Backup could one day centralise RDS+EBS snapshots under one policy; the others are for enterprise/on-prem scenarios.

4Databases — where structured data lives

The richest category to get lost in, because AWS has a database for every data shape. But the choice is almost always made by the shape of your data, not by preference. NEXA is relational, so RDS — but knowing the alternatives tells you when you'd add one.

Is the data relational (tables, joins, transactions, strong consistency)?
Yes → RDS (or Aurora for scale). This is NEXA.
No → keep going ↓
Is it simple key-value / document lookups at huge scale, no joins?
Yes → DynamoDB — serverless NoSQL, single-digit-ms.
No → keep going ↓
Do you just need to cache hot data / sessions to take load off the DB?
Yes → ElastiCache (Redis/Memcached).
No → keep going ↓
Is it analytics over huge historical datasets (reporting, BI)?
Yes → Redshift (warehouse) or query S3 with Athena.
No → it's probably one of the specialised stores (graph, time-series, ledger) — rare.
Amazon RDS Relational Database Service
NEXA needs it
Managed relational databases — PostgreSQL, MySQL, MariaDB, SQL Server, Oracle. AWS handles backups, patching, and optional failover; you get an endpoint.
Reach for it when: you have relational data and want a managed SQL engine without running it yourself. The default for most apps.
NEXA: PostgreSQL 16 in private subnets, encrypted, SSL-only, 5432 from the App SG. Holds everything except documents. Deep dive: doc 1 §10.
Amazon Aurora
meet it soon
AWS's own PostgreSQL/MySQL-compatible engine — same SQL, but a re-architected storage layer that's faster, auto-scaling, and more resilient. A drop-in upgrade from RDS Postgres.
Reach for it when: you outgrow standard RDS on performance/scale, or want Aurora Serverless v2 to scale the DB with load.
NEXA: overkill at this scale, but the natural "the DB is the bottleneck" upgrade path — and it's Postgres-compatible, so the migration would be minimal if it ever came to that.
RDS Proxy
meet it soon
A managed connection pooler that sits between your app and RDS, sharing and reusing DB connections.
Reach for it when: many short-lived clients (e.g. Lambda) would otherwise exhaust the DB's connections.
NEXA: not needed — one long-lived EC2 process pools in-app (recall the max=1 fix, doc 1 §10.4). Relevant only if NEXA ever moved to many small compute units.
Amazon DynamoDB
meet it soon
Serverless NoSQL key-value/document store — single-digit-millisecond reads at any scale, no servers, no connection limits. A completely different data model from SQL (no joins).
Reach for it when: massive-scale, simple-access-pattern data — sessions, event logs, hot lookups — where you don't need relational joins.
NEXA: wrong fit for the core (it's deeply relational — opportunities join pricing joins approvals). Could one day back a specific high-write side-feature, but not the main data.
Amazon ElastiCache Redis / Memcached
meet it soon
An in-memory cache. Stores hot data (query results, sessions) in RAM so repeated reads skip the database entirely.
Reach for it when: a few expensive queries dominate load, or you want a fast shared session/store across many app instances.
NEXA: not needed at this scale, but the first thing you'd add if the dashboard's pipeline aggregations got slow under more users — cache the computed numbers instead of recomputing per request.
Redshift · DocumentDB · Neptune · Timestream · Keyspaces · QLDB
know the name
The specialised stores: Redshift = data warehouse (BI/analytics); DocumentDB = managed MongoDB-compatible; Neptune = graph; Timestream = time-series; Keyspaces = Cassandra; QLDB = ledger.
NEXA: none apply. If the firm ever wants heavy reporting across NEXA's data, Redshift (or Athena-over-S3) is the category — but that's a separate analytics project, not this app.
The database rule of thumb One relational database (RDS) handles the vast majority of normal apps. You add a second store only to solve a specific pain: a cache (ElastiCache) for read load, a warehouse (Redshift) for analytics, DynamoDB for a genuinely huge simple-access feature. Resist collecting databases — each one is operational surface area. NEXA correctly uses exactly one.

5Security & identity — who's allowed, and is it encrypted

For a finance-data app this is the category that earns sign-off, so it has the most Core entries. Split it in your head into three sub-jobs: identity (who can do what), encryption (keys & secrets), and protection/detection (firewalls & threat-spotting).

5.1 — Identity: who can do what

AWS IAM Identity & Access Management
NEXA needs it
The permission system for AWS itself — principals, policies, roles. Decides if an identity may perform an action on a resource. Default deny.
Reach for it when: always — every machine and human acting on AWS needs an identity and a scoped policy.
NEXA: the EC2 instance role (S3 + secrets + logs, least privilege) and the GitHub Actions OIDC deploy role. Deep dive: doc 1 §12.
IAM Identity Center formerly AWS SSO
meet it soon
Central human sign-in across many AWS accounts — often wired to the corporate IdP (Azure AD) so people log into the AWS console with their work identity.
Reach for it when: a team needs console/CLI access across several accounts without juggling per-account IAM users.
NEXA: this is how you and the SDE would get into the AUXO console — a firm/IT concern, distinct from how NEXA's own users log in (that's Azure AD inside the app). Don't confuse the two.
Amazon Cognito
know the name
Managed user sign-up/sign-in for your application's end users (user pools, social/enterprise federation).
Reach for it when: you need an app login system and don't already have one.
NEXA: not needed — NEXA already delegates user login to Azure AD via NextAuth. Cognito would be the AWS-native alternative if it didn't.

5.2 — Encryption: keys & secrets

AWS KMS Key Management Service
NEXA needs it
Holds the encryption keys behind every "encrypted at rest" in AWS — RDS storage, S3 objects, SSM SecureStrings. Access to keys is itself IAM-controlled and audited.
Reach for it when: you enable encryption anywhere (which, for finance data, is everywhere).
NEXA: the keys for RDS, S3, and secret encryption. Mostly you just choose "encrypt with KMS" and move on. Deep dive: doc 1 §13.2.
SSM Parameter Store & Secrets Manager
NEXA needs one
Encrypted stores for non-AWS secrets (DB password, Azure client secret, HMAC keys). Parameter Store is free & simple; Secrets Manager adds automatic rotation for a per-secret fee.
Reach for it when: you have any secret — never put it in a plaintext .env that bakes into images/snapshots.
NEXA: Parameter Store (SecureString) is the pragmatic default; pull at boot via the instance role. Deep dive: doc 1 §13.1.
AWS Certificate Manager ACM
NEXA needs it
Issues and auto-renews free TLS certificates for your domains, attached to the ALB/CloudFront. No more expiry-day fire drills.
Reach for it when: anything serving HTTPS through an ALB or CloudFront.
NEXA: the cert for nexa.procdna.com on the ALB listener. Must be in the ALB's Region. Deep dive: doc 1 §9.2.

5.3 — Protection & detection: firewalls and threat-spotting

AWS WAF Web Application Firewall
meet it soon
Rule-based filtering at the edge — rate-limiting, IP allow/deny, common-attack signatures — attached to an ALB or CloudFront.
Reach for it when: you want to throttle brute-force attempts or lock sensitive endpoints to an IP range.
NEXA: the plan calls for it on the secret-bearing endpoints (the emergency/login routes). The ALB is its attach point — a strong reason to run the ALB. Mentioned in doc 1 §8.
AWS CloudTrail
meet it soon
Records every API call made to AWS in your account — who did what, when, from where. The audit log of the cloud itself (distinct from your app's own activity log).
Reach for it when: always-on, ideally account-wide. It's how you answer "who changed that security group?"
NEXA: the firm likely enables this account-wide. It complements NEXA's in-app ActivityLog — CloudTrail audits the infrastructure, the app audits the business actions.
GuardDuty · Shield
meet it soon
GuardDuty = managed threat detection (watches logs for malicious behaviour); Shield = DDoS protection (Standard is automatic & free; Advanced is paid).
Reach for it when: GuardDuty as a low-effort always-on safety net; Shield Advanced only under real DDoS risk.
NEXA: GuardDuty is a sensible firm-wide default; Shield Standard already covers you for free. Neither is a build task for you.
Security Hub · Inspector · Macie · Config
know the name
Security Hub = aggregates findings; Inspector = scans for vulnerabilities; Macie = finds sensitive data in S3; Config = tracks resource configuration & compliance over time.
NEXA: firm/governance-tier tooling. Macie is conceptually interesting (it could flag if a confidential doc landed in the wrong bucket), but none are yours to set up.

6Management & monitoring — is it healthy, what did it cost

The category that keeps you sane after launch: observability, governance, cost, and infrastructure-as-code. You need a little of this for NEXA and should recognise the rest as the firm's domain.

Amazon CloudWatch
NEXA needs it
Logs, metrics, and alarms. Collects your app's stdout, tracks CPU/memory/RDS-connections/ALB-latency, and alerts when something crosses a line.
Reach for it when: always — it's your eyes on the new infra (Vercel's logging vanishes on migration).
NEXA: app logs → a /nexa/* log group; a couple of alarms (5xx spike, no healthy targets). Deep dive: doc 1 §14.
Cost Explorer & Budgets
turn it on early
Cost Explorer visualises spend by service/tag; Budgets emails you when spend crosses a threshold you set.
Reach for it when: day one — set a budget alarm before you forget a running resource.
NEXA: set a small monthly budget alert so a left-on dev instance or a surprise NAT/data-transfer charge can't quietly grow. The single best habit for a newcomer.
CloudFormation & AWS CDK infrastructure as code
meet it soon
Define your whole stack (VPC, EC2, RDS, IAM…) as code — CloudFormation in YAML/JSON, CDK in real languages (TypeScript!) — so it's repeatable, reviewable, and tear-down/rebuildable.
Reach for it when: you've clicked it together once and now want it reproducible across dev/staging/prod.
NEXA: build it in the console first to learn (doc 1 §18), then capture it as CDK (TypeScript suits your stack) so the dev→prod environments in doc 1 §16.4 are identical by construction. The right "phase 2" discipline. Terraform is the popular non-AWS alternative the firm may prefer.
AWS Organizations
meet it soon
Manages many AWS accounts under one umbrella — consolidated billing and Service Control Policies (guardrails that apply account-wide).
Reach for it when: a firm runs more than one account (almost all do).
NEXA: the AUXO account lives inside the firm's Organization — context for "where does a nexa-dev sub-account come from." A firm/IT concern, but know the word. Mentioned in doc 1 §2.1.
AWS Systems Manager SSM
you already use part of it
A grab-bag of ops tools: Parameter Store (you're using it for secrets), Session Manager (shell into an instance with no SSH key or open port), Patch Manager, Run Command.
Reach for it when: you want keyless, audited access to an instance, or to run a command/patch across fleet.
NEXA: Session Manager is a quiet win — connect to the EC2 box through the console with no SSH key and no port 22 open, which closes an attack surface. Parameter Store you already plan to use.
Control Tower · Trusted Advisor · Health Dashboard
know the name
Control Tower = automated multi-account landing zone; Trusted Advisor = best-practice/cost/security checks; Health Dashboard = AWS telling you about outages affecting you.
NEXA: Trusted Advisor is worth a glance for free cost/security tips; the others are firm-scale governance.

7Developer tools & CI/CD — how code ships

How code travels from the repo to running infrastructure. NEXA already made the key choice — GitHub Actions — so most of AWS's own Code* family is "know it exists." The one AWS piece you'll genuinely use on the container path is ECR.

Amazon ECR Elastic Container Registry
on the ECS path
Private Docker-image registry (covered under Compute too — it bridges both jobs). CI pushes images here; ECS/Fargate pulls them.
Reach for it when: you containerise — it's where the images live.
NEXA: only on the ECS upgrade. GitHub Actions builds the image, pushes to ECR, updates the service. Mentioned in doc 1 §17.
The AWS Code* family CodePipeline · CodeBuild · CodeDeploy
alternative to GitHub Actions
AWS's native CI/CD: CodePipeline orchestrates, CodeBuild builds, CodeDeploy rolls out. The AWS-resident equivalent of what GitHub Actions does for you.
Reach for it when: a firm wants its pipeline inside AWS rather than in GitHub, or needs deep CodeDeploy/ECS integration.
NEXA: not needed — you chose GitHub Actions + OIDC, which is simpler and keeps CI next to the code. Recognise these names if the SDE proposes them instead.
Why GitHub Actions + OIDC beats stored keys (recap) The modern pattern, already baked into NEXA's plan: GitHub and AWS trust each other directly via OIDC, so a deploy gets short-lived credentials and no AWS access key ever sits in GitHub. This is the keyless federation from doc 1 §12.6 — the same idea whether your CI is GitHub Actions or the Code* family.

8Integration & messaging — how parts talk asynchronously

These services let pieces of a system communicate without waiting on each other — queues, events, workflows, email. NEXA is a single app today, so these are mostly Adjacent — but they're exactly the tools you'd reach for to make it more resilient, and one (SES) directly shadows a feature NEXA already has.

Amazon SQS Simple Queue Service
meet it soon
A managed message queue. One part drops a job on the queue; another picks it up later. Decouples a slow task from the request that triggered it.
Reach for it when: a request kicks off slow work (generate a big export, send a batch of emails) that shouldn't block the user.
NEXA: the clean fix for the synchronous Excel export/parse that briefly blocks the Node thread (doc 1 §7.3) — enqueue the job, process it off the request path. A real, concrete future use.
Amazon SNS & EventBridge
meet it soon
Publish/subscribe messaging. SNS = fan-out notifications to many subscribers; EventBridge = a smarter event bus that routes events by rules (and connects SaaS/AWS events).
Reach for it when: one event should trigger several reactions, or you want services to react to each other without direct coupling.
NEXA: not needed now. If the approval workflow ever grew (e.g. "on approval, notify Slack and update a dashboard and email finance"), EventBridge is how you'd fan that out cleanly.
AWS Step Functions
meet it soon
A visual state-machine orchestrator — chains multiple steps (Lambdas, waits, branches) into a reliable, resumable workflow.
Reach for it when: a multi-step process needs reliability, retries, and visibility (think a long approval pipeline with timeouts).
NEXA: interesting parallel — NEXA's approval state machine is coded in-app today. At much larger scale, Step Functions is what you'd externalise it into. Not needed now; a nice mental bookmark.
Amazon SES Simple Email Service
shadows a NEXA feature
Bulk/transactional email sending at low cost — the AWS-native way to send application email.
Reach for it when: your app sends email and you don't already have a provider.
NEXA: notable because NEXA already sends approval email via Microsoft Graph (so it stays inside the corporate mail identity, and Safe Links work). SES is the AWS alternative you'd consider only if you ever wanted to move email off Microsoft — which the SSO/Safe-Links constraint argues against. Good to know the option, good to know why you're not taking it.

11Survival rules — staying oriented across all of it

A handful of cross-cutting reflexes that apply to every service above and save newcomers the most pain. Internalise these and the breadth stops being scary.

11.1 — "Is this global or Regional?" — ask it every time

The single most disorienting thing for newcomers is a resource "vanishing" because you're looking in the wrong Region. Keep this short list in your head:

Global (Region selector doesn't matter) IAM · Route 53 · CloudFront · WAF (for CloudFront) · Organizations · Budgets. Create them once; they're everywhere.
Regional (must match your Region) VPC · EC2 · RDS · S3 buckets (named globally, but data is Regional) · ALB · ACM (for an ALB) · SSM · CloudWatch · KMS keys. If you can't find something, check the Region selector first.

11.2 — Tag everything, from the first resource

A tag is a key/value label (e.g. Project=NEXA, Env=dev). It's tedious-seeming and enormously worth it: tags are how Cost Explorer tells you "NEXA cost $X this month," how you find every NEXA resource to tear down a dev stack, and how access policies can be scoped. Agree a tag convention with the SDE on day one and apply it to every resource.

11.3 — The cost reflexes that prevent bill surprises

ReflexWhy
Things bill while idle. An EC2 instance, RDS, a NAT gateway, an ALB all cost money just sitting there — not only when used.The classic newcomer surprise is a forgotten dev instance running all month. A Budget alarm (service #13) is the cheap insurance.
Data transfer out costs; cross-AZ/cross-Region costs.Moving bytes out of AWS, or between AZs/Regions, has a price. Keep NEXA's pieces in one Region (doc 1 §3) partly for this.
NAT gateways and idle ALBs are quiet line items.Both bill hourly + per-GB. NEXA avoids NAT by design; the ALB is a deliberate, known cost.
Free tier expires.The 12-month free tier lulls you; set the budget alarm so the cliff doesn't surprise you.

11.4 — The two-question debugging reflex (from doc 1, worth repeating)

When anything won't work, ask in order:
  1. Is it a place problem? Timeout / connection refused / "can't reach" → routing or a security group. Check subnets, route tables, SGs.
  2. Is it a trust problem? "AccessDenied" / "not authorized to perform…" → IAM. Check the role's permission policy (and, for assume-role failures, its trust policy).
Naming which of the two it is tells you which console to open — and that's most of the battle. Almost every AWS error a newcomer hits is one of these two sentences.

11.5 — Don't collect services

The discipline that matters most Every service you add is operational surface area — another thing to secure, patch, pay for, and reason about. NEXA's whole footprint is ~13 services on purpose. Resist adding a cache, a queue, a second database, or a fancier compute model until a specific pain demands it. The mark of someone who actually understands AWS isn't how many services they use — it's how few they need to.

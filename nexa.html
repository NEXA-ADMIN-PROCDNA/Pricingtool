NEXA Migration to AWS (AUXO): architecture briefing

NEXA is ProcDNA's internal Business Development & Pricing application. It currently runs on Vercel + Supabase and we want to move it onto the firm's AUXO AWS account. This document gives you the technical picture about what exists today, the stack I'm proposing (EC2 + RDS + S3) with compute sizing, the networking/security model, and the IAM access I'd need — so you can validate the design and allocate the right resources and permissions.

55+ users and would scale up· US + India Next.js 16 PostgreSQL + object storage Azure AD SSO Financial data (pricing, rates, margins, SOW/PO)

0Purpose & summary

The ask: move NEXA from Vercel + Supabase to AWS (AUXO) without changing the application architecture and only the hosting/data providers. One constraint is a stable public URL: we use Microsoft SSO and email "Safe Links", both of which are registered against a domain, so the public entry point must never change across restarts or redeploys.

1Some details about the application and stack

The application, NEXA tracks opportunities (BDs) through pricing, multi-level approval, SOW/PO signing, and project-code stages. It is used by roughly 55 internal staff across US and India, with role-based visibility (SEL-> D -> ED -> PARTNER → ADMIN {finance team}). Concurrency is low (internal tool — a handful to low-tens of simultaneous users), but the data is commercially sensitive: deal pricing, cost & bill rates, gross margins, discounts, client billings, and signed SOW/PO documents.

NEXA is a single Next.js 16 (App Router) codebase that contains both the UI and the API in one deployable unit so there is no separate backend service to host:

  • Frontend = React 19 components (pages, the pricing editor, tables, forms) rendered by the same Next.js server.
  • Backend = "API routes" inside the same project (app/api/**) that handle every create/update/delete, file upload, and approval, plus server-side data fetching via Prisma.

So one process serves HTML and JSON. That's why a single app server (EC2 or one ECS service) is all that's needed. state lives entirely in the database and object store, which makes the app horizontally scalable and easy to containerise later.

5Networking & security model

Standard two-tier isolation: the app is reachable from the internet; the database is not.

VPC  (e.g. 10.0.0.0/16)  — target region
├─ Public subnet   10.0.1.0/24   (AZ-a)  →  EC2 app server (+ ALB)
├─ Private subnet  10.0.11.0/24  (AZ-a)  →  RDS
└─ Private subnet  10.0.12.0/24  (AZ-b)  →  RDS (DB subnet group needs 2 AZs)

Security groups

  • App Security Group (EC2): inbound 443/80 from internet (or from the ALB);
  • DB Security Group (RDS): inbound 5432 only from the App SG — referenced by SG, not IP. Nothing else can reach the database.

Data protection

  • At rest: RDS storage encryption (KMS) + S3 default SSE.
  • In transit: HTTPS to the app; sslmode=require to RDS.
  • Isolation: RDS not publicly accessible; private subnets.
  • Identity: MFA and (SSO) Conditional Access inherited from Azure AD.

Application-level security already in place now

Beyond the AWS infrastructure controls above, NEXA already ships these protections in the application itself — they carry over to AWS unchanged:

ControlHow it works
Single Sign-On (Azure AD)Login is delegated to corporate Azure AD over OIDC — the app stores no passwords of its own. MFA / Conditional Access is enforced at the identity provider.
Provisioning gateOnly users who are pre-provisioned and marked active are allowed in; simply having a tenant account is not enough to sign in.
Signed session tokensSessions are HMAC-signed JWTs held in httpOnly + Secure + SameSite cookies with an 8-hour expiry — not readable by browser scripts and not forgeable.

Role-based access control (RBAC)Every data query is scoped by the user's role (SEL → Director → ED → Partner → Admin), so users only ever receive the records they're entitled to see.
SQL-injection protectionAll database access goes through Prisma's parameterized query builder — there is no raw, string-concatenated SQL anywhere in the app.
XSS protectionThe React / Next.js UI auto-escapes all rendered output, so user-supplied content (names, notes, justifications) cannot inject executable script into the pages.
CSRF protectionThe authentication layer issues anti-CSRF tokens on its sign-in / sign-out endpoints, preventing forged cross-site auth requests.
Private document storageSOW / PO files are never publicly accessible; they are served only through short-lived (1-hour) signed URLs scoped to a single object.
Tamper-proof email actionsThe one-click approve / reject email links carry an HMAC-signed, time-limited token that cannot be forged or altered .
Secret hygieneCredentials live in environment / secret storage and are never committed to the codebase (enforced via .gitignore); all secrets are rotated as part of the migration.

8Flow diagrams — how the mechanics work

8.1 — A normal request & authentication

Every request is gated centrally before it reaches any page or API. This is the path for both an HTML page load and an API call.

  1. Browser — sends the request with the session cookie (a signed JWT) over HTTPS to the stable subdomain.
  2. Edge — Nginx / ALB terminates TLS and forwards to the EC2 app, adding X-Forwarded-Proto: https.
  3. Auth gate (proxy.ts) — the global middleware verifies the JWT. No valid token → redirect to /login → Azure AD SSO → the app confirms the user is provisioned & active → a fresh JWT is issued.
  4. Handler — request reaches a Server Component (page) or an API route. The token yields { id, role, location }.
  5. RBAC — the user's role is turned into a query filter (SEL sees own; Director/ED down the hierarchy; Partner/Admin all), applied to the Prisma query.
  6. Data — Prisma → RDS (PostgreSQL) over SSL. Documents are returned as short-lived S3 presigned URLs, never as raw files.
  7. Response — rendered HTML (pages) or JSON (API) back to the browser.

8.2 — Email approval & one-click verification

Approvers can act straight from their inbox without logging in. The security of that rests on a signed, expiring token — this is the "email verification" mechanism.

  1. Request raised — a user sends an opportunity for approval. An ApprovalRequest row is created and the opportunity stage advances.
  2. Email sent (MS Graph) — an email goes to the approver with Approve / Reject buttons. Each button's URL carries a token: 
    // HMAC-SHA256 signed, 7-day expiry
token = base64url( approvalId | approverId | action | expiry )
        + "." + HMAC_SHA256(payload, SECRET)
    The signature means the link cannot be forged or tampered with — change any field and the signature no longer matches.
  3. Approver clicks (GET) — hits /api/approvals/email-action?token=…. The server verifies the token: signature valid, not expired, the approver matches, the request is still PENDING. It then renders an "Are you sure?" confirmation page. GET never changes anything — deliberately, so email-scanner / Safe-Links pre-fetches can't auto-approve.
  4. Approver confirms (POST) — the confirm button POSTs the same token. The server re-verifies, updates the ApprovalRequest to APPROVED/REJECTED, moves the opportunity stage, and (for a pricing rejection) cascades to invalidate the parallel SOW track.
  5. Result email — both requester and approver are emailed the outcome. The original inbox links are now inert (the request is no longer PENDING).

Why this matters for hosting: the email links are absolute URLs built from the app's base URL, and Safe Links rewrites/pre-scans them — another reason the public domain must be stable and the GET action must be side-effect-free.

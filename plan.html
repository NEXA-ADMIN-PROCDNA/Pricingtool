ProcDNA · NEXA — Business Development & Pricing Tool

AWS Migration & Architecture Plan

A working document for moving NEXA off Vercel + Supabase onto the firm's AUXO AWS account, onto a ProcDNA GitHub repo, and for explaining how the system is built and secured. Written to be shared with the IT team and the senior SDE.

Stack today: Next.js 16 (App Router) · React 19 · Prisma 7 · PostgreSQL (Supabase) · NextAuth v4 (Azure AD) · Microsoft Graph (mail) · Supabase Storage. Region fra1 (Frankfurt).

Current state → target state, at a glance

ConcernToday (Vercel + Supabase)Target (AWS · AUXO)
Source controlPersonal GitHub repo, personal commit identityRepo under the ProcDNA org, work identity, branch protection
ComputeVercel serverless functionsEC2 (Node + Nginx) to start → optionally ECS Fargate later
DatabaseSupabase Postgres (pgBouncer pooler)Amazon RDS for PostgreSQL, private subnet, SSL
File storageSupabase Storage (SoW_bucket, PO_busket)Amazon S3, private buckets, presigned URLs

The one constraint that drives everything: the public URL must be stable. Microsoft SSO redirect URIs and email "Safe Links" are registered against a domain, so nothing external may ever point at a raw EC2 IP. Solve that once (Elastic IP or a load balancer behind a fixed subdomain) and an EC2 stop/start, a redeploy, or a scale event never breaks login or email.

1Moving the repo into the ProcDNA GitHub organisation

You are on a personal repo committing as your personal GitHub identity. The goal is a repo owned by the ProcDNA organisation, with you (and the team) contributing under work identities, managed with the IT team. There are two distinct things to fix: where the repo lives (ownership) and who the commits are attributed to (identity).

1.1 — Two ways to land it in the org

OptionWhat happensPros / cons
A. Transfer the existing repo
Preserves history		 GitHub's Settings → Transfer ownership moves the repo (with all commits, branches, issues, PRs, stars) to the ProcDNA org. IT must accept the incoming transfer and you need permission to create repos in the org. Keeps full history and metadata. But it carries your entire commit history — including personal-email commits and anything ever committed (see §1.3 secret scan first).
B. Fresh repo, push code in
Cleanest start		 IT creates an empty private repo in the org and grants you write. You add it as a remote and push. You can push full history (--mirror) or start from a single clean commit. You choose how much history travels. A clean initial commit drops historical baggage (the junk files, any old secrets) but loses the commit timeline. Best when the data so far is throwaway anyway.
Recommendation: Option B with a clean initial commit. Your data is currently fake/test, the history contains accidental junk (the mangled git commit -m … filenames, dummy.txt with merge-conflict markers, test_file.txt), and a fresh start guarantees no historical secret or personal email rides into a firm-owned repo. If the IT/security team specifically wants the audit trail of history, use Option A but only after the secret scan in §1.3.

Concrete steps (Option B, clean start)

# 0) Ask IT to: create an empty PRIVATE repo `nexa` (or `procnexa`) under the ProcDNA org,
#    add you as a Maintainer/Write member, and confirm the org's default branch + rules.

# 1) Set your WORK identity for this repo (local override, doesn't touch global config)
git config user.name  "Shreeraj Deshmukh"
git config user.email "shreeraj.deshmukh@procdna.com"   # your @procdna address

# 2) Remove junk before it ever reaches the org repo (see the audit doc, §R1–R4)
git rm -f dummy.txt test_file.txt app/main.txt
git rm -f -- "errole changed*"  "app/errole changed*"   # the botched-commit files

# 3) (clean-start variant) collapse history into one commit
#    keep the working tree, drop the old timeline:
rm -rf .git
git init -b main
git add -A
git commit -m "Initial import: NEXA BD & Pricing Tool"

# 4) Point at the org repo and push
git remote add origin https://github.com/ProcDNA/nexa.git
git push -u origin main

Full-history variant of step 3–4 (Option A-style push, no rm -rf .git):

git remote add origin https://github.com/ProcDNA/nexa.git
git push origin --all
git push origin --tags

1.2 — Fixing commit identity

Setting user.email (step 1 above) fixes future commits. Two more points:

  • GitHub links commits to accounts by email. For your commits to show as "you" on the org repo, your @procdna.com address must be added to your GitHub account (Settings → Emails), or use the GitHub-provided noreply address. The IT team may instead want you on a separate work GitHub account that's a member of the org — clarify which.
  • Past commits keep their old author email unless you rewrite history. With a clean initial commit (Option B) this is moot. If you keep history and must re-attribute, git filter-repo --mailmap can map personal→work email, but rewriting shared history is disruptive — only do it before anyone else clones.
  • Authentication to the org repo: use a Personal Access Token (fine-grained, scoped to the org) or SSH key tied to your work identity, not your personal password.

1.3 — Secret scan & hygiene (do this BEFORE handing the repo to IT)

Critical for finance data: once the repo is in the org it is visible to more people. Before transferring or pushing full history, scan the working tree and git history for any committed secret. .gitignore already excludes .env, which is good — but verify nothing slipped in earlier.
# scan committed history for secrets (run one of these)
gitleaks detect --source . --redact
# or
docker run --rm -v "$(pwd):/repo" trufflesecurity/trufflehog git file:///repo

If anything is found (an old SUPABASE_SERVICE_ROLE_KEY, an Azure secret, NEXTAUTH_SECRET, EMERGENCY_TOKEN, …): rotate that credential (assume it is compromised) and prefer the clean-start option so it never enters the org repo. You are rotating all secrets during the AWS move anyway, which is the ideal moment.

Also commit a tidy repo: remove the redundant duplicates flagged in the audit (lib/prisma.js, the app/ duplicate configs, the boilerplate SVGs, the cors dependency) so the firm's first impression of the codebase is clean.

1.4 — Org repo settings to ask IT for

  • Private repository; add a team (not individuals) for access; you as Maintainer.
  • Branch protection on main: require a pull request + at least one review, require status checks (lint/build) to pass, no direct pushes.
  • Secret scanning + push protection (GitHub Advanced Security if the org has it) so a secret can never be pushed again.
  • Dependabot alerts/updates — relevant because the app pins bleeding-edge and one known-vulnerable dependency (xlsx@0.18.5).
  • A CODEOWNERS file and a short README/CONTRIBUTING so the team can onboard.
  • Decide the deploy trigger now: GitHub Actions building and deploying to AWS (see §3) — the org will likely want OIDC federation from Actions to AWS rather than long-lived AWS keys.

2Database & storage — Amazon RDS + Amazon S3

Your instinct (RDS + S3) is the right shape: the app already splits into a relational database (everything except files) and object storage (the SOW/PO documents). RDS replaces Supabase Postgres; S3 replaces Supabase Storage. Because the current data is throwaway, there is no data migration — only the schema and an empty bucket need to exist.

Two things get simpler, one needs care:
  • simpler No more pgBouncer. Supabase forced the transaction-mode pooler (port 6543) because Vercel opened many short-lived serverless connections. A single long-lived Node process on EC2 doesn't, so point Prisma straight at RDS on 5432. DATABASE_URL and DIRECT_URL can converge on the same endpoint.
  • simpler You can run real migrations (prisma migrate deploy) instead of hand-applying SQL.
  • care The connection pool in lib/prisma.ts is hard-capped at max=1 in production — correct for Vercel, a severe bottleneck on one EC2 process where the whole app would serialise through a single connection. Raise it (e.g. 10) sized to the RDS instance's connection limit. (Audit item A1.)

Cut-over for the DB (no data to move)

# point Prisma at RDS, then build the schema and seed
DATABASE_URL="postgresql://USER:PW@nexa.xxxx.eu-central-1.rds.amazonaws.com:5432/nexa?schema=procdna_database&sslmode=require"
npx prisma migrate deploy        # or: prisma db push  — creates tables/enums fresh
npx tsx prisma/add-admin.ts      # bootstrap the first ADMIN
npx tsx prisma/sync-azure-users.ts   # pull the real BD users from Azure AD

Gotcha: prisma/sync-azure-users.ts reads AZURE_TENANT_ID / AZURE_CLIENT_ID / AZURE_CLIENT_SECRET (no _AD_), while the app uses AZURE_AD_*. Provide both naming styles in the EC2 environment, or the sync silently exits. (Audit A10.) And prisma/seed.ts references old stage names — treat it as historical sample data, not a live seed.

2.2 — Amazon S3 (SOW & PO documents)

This is the one part that is real code work, not just configuration — the storage layer in lib/supabase.ts plus the SOW/PO upload routes get rewritten to the AWS SDK. The bucket starts empty, so nothing migrates.

SettingRecommendation
BucketsOne private bucket (or two) for SOW/PO objects, keyed by opportunity, e.g. nexa-docs/<oppId>/<version>_<ts>.pdf
Public accessBlock Public Access = ON. Files are only ever reachable through short-lived presigned URLs.
EncryptionDefault SSE (SSE-S3 or SSE-KMS) — encryption at rest by default.
VersioningON — recovers an overwritten/deleted document (the app does soft-deletes; S3 versioning is the safety net).
Access from appIAM instance role on EC2 (or task role on ECS) granting s3:GetObject/PutObject on this one bucket. No AWS keys in .env.
URLs@aws-sdk/s3-request-presigner, expiresIn: 3600 — same 1-hour expiry the app already assumes.
The single highest-impact new risk is a mis-set bucket. One "Block Public Access = off" click and every confidential pricing/SOW document is world-readable. Keep buckets private; serve only via presigned URLs; constrain the upload presign to a content-type + size limit (which also fixes the "client-claimed MIME" weakness, audit S11).

2.3 — Code that changes

  • lib/supabase.ts → a new lib/s3.ts: getSupabase() → an S3 client; getSignedUrl() → presigner. Keep the 55-min cache but add eviction (on a long-lived process the current cache grows unbounded — audit A5).
  • The four document routes (…/sow, …/sow/confirm, …/po, …/po/confirm): swap "signed upload URL" calls for S3 presigned PUT.
  • Env: drop SUPABASE_*; add S3_BUCKET + region (credentials come from the instance role, not env).

3Compute — ECS vs EC2, and machine sizing

Next.js 16 needs a live Node process (next start) — it is not a static site. The question is what runs that process.

3.1 — The realistic options

OptionWhat it isGood for you?
EC2 + Nginx + PM2/systemd
recommended to start		 A single Linux VM. You install Node, run next start under a process manager, put Nginx in front for TLS + reverse proxy. Elastic IP or an ALB gives the stable address. Yes, to begin. Simplest mental model for an AWS newcomer, fewest moving parts, easiest to reason about for login/cookies. You patch the OS and manage uptime yourself.
ECS on Fargate
the "do it properly" upgrade		 You containerise the app (Dockerfile) and AWS runs the container serverlessly — no VM to patch. An ALB fronts it; rolling deploys and rollbacks are built in; scale by task count. Great target once stable. No server patching, clean blue/green deploys, easy horizontal scale. Costs more setup learning (Docker, task defs, ECR) up front.
ECS on EC2 Same ECS orchestration but you own the underlying EC2 capacity. Only if the firm standardises on it or you need cost control at scale. More to manage than Fargate.
Elastic Beanstalk / Amplify Higher-level PaaS wrappers. Beanstalk can work but hides detail you'll want; Amplify is awkward for a full Node-server Next app. Skip unless the firm already uses them.
Recommendation: Start on a single EC2 (Nginx + systemd-managed next start, Elastic IP or ALB, ACM TLS). It gets you live fastest and is the easiest place to get the SSO cookie/redirect behaviour right. Plan to graduate to ECS Fargate once the app is stable and you want zero-downtime deploys and no OS patching. The app is already deploy-from-one-branch and stateless (state lives in RDS/S3), so containerising later is low-friction. Ask the SDE which the firm prefers to support — match their standard if they have one.
Whichever you pick, three things must be configured or login breaks:
  1. TLS terminates upstream (Nginx/ALB), so the Node app sees http internally. Set X-Forwarded-Proto and NEXTAUTH_URL=https://… so secure cookies are issued correctly. (Audit A2.)
  2. Stable address — Elastic IP or ALB DNS behind a fixed subdomain, registered as the Azure AD redirect URI and MAIL_BASE_URL.
  3. Process supervision — systemd Restart=always (or PM2), so a crash or reboot brings the app back automatically. Vercel did this for free.

This section is deliberately honest — it separates what the code actually implements today from what AWS adds and what still needs doing before this is genuinely finance-grade. 4.3 — Gaps that must be closed for finance-grade security

These are real and should be on the migration checklist. They exist today and the move to AWS does not fix them by itself.

GapRiskFix
Shared-password backup login (/login2)Critical One shared CREDENTIALS_SECRET logs you in as any user, including ADMIN. Full account takeover if it leaks.Remove it (go SSO-only), or replace with per-user hashed passwords. At minimum IP-restrict + rotate.
Emergency admin via URL token (/api/emergency)High Static token in the query string → leaks into logs/history; grants instant 8-hour admin.POST + short-lived signed token + IP allow-list/WAF, or remove in prod.
Authorization on writes (IDOR)High Most mutating API routes check that you're logged in but not that you own the record — a user can act on another team's opportunity, documents, or pricing by changing the ID.One shared ownership/role guard at the top of every mutating route (the read-side filter already exists; mirror it for writes).
Stored XSS / HTML injectionHigh/Med The email-action pages and the outbound emails interpolate user text (opportunity names, justifications) into HTML without escaping.HTML-escape every interpolated value (one small helper).
No real-time revocationMedium Role/active status is cached in the 8-hour JWT; deactivating a user takes effect only on next sign-in.Re-check isActive/role server-side periodically, or shorten the session.
Vulnerable dependency (xlsx@0.18.5)Medium Known prototype-pollution / ReDoS CVEs; it parses admin-uploaded spreadsheets.Move to the patched SheetJS build or exceljs.
Audit logging not enforcedMedium Finance data should have a "who viewed/changed what, when" trail; the tables exist but writes are inconsistent.Write ActivityLog on every create/update/approve/export; consider read-logging for documents.
No rate limiting todayMedium On raw EC2 the secret-bearing endpoints are brute-forceable without Vercel's shield.Nginx limit_req / AWS WAF (part of §3 setup).
Bottom line for IT/security sign-off: the foundations are sound (SSO, signed sessions, role-scoped reads, private storage, parameterised queries), and AWS adds the infrastructure controls (encryption, isolation, secrets, backups, MFA). But before this holds real client financials, the shared-password login, the emergency endpoint, and the write-side authorization (IDOR) items are must-fix, and audit logging should be turned on. Treat those four as release-blockers for the production cut-over.

5.2 — Frontend vs backend (it's one integrated Next.js app)

You're right that it's integrated — there is no separate backend service. Next.js App Router blends both, and the distinction is which kind of file handles a request:

"Frontend" (runs in browser)"Backend" (runs on the server)
Client Components — files with 'use client' at the top.
e.g. PricingDrawer.tsx, OpportunityTable.tsx, the modals/forms, the badges/selects.
They handle interactivity (state, clicks, editing) and call the API routes with fetch(). 		Server Components & pages (no 'use client') — e.g. dashboard/page.tsx, opportunities/[id]/page.tsx. They run only on the server and fetch data directly via lib/db/* (no HTTP round-trip).
API Routes — app/api/**/route.ts. The JSON HTTP backend: every create/update/delete, uploads, approvals, export. Each calls getAuthToken() then Prisma.
lib/ services — auth, Prisma client, mail, storage, pricing math.

So a page load is mostly server-rendered (Server Component → lib/db → RDS → HTML), and any change the user makes goes Client Component → fetch → API Route → Prisma → RDS. Same codebase, same deployed process.

5.4 — The opportunity lifecycle (approval state machine)

The heart of the domain. An opportunity moves through stages as pricing and SOW get prepared, approved, and signed. Pricing approval and SOW verification can run as parallel tracks, and a pricing rejection/withdrawal cascades to invalidate the SOW track.

LEAD("WLBU Pending" in the UI) — create a pricing version
▼
PRICE_LINKING_PENDINGmark a version as final
▼
PRICE_LINKEDrequest pricing approval
▼
APPROVAL_PENDINGapprover decides
approve → SOW_PENDING  ·  reject → back to PRICE_LINKED  ·  withdraw (owner) → PRICE_LINKED
▼
SOW_PENDINGupload SOW + PO, set pre-contract-agreed
▼
SOW_SUBMITTEDrequest SOW verification
▼
SOW_REVIEW_PENDINGapprover verifies documents
approve → TO_BE_ARCHIVED  ·  reject → back to SOW_SUBMITTED
▼
TO_BE_ARCHIVEDdeal effectively done · status set manually to WON

Parallel-track rule: rejecting or withdrawing the PRICING request auto-withdraws any pending/approved SOW verification and resets the opportunity to PRICE_LINKED. Status (OPEN/WON/LOST/ABANDONED) is a manual dropdown, decoupled from stage. Editing the date window resets pricing and rolls the stage back.

6.3 — Pricing engine

app/opportunities/[id]/pricing/utils.ts is the pure-function math core: it turns staffing rows × weekly hours × rates into billings, cost, gross margin, offshore %, effective rate, discount/premium, and FTE — and does all the calendar/working-day/proration logic in UTC so results are timezone-independent. types.ts shares the shapes; the five pricing sub-tabs (Basic Details, Efforts, Other Cost, Financial, Schedule of Payments) are the UI over it.

Putting it together — a migration sequence

  1. Repo: secret-scan history → clean junk → create the ProcDNA org repo (IT) → push with your work identity → set branch protection + Dependabot.
  2. Accounts & network (with the SDE): AUXO account access, region eu-central-1, a VPC with a public subnet (app) + private subnet (RDS), security groups.
  3. Domain & TLS: get the stable subdomain (Route 53 / corporate DNS), an ACM cert, decide Elastic IP vs ALB. Do this early — it's the long pole and the SSO/Safe-Links constraint.
  4. Data layer: RDS (private, encrypted, SSL) + an empty private S3 bucket with an instance role.
  5. Code: rewrite the storage layer (Supabase → S3); raise the Prisma pool size; trust X-Forwarded-Proto; harden the chunked-cookie detection. Fix the four release-blockers from §4.3 (login2, emergency, IDOR, audit logging).
  6. Compute: stand up EC2 (Nginx + systemd) or an ECS Fargate service; secrets in SSM/Secrets Manager; logs to CloudWatch; /healthz.
  7. Identity: add the new redirect URI to the Azure App Registration before go-live (SSO breaks the moment the URL changes if it isn't pre-registered).
  8. Cut-over: prisma migrate deploy → run the Azure sync + add-admin → point DNS at the app → test SSO + an upload + an approval email end-to-end.
  9. After: WAF/rate-limiting, backups verified, Dependabot/xlsx fix, decide if/when to graduate EC2 → ECS Fargate.

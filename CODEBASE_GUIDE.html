ProcDNA BD Tracker — Codebase Guide

Generated May 2026  ·  Next.js 16.2 App Router  ·  Prisma 7  ·  Supabase PostgreSQL

1. High-Level Architecture

One app, three layers:
Frontend (React/Next.js) — pages and components served from app/
Backend (Route Handlers) — REST API endpoints at app/api/, each a route.ts file
Database (Supabase) — PostgreSQL hosted on Supabase, accessed via Prisma ORM through a connection pool

There is no separate backend server. Next.js runs everything — it serves the React pages and also handles the API calls. When the browser (or a React component) calls fetch("/api/something"), that request goes to the matching route.ts file in the same project, which talks to the database via Prisma.

3. How Data Flows — The Request Lifecycle

Two patterns are used depending on whether the component is a server component (page) or a client component (interactive UI):

Pattern A — Server-Side Fetch (pages, initial load)

1
User navigates to /dashboard. Next.js runs app/dashboard/page.tsx on the server.
2
The page calls getDashboardStats() from lib/db/opportunities.ts directly — no HTTP, just Prisma querying Supabase.
3
Prisma sends a SQL query to Supabase PostgreSQL (schema procdna_database) via the connection pool in lib/prisma.ts.
4
Data comes back, the page renders HTML with the data pre-filled, and sends it to the browser.

Pattern B — Client-Side Fetch (interactive components)

1
User clicks a button (e.g. "Add staffing resource" in PricingDrawer.tsx).
2
The React component calls fetch("/api/pricing-versions/[pvId]/staffing", { method: "POST", body: ... }).
3
Next.js routes that request to app/api/pricing-versions/[pvId]/staffing/route.ts — the POST function there runs on the server.
4
The route handler imports prisma from lib/prisma.ts and calls Prisma methods which send SQL to Supabase.
5
The handler returns a JSON response. The React component updates its local state with the response and re-renders.
Where does the data actually live? All application data is stored in Supabase PostgreSQL, schema procdna_database, hosted on AWS Mumbai (ap-south-1). The connection string is in the DATABASE_URL environment variable. Prisma is the only thing that reads/writes the database — there is no direct SQL anywhere else in the codebase.

7. Authentication Flow

Library: NextAuth v4  •  Provider: Credentials (email + password)  •  Config: lib/auth.js
1
User submits email + password on /login.
2
NextAuth's CredentialsProvider.authorize() checks if the user exists in the User table by email.
3
If not found, it auto-creates a new User record with role BD — so any email can log in for the first time.
4
A JWT is issued containing user.id and user.role. Stored in a cookie.
5
On subsequent requests, getServerSession(authOptions) (in API routes) or useSession() (in client components) reads the JWT to know who is logged in.
6
Comments use getServerSession to set the authorId server-side — the client cannot fake who posted a comment.

Connection poollib/prisma.ts creates a pg.Pool with max: 5 connections. This avoids hitting Supabase's session-mode limit.

Schema namespaceAll tables are under procdna_database schema (not the default public). Set via the adapter's schema option.
Schema changesRun npx prisma db push to sync schema changes. Run npx prisma generate to regenerate the TypeScript client after any model change.
SeedingRun npx tsx prisma/seed.ts. Also syncs BD users from Azure Active Directory via Microsoft Graph API.

After any edit, patchVersion() recomputes and saves to the PricingVersion record: totalHours, totalCost, proposedBillings (Σ hours × effRate), grossMarginPct, offshorePct, effectiveRatePerHour.

In the Other Cost tab:

  • Bill checkbox → isBillable. Unchecked = not charged to client, Billed shows $0.
  • Markup % → stored as markupPct. Click to edit.
  • Billed = amount × (1 + markupPct/100). Click to enter manually — back-calculates markupPct.

01Overview

NEXA is ProcDNA's internal app for tracking new business opportunities from lead through pricing, approval, SOW signoff, and project-code allocation. It replaces a scattered set of Excel files and email threads with a single auditable workflow.

Production URL
https://procdnanexa.vercel.app (also legacy: pricingtoolprimero.vercel.app)
Repository branch
main — both Vercel projects deploy the same branch
Region
fra1 (Frankfurt, set via vercel.json)
DB
Supabase Postgres, schema procdna_database
Auth
Azure AD SSO (NextAuth v4) + fallback credentials at /login2

07Approval flow

LEAD │ create pricing version ▼ PRICE_LINKING_PENDING │ mark version as final ▼ PRICE_LINKED │ request PRICING approval ▼ APPROVAL_PENDING ├─ approve → SOW_PENDING ├─ reject → PRICE_LINKED └─ withdraw → PRICE_LINKED (requester only; stores withdrawalReason) SOW_PENDING │ upload SOW/PO + preContractAgreed = true ▼ SOW_SUBMITTED │ request SOW_VERIFICATION approval ▼ SOW_REVIEW_PENDING ├─ approve → TO_BE_ARCHIVED └─ reject → SOW_SUBMITTED TO_BE_ARCHIVED

Pricing version lock

A pricing version becomes read-only when isFinal === true AND stage ∈ { APPROVAL_PENDING, SOW_PENDING, SOW_SUBMITTED, SOW_REVIEW_PENDING, TO_BE_ARCHIVED }. Lock check uses the live oppStage state passed as currentStage prop to PricingDrawer, not the static opp.stage from server. After rejection/withdrawal the stage reverts and the lock lifts immediately without page reload.

Tab gating (visual dim only)

TabDimmed when
SOW / POStage NOT in {SOW_PENDING, SOW_SUBMITTED, SOW_REVIEW_PENDING, TO_BE_ARCHIVED}
Project CodeStage !== TO_BE_ARCHIVED

Withdraw button visibility

Shows only when sessionUserId === opp.ownerId AND a pending PRICING approval exists. Owner-only — admins cannot withdraw on someone else's behalf in the current UI.

08Auth flow

Sign-in

  1. User clicks Sign in with Microsoft on /login
  2. NextAuth redirects to Azure AD (login.microsoftonline.com) with state + nonce
  3. Microsoft authenticates and redirects back to /api/auth/callback/azure-ad?code=…
  4. NextAuth exchanges code for ID token, calls signIn callback in lib/auth.js:
    • Verifies profile.email exists in the users table
    • Updates azureId and location on first sign-in
    • Returns true (allow) or false/error URL (deny)
  5. Session JWT issued, signed with NEXTAUTH_SECRET, set as __Secure-next-auth.session-token cookie
  6. Browser redirected to callbackUrl (usually /dashboard)

Per-request auth

proxy.ts intercepts every non-excluded request, calls getToken with explicit secret + secureCookie: true + cookieName, and either lets the request through or redirects to /login.

API routes do the same check via getAuthToken(req) from @/lib/getAuthToken (returns 401 if missing).

RBAC

Visibility hierarchy (highest first): ADMIN → PARTNER → ED → DIRECTOR → SEL. Filter built in lib/db/opportunities.ts via resolveOwnerFilter(auth).

  • SEL: own opportunities only
  • DIRECTOR: own + SELs in team
  • ED: own + 2-level subtree
  • PARTNER / ADMIN: all

09Mail (lib/mail.ts)

Sent via Microsoft Graph API endpoint POST /users/{MAIL_SENDER}/sendMail. Token via client-credentials (@azure/identity's ClientSecretCredential).

Graph blocks RFC headers: Message-ID, In-Reply-To, and References all return InvalidInternetMessageHeader. Threading relies on identical subject lines (Gmail/Outlook thread by subject) — not RFC headers.

mailApprovalRequested()approver (with action links), CC list (info), requester (confirmation)POST /api/opportunities/[id]/approvals.
mailApprovalApproved()requester + approverAfter approve.
mailApprovalRejected()requester + approverAfter reject. Includes reason.
mailApprovalWithdrawn()approver + requesterAfter withdraw. Includes reason. Same subject as original for threading.

Email-action endpoint

Approval emails contain one-click Approve / Reject links pointing to /api/approvals/email-action?token=…. Token signed with HMAC-SHA256 using EMAIL_ACTION_SECRET, 7-day TTL, encodes (approvalId, approverId, action). Endpoint renders a branded HTML confirmation page (POST-back to commit the action).

10File storage

All file uploads go through Supabase Storage. Use getSupabase() from lib/supabase.ts (lazy singleton — never instantiate at module level or you'll leak connections in serverless).

Buckets

  • SoW_bucket — SOW documents
  • PO_bucket — PO documents (typo is baked in)

Flow

  1. Client POSTs file via multipart to /api/opportunities/[id]/sow
  2. Server uploads to Supabase Storage with service role key (bypasses RLS)
  3. Server creates SOWDocument row with storagePath
  4. On subsequent GETs, server issues a fresh 55-min signed URL (cached in lib/supabase.ts)

12Conventions

Code patterns

  • Auth in API routes: const token = await getAuthToken(req) from @/lib/getAuthToken. Return apiError('UNAUTHORIZED') on miss.
  • Errors: use apiError(code) from lib/errors.ts with codes from APP_ERRORS. Don't return raw NextResponse.json({error:…}) for known errors.
  • Prisma Decimal: always Number(value) when transferring to client / doing math.
  • Server components fetch data; client components handle interactivity. Don't put 'use server' at the top of a page.tsx — it breaks rendering in Next 16.
  • Locked pricing: pass live state (oppStage) into PricingDrawer as currentStage prop; don't rely on opp.stage from the server (it's stale once the user takes actions).

Naming & UI

  • Opportunity IDs: BD-NNN. Client IDs: CL-NNN.
  • Tabs visually dimmed (40% opacity) when their stage gate isn't met — still clickable so users can inspect.
  • Domain pills (Analytics, MS, DS, Design, Technology, AUXO) used for LoB breakdown.
  • Status field is now a manual dropdown (OPEN / WON). Auto-WON-on-approval was removed.

13Gotchas

Never write 'use server' at the top of a page.tsx. It marks every export in the file as a Server Action, which is incompatible with a page-render default export. Caused the silent login → redirect-to-login loop on the new Vercel project until removed from app/dashboard/page.tsx.
NextAuth env-var auto-detection fails in Edge runtime. Internal NextAuth code reads process.env.NEXTAUTH_URL / VERCEL_URL buried inside node_modules. Vercel's build can fail to inline those references for Edge bundles, leaving useSecureCookies defaulting to false → looking for next-auth.session-token (no prefix) → never finding the actual __Secure-next-auth.session-token cookie. Always use getAuthToken / the explicit pattern in proxy.ts.
Vercel doesn't run prisma migrate deploy — only prisma generate (via postinstall). Any DB schema change must be applied as SQL in Supabase manually. The prisma/migrations/ folder is historical only.
Supabase typo: PO_bucket — don't fix it without a migration plan. Existing PO docs reference this bucket name in their storagePath.
Microsoft Graph blocks RFC email headers. Message-ID, In-Reply-To, References all rejected. Email threading relies on identical subject lines.

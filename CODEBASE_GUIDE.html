ProcDNA BD Tracker — Codebase Guide

Updated May 2026  ·  Next.js 16.2 App Router  ·  Prisma 7  ·  Supabase PostgreSQL (PgBouncer port 6543)

1. High-Level Architecture

One app, three layers:
Frontend (React/Next.js) — pages and components served from app/
Backend (Route Handlers) — REST API endpoints at app/api/, each a route.ts file
Database (Supabase) — PostgreSQL hosted on Supabase, accessed via Prisma ORM through PgBouncer connection pool (port 6543)

There is no separate backend server. Next.js runs everything — it serves the React pages and also handles the API calls. When the browser (or a React component) calls fetch("/api/something"), that request goes to the matching route.ts file in the same project, which talks to the database via Prisma.

3. How Data Flows — The Request Lifecycle

Two patterns are used depending on whether the component is a server component (page) or a client component (interactive UI):

Pattern A — Server-Side Fetch (pages, initial load)

1
User navigates to /dashboard. Next.js runs app/dashboard/page.tsx on the server.
2
The page calls getDashboardStats() from lib/db/opportunities.ts directly — no HTTP, just Prisma querying Supabase.
3
Prisma sends a SQL query to Supabase PostgreSQL (schema procdna_database) via the PgBouncer pool in lib/prisma.ts.
4
Data comes back, the page renders HTML with the data pre-filled, and sends it to the browser.

Pattern B — Client-Side Fetch (interactive components)

1
User clicks a button (e.g. "Add staffing resource" in PricingDrawer.tsx).
2
The React component calls fetch("/api/pricing-versions/[pvId]/staffing", { method: "POST", body: ... }).
3
Next.js routes that request to app/api/pricing-versions/[pvId]/staffing/route.ts — the POST function there runs on the server.
4
The route handler imports prisma from lib/prisma.ts and calls Prisma methods which send SQL to Supabase.
5
The handler returns a JSON response. The React component updates its local state with the response and re-renders. Many updates use optimistic state — the UI updates immediately and the API call fires in the background.
Where does the data actually live? All application data is stored in Supabase PostgreSQL, schema procdna_database, hosted on AWS Mumbai (ap-south-1). The connection string is in the DATABASE_URL environment variable. Prisma is the only thing that reads/writes the database — there is no direct SQL anywhere else in the codebase.

7. Authentication Flow

Library: NextAuth v4  •  Provider: Credentials (email + password)  •  Config: lib/auth.js
1
User submits email + password on /login.
2
NextAuth's CredentialsProvider.authorize() checks if the user exists in the User table by email.
3
If not found, it auto-creates a new User record with role BD — so any email can log in for the first time.
4
A JWT is issued containing user.id and user.role. Stored in a cookie.
5
On subsequent requests, getServerSession(authOptions) (in API routes) or useSession() (in client components) reads the JWT to know who is logged in.
6
Comments use getServerSession to set the authorId server-side — the client cannot fake who posted a comment.

8. Supabase Connection & PgBouncer

Important — port matters: Supabase exposes two ports. Port 5432 is direct (session mode) and holds one dedicated DB connection per app connection — this causes the EMAXCONNSESSION error ("max clients reached") under Vercel's serverless model where many parallel invocations open many connections simultaneously. Port 6543 goes through PgBouncer in transaction mode, which multiplexes many app connections through a small pool of real DB connections. Always use port 6543 in DATABASE_URL.
StepDetail
Connection stringStored in DATABASE_URL env var. Format: postgresql://postgres.[project]:[password]@aws-1-ap-south-1.pooler.supabase.com:6543/postgres
Connection poollib/prisma.ts creates a pg.Pool with max: 5 connections. Redundant safety net alongside PgBouncer.
PgBouncer (port 6543)Supabase's built-in connection multiplexer. Transaction mode: each query borrows a real DB connection for the duration of the transaction, then returns it. Many serverless function invocations share a small pool without hitting session limits.
Prisma adapterPrismaPg from @prisma/adapter-pg bridges Prisma 7 to the pg pool.
Schema namespaceAll tables are under procdna_database schema (not the default public). Set via the adapter's schema option.
Schema changesRun npx prisma db push to sync schema changes. Run npx prisma generate to regenerate the TypeScript client after any model change. If a column removal causes data loss, the CLI requires --accept-data-loss.
SeedingRun npx tsx prisma/seed.ts. Also syncs BD users from Azure Active Directory via Microsoft Graph API. Rate cards are upserted — re-running is safe and will backfill any new fields (e.g. domain).

9d. Key Functions & Callbacks

FunctionWhat it does
computeFromRows(rows) Pure function. Iterates all active StaffRow items and their weeklyHours. Returns ComputedMetrics: billedHours (active+billable), unbilledHours (active+non-billable), totalCost (all active), proposedBillings (active+billable × effRate), grossMargin, grossMarginPct, offshorePct (India billed hrs / billed hrs), effectiveRatePerHour. Never touches the DB.
patchVersion(rows) Runs computeFromRows(rows), updates versionMetrics state, then sends PATCH /api/pricing-versions/[pvId] with all computed fields. Called automatically after every edit that changes hours, rates, or row toggles. Also called explicitly by the Save & Close button.
toStaffRow(sr) Maps a raw staffingResource DB record (with Decimal types, Date objects) into a typed StaffRow. Normalises: Number() for all Decimal fields, toISOString().slice(0,10) for weekStartDate.
addRow(rc) Posts to POST /api/pricing-versions/[pvId]/staffing with { rateCardId }. Appends the new row to staffRows state and calls patchVersion(). The same rate card can be added multiple times.
removeRow(srId) Sends DELETE /api/pricing-versions/[pvId]/staffing/[srId], removes row from state, calls patchVersion().
commitHours(srId, wk, val) Sends PUT /api/pricing-versions/[pvId]/staffing/[srId]/hours with { weekStartDate, hours }. Updates row's weeklyHours in state (upsert by weekStartDate). Calls patchVersion().
applyUtilization(srId, util) Computes hoursPerWeek = round(util/100 × 40). Sends PATCH with { utilization, weekEntries: allWeeks × hoursPerWeek }. Updates all week cells and utilization on the row in state. Calls patchVersion().
commitEffectiveRate(srId, val) Sends PATCH with { effectiveBillRate: eff }. Updates effectiveBillRate in state. Calls patchVersion(). D/P% is derived from this on render: ((eff − sys) / sys) × 100.
commitDP(srId, val) Inverse of above. Converts D/P% input to eff rate: eff = sysRate × (1 + dp/100). Sends PATCH with { effectiveBillRate: eff }.
toggleRow(srId, isActive) Sends PATCH with { isActive }. Updates state. Calls patchVersion().
toggleStaffBillable(srId, isBillable) Sends PATCH with { isBillable }. Updates state. Calls patchVersion().
addOtherCost() Posts to POST /api/opportunities/[id]/other-costs. Appends to otherCosts state.
toggleBillable(costId, billable) Optimistic: updates state immediately, then sends PATCH { isBillable } to /other-costs/[costId].
commitMarkup(costId, val) Sends PATCH { markupPct }. Updates markupPct in state. Billed is derived: amount × (1 + pct/100).
commitBilled(costId, val) Back-calculates: markupPct = (billed/amount − 1) × 100. Sends PATCH { markupPct }. Bidirectional link with Markup.
removeOtherCost(costId) Sends DELETE to /other-costs/[costId]. Removes from state.

9e. Pure Helper Functions (module-level)

FunctionSignatureWhat it does
getWeekColumns (start, end) → Date[] Returns an array of Monday dates (as Date objects) covering the full opportunity date range. Max 52 weeks. Used to generate the W1…Wn header columns in the Efforts table.
weekKey (d: Date) → string Returns 'YYYY-MM-DD' string for a Monday Date. Used as the key for the hoursMap lookup on each row.
getProjectMonths (start, end) → string[] Returns every 'YYYY-MM' month key from the opportunity start month to end month inclusive. Used as the column list for the Financial tab.
distributeWeekToMonths (weekStartDate: string, value: number) → [string, number][] The core Financial tab algorithm. Takes a Monday date string and a value (hours, cost, revenue, etc.). Iterates Mon through Fri (5 working days). For each day, determines the 'YYYY-MM' month key. Distributes value ÷ 5 per day, accumulating into months. Returns [monthKey, portion] pairs.
Example: A week starting Mon 2025-01-27 has Tue 28 Jan, Wed 29 Jan in January and Thu 30 Jan, Fri 31 Jan, Mon... wait — Mon 27, Tue 28, Wed 29, Thu 30, Fri 31 are all in Jan. A week starting Mon 2025-01-27 → 5/5 in Jan. A week starting Mon 2025-01-27 with last day Fri 2025-01-31 → all Jan. A week Mon 2025-02-03 with days Mon–Fri all in Feb. A split week: Mon 2025-01-27... if the month ends on Tue 28 Jan, then Mon+Tue = 2 days Jan, Wed+Thu+Fri = 3 days Feb → 40 % Jan, 60 % Feb.
computeFromRows (rows: StaffRow[]) → ComputedMetrics Pure function. Filters to active rows. For each active row, iterates weeklyHours. Billable rows contribute to billedHours, proposedBillings, and indiaHours. Non-billable active rows contribute to unbilledHours only. All active rows contribute to totalCost. Derives grossMarginPct, offshorePct, effectiveRatePerHour.
fmtRole (r: string) → string Converts BUSINESS_ANALYST_SENIOR → Business Analyst Senior. Replaces underscores with spaces, title-cases each word.
fmt (n: number) → string Currency shortener: ≥1M → $X.XXM, ≥1K → $XK, else $X. Returns '!' for null/undefined.

9h. Financial Tab — Architecture

The Financial tab is 100% client-side computed — no API calls, no saved snapshots. It reads staffRows and otherCosts state (already in memory) and recomputes on every render. Columns = one per project month + a Total column.

Week-to-Month Distribution Algorithm

Function: distributeWeekToMonths(weekStartDate, value)

  • Each week entry's weekStartDate is a Monday ('YYYY-MM-DD').
  • The week spans Mon through Fri — exactly 5 working days.
  • Each day contributes value ÷ 5 to whichever calendar month it falls in.
  • If all 5 days are in one month → 100% to that month.
  • If Mon–Tue are in Month A and Wed–Fri in Month B → 40% to A, 60% to B.
  • Dates are parsed as local time using new Date(y, mo-1, d+i) (no UTC conversion) to avoid timezone-shift bugs.

This same algorithm is applied to: hours (for F1, F2), revenue (A1), cost (B), and the India/US sub-totals (for J, K).

Other Cost Monthly Distribution

All other costs (raw amount → C; billed amount → A2) are divided equally across the total number of project months. E.g. a $12,000 travel cost over a 6-month project = $2,000/month for C and A2.

Total Column

The Total column sums the raw accumulatable fields (a1, a2, b, c, f1, f2, indiaRev, indiaHrs, usRev, usHrs, recRev) across all months first, then re-derives all computed rates (d, dPct, ePct, g, h, i, j, k) from those sums. This ensures rate calculations are correct for the project total rather than averaging monthly rates (which would be wrong).

10. Pricing Version Management

Each opportunity can have multiple pricing versions (v1, v2, etc.). The Pricing tab in OpportunityTabs.tsx shows a card for each version with the following actions:

Save & Close

The drawer header contains a "Save & Close" button. Clicking it calls patchVersion(staffRows) (which saves computed metrics to the DB) and then onClose(). This lets users explicitly save before leaving the drawer without relying on the auto-save that triggers on each edit.

Mark as Final

Each version card has a "Mark as Final" button. Clicking it sends PATCH /api/pricing-versions/[pvId] with { isFinal: true }. The server atomically:

  1. Runs updateMany to set isFinal = false on all sibling versions for the same opportunity
  2. Runs update to set isFinal = true on the selected version

The selected version card shows a Selected for approval badge and an emerald border ring. There can only be one final version per opportunity at a time. The Mark as Final button is replaced by the badge on the final version's card.

Duplicate

Each version card has a "Duplicate" button. Clicking it sends POST /api/pricing-versions/[pvId]/duplicate. The server creates a full deep copy inside a single Prisma $transaction:

  • New PricingVersion with versionNumber = max + 1, isFinal = false, all other fields copied
  • All StaffingResource rows copied (same rates, designation, flags)
  • All StaffingWeekEntry rows for each resource copied
  • All ScheduleOfPayment rows copied
  • All FinancialSnapshot rows copied

The new version is appended to the state list immediately. It is independent — editing it does not affect the source version.

Delete

Each version card has a "Delete" button. Clicking it enters an inline confirmation state (only that specific card shows the confirmation — other cards are unaffected). The user sees "Delete? / Yes, delete / Cancel". Confirming sends DELETE /api/pricing-versions/[pvId]. The server hard-deletes the record; Prisma cascade deletes all child records (staffing resources, week entries, schedule-of-payment rows, financial snapshots). The version is removed from the UI state immediately. There is no soft-delete / recycle bin — deletion is permanent. Use Duplicate as a backup before deleting.

12. Vercel Deployment

Auto-deploy is always on. Every push to the branch connected to Vercel triggers a new build automatically — there is no toggle to enable it. The connected branch is typically main.
ConceptDetail
Production URLThe stable URL shown on the Vercel dashboard (e.g. pricingtool.vercel.app). Always points to the latest successful build of the connected branch.
Preview / Deployment URLEach build also gets a unique URL (e.g. pricingtool-abc123.vercel.app). These are immutable snapshots of that exact build — useful for testing a specific deploy.
Branch requirementChanges on feature branches (e.g. efforts-refinement) only deploy if Vercel is configured to deploy that branch, or after the branch is merged into the connected production branch (main).
Prisma on deploypackage.json has "postinstall": "prisma generate" — this ensures the Prisma TypeScript client is regenerated during every Vercel build, so the deployed app always has the current schema types.
Environment variablesSet in Vercel project settings. Must include DATABASE_URL (port 6543), NEXTAUTH_SECRET, NEXTAUTH_URL, and Azure AD credentials for seed sync.

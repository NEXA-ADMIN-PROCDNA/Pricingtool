ProcDNA BD Tracker — Codebase Guide

Updated May 2026  ·  Next.js 16.2 App Router  ·  Prisma 7  ·  Supabase PostgreSQL + Storage  ·  PgBouncer port 6543

1. High-Level Architecture

One app, three layers:
Frontend (React/Next.js) — pages and components served from app/
Backend (Route Handlers) — REST API endpoints at app/api/, each a route.ts file
Database + Storage (Supabase) — PostgreSQL for structured data (via Prisma/PgBouncer), and private Storage buckets for file uploads (SoW, PO documents)

There is no separate backend server. Next.js runs everything — it serves the React pages and also handles the API calls. When the browser (or a React component) calls fetch("/api/something"), that request goes to the matching route.ts file in the same project, which talks to the database via Prisma.

3. How Data Flows — The Request Lifecycle

Two patterns are used depending on whether the component is a server component (page) or a client component (interactive UI):

Pattern A — Server-Side Fetch (pages, initial load)

1
User navigates to /dashboard. Next.js runs app/dashboard/page.tsx on the server.
2
The page calls getServerSession(authOptions) to get userId and role, then calls getDashboardStats(auth) and getOpportunities(status, q, auth) from lib/db/opportunities.ts directly — no HTTP, just Prisma.
3
Prisma applies the RBAC where-filter (resolveOwnerFilter) and sends a SQL query to Supabase via PgBouncer.
4
Data comes back filtered by role. The page renders HTML with pre-filled data and sends it to the browser.

Pattern B — Client-Side Fetch (interactive components)

1
User drops a file onto the SoW upload zone in TabSoW.tsx.
2
The component calls fetch("/api/opportunities/[id]/sow", { method: "POST", body: formData }).
3
Next.js routes the request to app/api/opportunities/[id]/sow/route.ts. The handler validates the NextAuth JWT, then uploads the file to Supabase Storage using the service role key.
4
The handler creates a SOWDocument DB record via Prisma and returns a signed URL (1-hour TTL). The UI appends the new doc to state.
Where does data actually live? Structured data is in Supabase PostgreSQL, schema procdna_database. Files (SoW, PO) are in Supabase Storage in private buckets (SoW_bucket, PO_busket). Files are accessed via temporary signed URLs generated server-side (1-hour TTL). Prisma is the only thing that reads/writes the database.

7. Authentication & RBAC

Library: NextAuth v4  ·  Provider: Credentials (email + password)  ·  Config: lib/auth.js

Auth Flow

1
User submits email + password on /login.
2
CredentialsProvider.authorize() looks up the user by email. If not found, auto-creates with role BD.
3
A JWT is issued containing user.id and user.role. Stored in a cookie. Signed with HMAC-SHA256 using NEXTAUTH_SECRET — clients cannot forge or modify the token.
4
In API routes: getToken({ req }) reads and verifies the JWT. Returns null if invalid/missing → route returns 401.
5
In server components (pages): getServerSession(authOptions) returns the session object with user.id and user.role.

Role Hierarchy & RBAC

Role hierarchy (highest to lowest): ADMIN → PARTNER → ED → DIRECTOR → SEL → BD.

RBAC is enforced in lib/db/opportunities.ts by resolveOwnerFilter(auth: AuthCtx) — a synchronous pure function that returns a Prisma where fragment. No managerId traversal, no recursive DB calls — just a single query with a role-based filter:

RoleWhat they seePrisma where fragment
ADMIN, PARTNERAll opportunities{} (no filter)
EDOwn + DIRECTOR-owned + SEL-ownedOR: [{ ownerId }, { owner: { role: { in: ['DIRECTOR','SEL'] } } }]
DIRECTOROwn + SEL-ownedOR: [{ ownerId }, { owner: { role: 'SEL' } }]
SELOwn only{ ownerId }
BD (default)Own only (fallback){ ownerId }
Security model: The JWT is signed server-side — a user cannot inspect or modify the role field in the token. RBAC filtering happens entirely in server-side Prisma queries; the browser never receives data it isn't supposed to see. Opportunities are the primary restricted resource.

8b. Supabase Storage (File Uploads)

Why service role key? NextAuth (not Supabase Auth) is used for authentication, so auth.uid() is always null in Supabase RLS policies. The service role key bypasses RLS entirely — this is safe because uploads only go through server-side API routes that verify the NextAuth JWT first.
ItemDetail
Client factorygetSupabase() in lib/supabase.ts. Lazy — reads env vars at call time (not module load), so the module imports cleanly even if env vars are missing in some build contexts.
CredentialsSUPABASE_URL = https://<project-id>.supabase.co (NOT the dashboard URL). SUPABASE_SERVICE_ROLE_KEY = starts with eyJhb… (JWT), found under Project Settings → API → Service Role Key.
BucketsSoW_bucket (Statement of Work) · PO_busket (Purchase Orders). Both are private — no public access.
Storage path pattern{opportunityId}/v{version}_{timestamp}.{ext} — e.g. BD-001/v2_1715000000000.pdf.
Allowed MIME typesPDF, Word (.doc/.docx), Excel (.xls/.xlsx), PNG, JPEG. Max 20 MB per file.
Download accessSigned URLs generated server-side with 1-hour TTL via supabase.storage.from(bucket).createSignedUrl(path, 3600). Returned in API responses and used directly as <a href> download links.
Soft deleteDeleting a document sets isActive: false in the DB record. The physical file in Supabase Storage is not removed — preserves history. GET endpoints filter by isActive: true.
VersioningEach upload increments version: existing count + 1. All versions are kept in storage, only the active ones are returned by the GET endpoint.

10. Pricing Version Management

Mark as Final

Sends PATCH /api/pricing-versions/[pvId] with { isFinal: true }. Server atomically sets all siblings to isFinal: false then sets target to true. Only one final version per opportunity at any time. Final version shows Selected for approval badge.

Duplicate

Sends POST /api/pricing-versions/[pvId]/duplicate. Server deep-copies in a single $transaction: new version + all staffing resources + all weekly hour entries + all SoP rows + all financial snapshots. Fully independent from source.

Delete

Inline confirmation required per card. Sends DELETE /api/pricing-versions/[pvId]. Hard-delete with cascade — permanent, no recycle bin. Use Duplicate before deleting to preserve a backup.
